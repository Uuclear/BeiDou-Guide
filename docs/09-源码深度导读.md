# 源码深度导读（gms-server / gms-ui）

本文档说明如何在 **BeiDou-Server** 仓库中阅读与调试源码，并与本仓库中的「机器可读索引」配合使用。

---

## 1. 为什么不用几十页手写「每个函数说明」

当前 **gms-server** 约 **1116** 个 `.java` 源文件、公开类型逾 **1100** 个；若对每一公开方法都写一段自然语言说明，体积与维护成本会远超单一 Markdown 文档的合理范围，且极易与上游分叉后过时。

更稳妥的做法是：

1. **架构与分层**：用少量文档讲清目录职责、请求链路、数据流（见 [02-架构设计文档.md](./02-架构设计文档.md) 及下文补充）。
2. **全量索引**：用脚本从**你本地的** `BeiDou-Server/gms-server` 生成「按包分组的类型声明 + public 方法签名」清单（见 [reference/gms-server-api-index.md](./reference/gms-server-api-index.md)）。
3. **按需深入**：对某个类有疑问时，在索引中定位包名与类名，再打开对应 `.java` 阅读实现；网络包可结合 `RecvOpcode` / `SendOpcode` 与 Handler 类名对照。

## 2. gms-server 包级地图（`org.gms.*`）

以下按**源码文件数量**大致排序，便于建立 mental model（具体数字会随上游变动）。

| 包 / 目录 | 角色（阅读顺序建议） |
|-----------|----------------------|
| `org.gms` | 入口 `ServerApplication`：启动前解析 YAML、若库不存在则 `CREATE DATABASE`（避免 Flyway 等自动配置在空库名上失败）。 |
| `org.gms.config` / `org.gms.property` | Spring 配置、业务配置项绑定。 |
| `org.gms.aop` | `ServerFilter`、JWT 相关过滤器与鉴权入口。 |
| `org.gms.controller` | **REST 管理端**：与 gms-ui 对应，按资源划分（账号、角色、服务器、商店等）。 |
| `org.gms.service` | 管理端业务编排；常调用 `dao` 与 `client` 侧领域逻辑。 |
| `org.gms.dao` | MyBatis-Flex Mapper 与数据访问。 |
| `org.gms.model` | 实体、DTO、枚举等与持久化或 API 契约相关的类型。 |
| `org.gms.client` | **游戏域模型**：角色、地图对象、任务、技能等（体量最大之一），与 Netty 会话紧耦合。 |
| `org.gms.net` | **Netty 网络栈**：登录服/频道服 Handler、`PacketHandler` 分发、收发包；与 `constants` 中的 opcode 配合阅读。 |
| `org.gms.server` | 世界/频道/地图等**运行时服务**与调度（与 `client` 配合）。 |
| `org.gms.scripting` | GraalVM JS 脚本桥接与脚本事件。 |
| `org.gms.provider` | WZ/数据提供抽象（与游戏数据加载相关）。 |
| `org.gms.constants` | opcode、游戏常量、枚举。 |
| `org.gms.util` / `org.gms.exception` | 工具类与业务异常。 |
| `org.gms.manager` | 少量全局管理器式类。 |

**推荐阅读路径（第一次拉代码）**

1. `ServerApplication.java` → 理解启动与建库逻辑。  
2. 任选一个 `org.gms.controller.*Controller` → 看一条 HTTP 请求如何进 `service`。  
3. `org.gms.net` 下 `PacketHandler` 相关接口与 `AbstractPacketHandler` → 看游戏包如何进 `client` 域。  
4. 打开 `org.gms.client.Character`（或当前版本中的主玩家类）→ 理解状态如何落在内存与数据库之间。

## 3. 网络层：从 Netty 到 Handler

游戏客户端（经 ijl15 等）连到 **8484** 的 Netty 服务后，数据会按 **opcode** 分发给各 `*Handler`。

- **登录流程**：`org.gms.net.server.handlers.login` 下各类 `*Handler`（如选角、建角、PIN 等）。
- **频道内玩法**：`org.gms.net.server.channel.handlers` 下大量 `*Handler`（移动、技能、物品、伤害等）。

阅读技巧：在 `org.gms.constants` 中搜索 `RecvOpcode` / `SendOpcode`（或当前仓库中的等价命名），用 opcode 值反查注册 Handler 的代码位置（通常在 Netty 初始化或 packet 工厂类中）。

## 4. gms-ui（Vue 3）如何对照后端

前端源码在 **BeiDou-Server** 仓库的 `gms-ui/` 下（本指南仓库仅提供文档）。

- **页面与路由**：`gms-ui/src/views`、`router` 配置。  
- **API 调用**：通常按模块分在 `api` 或 `service` 目录；与 `org.gms.controller` 的路径一一对应。  
- **对照方法**：在 UI 中搜 `axios`/`request` 的 URL 片段，再到 `controller` 中搜相同路径。

## 5. 再生成本地「全量类/方法索引」

本仓库已提供生成脚本（需你本机存在 **BeiDou-Server** 的 `gms-server` 目录）：

```bash
export BEIDOU_SERVER_ROOT=/你的路径/BeiDou-Server/gms-server
python3 scripts/generate-gms-server-api-index.py
```

生成结果：`docs/reference/gms-server-api-index.md`

**说明**

- 索引按 **包名** 分组，每个 **public 顶层类型** 下列出启发式提取的 **public 方法签名**（含多行参数列表的合并行）。
- 非 `public` 的方法、内部类、Lombok 生成的方法不会完整出现在索引中；以 IDE 与源文件为准。
- 若某文件无 `package` 声明或类型声明非常规，可能出现「未解析」占位，请直接打开源文件。
