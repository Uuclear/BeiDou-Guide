## 2. gms-server：Spring Boot 与游戏服的关系

仓库里 **`gms-server`** 同时承担：

1. **管理端 REST API**（供 `gms-ui` 调用）：`org.gms.controller` + `org.gms.service` + `org.gms.dao` 等典型三层结构。  
2. **游戏客户端长连接**（Netty，常见端口与架构文档一致）：`org.gms.net` 下 `LoginServer` / `ChannelServer` 等与 `org.gms.client` 领域对象配合。

二者共用同一 JVM 进程时，**注意不要**把「HTTP 请求里的 `Client`」与「游戏会话里的 `org.gms.client.Client`」混为一谈——后者是 Maple 风格会话对象，持有角色、地图、背包等运行时状态。

## 3. 启动入口：`ServerApplication`

类 `ServerApplication` 是 Spring Boot 入口。在调用 `SpringApplication.run` 之前，会先执行 **`initDb`**：

- 从 **`spring.config.location`** 指定的外部 yml，或 jar 内 **`application.yml`**，用 SnakeYAML 读出 **`mybatis-flex.datasource.mysql`** 一段。  
- 解析出 JDBC URL 里的**库名**，用不带库名的 URL 连上 MySQL，执行 **`SHOW DATABASES`**；若库不存在则 **`CREATE DATABASE … utf8mb4`**。  
- 设计动机（见源码注释）：Flyway / JPA 等自动配置会在库不存在时反复连库失败，因此在它们之前**手动建库**。

辅助方法 **`getStartParam`**：按优先级读取配置——**JVM 系统属性** → **`--key=value` 启动参数** → **环境变量**（点号替换为下划线）。用于与容器、K8s 注入的环境变量对齐。

## 4. 管理 API 的一道业务闸：`ServerFilter`

`ServerFilter` 继承 `HttpFilter`，注释写明：请求到达时**已经过 Spring Security**。本过滤器再做三件事（顺序上有意设计）：

1. **解析真实客户端 IP**：`X-Forwarded-For`、`X-Real-IP`、`getRemoteAddr` 组合，空则抛业务异常。  
2. **封禁 IP**：`accountService.isBanned(remoteAddr)`，命中则关输入流并抛 `BizException`。  
3. **按 IP 限流**：`RateLimitUtil.getInstance().check(remoteAddr)`。  

之后若路径属于静态资源、Swagger、根路径 `/`，则直接放行；若为 **`multipart/form-data`** 也直接放行（避免对大上传做 body 缓存）。其余请求用 **`CachedHttpServletRequest`** 把 body 读入字节数组，使下游可以**多次读取** body（例如日志、重复解析），避免默认只能读一次流的问题。

## 5. 游戏网络：从 TCP 连接到 `*Handler`（非常详细）

本节按**时间顺序**描述一条游戏连接在服务端经历什么；读代码时可对照类名在 IDE 里跳转。

### 5.1 两类监听：`LoginServer` 与 `ChannelServer`

- **`LoginServer`**（`org.gms.net.netty.LoginServer`）  
  - 构造时只收**端口号**。  
  - `start()`：`NioEventLoopGroup` 父/子组 + `ServerBootstrap` + `NioServerSocketChannel`，`childHandler` 固定为 **`LoginServerInitializer`**。  
  - 常量 **`WORLD_ID = -1`、`CHANNEL_ID = -1`**：表示「当前连接属于登录服逻辑」，后面拿 `PacketProcessor` 时也会用这对值做 key。

- **`ChannelServer`**（`org.gms.net.netty.ChannelServer`）  
  - 构造时收 **端口 + world + channel**。  
  - `start()` 同上，但 `childHandler` 为 **`ChannelServerInitializer(world, channel)`**，从而把**物理端口**与**逻辑世界/频道号**绑定。

两者都继承 **`AbstractServer`**，对外暴露 `start()` / `stop()`，便于上层（通常是 `Server` 单例在启动流程里）统一拉起或关闭 Netty。

### 5.2 新连接建立：`LoginServerInitializer` / `ChannelServerInitializer`

两条 Initializer 结构对称，差异在「用哪套 `PacketProcessor`」和「`Client` 类型」。

**共同步骤（建议对照源码逐行）：**

1. **打日志**：打出远端 IP（便于审计连接风暴）。  
2. **取分包器**：  
   - 登录：`PacketProcessor.getLoginServerProcessor()` → 内部用 `world=-1, channel=-1` 做缓存 key。  
   - 频道：`PacketProcessor.getChannelServerProcessor(world, channel)` → **要求**事先执行过 `PacketProcessor.registerGameHandlerDependencies(...)`，否则抛 `IllegalStateException`（频道 Handler 往往依赖 `ChannelDependencies` 注入的 Spring Bean，例如 `noteService`）。  
3. **分配会话号**：`sessionId` 来自 **`ServerChannelInitializer`** 里的静态 `AtomicLong`（从 7777 起递增），每个 Netty 连接一个 **`clientSessionId`**，用于和别的 `Client` 区分、排查日志。  
4. **取规范化远端地址字符串**：`getRemoteAddress(socketChannel)`，与后面限流、封禁、多开协调共用同一套「字符串形式的 IP」。  
5. **连接级限流**：`RateLimitUtil.getInstance().check(remoteAddress)`，失败则 **直接 `socketChannel.close()`**（注意：这里关的是**刚建立的 TCP**，尚未进入游戏逻辑）。  
6. **构造会话对象**：  
   - 登录：`Client.createLoginClient(...)` → 内部 `Type.LOGIN`。  
   - 频道：`Client.createChannelClient(...)` → 内部 `Type.CHANNEL`。  
7. **会话准入（登录线独有）**：`SessionCoordinator.getInstance().canStartLoginSession(client)`，失败则关连接（见下文 **第 7 节**）。  
8. **频道线独有**：若 `Server.getInstance().getChannel(world, channel) == null`（世界未起、频道号错、尚未注册 `Channel` 实例），则 `SessionCoordinator.closeSession(client, true)` 并关 TCP。  
9. **`initPipeline(socketChannel, client)`**：真正把 **编解码 + `Client` 进 pipeline**（见 **5.3**）。

摘录（上游 `BeiDou-Server`，`LoginServerInitializer#initChannel`）：取登录用 `PacketProcessor` → 限流 → `createLoginClient` → `SessionCoordinator.canStartLoginSession` → `initPipeline`。

### 5.3 Pipeline：`ServerChannelInitializer.initPipeline` 与首包握手

`ServerChannelInitializer`（`Login`/`Channel` 的父类）在 **`initPipeline`** 里完成三件大事：

1. **为会话生成一对 IV**：`InitializationVector.generateSend()` / `generateReceive()`，供后续包体加解密。  
2. **`ProtocolFactory`**：`ClientCyphers.of(sendIv, recvIv)` 绑定到具体协议实现。  
3. **`writeInitialUnencryptedHelloPacket`**：在**尚未加密**的链路上，按 `ServerConstants.VERSION` 把「版本握手 / IV 通告」写给客户端——客户端必须先完成这步，后面的 `PacketCodec` 才能双方对齐。

随后 **`setUpHandlers`** 向 pipeline 追加（顺序很重要）：

| 顺序名 | 类型 | 作用 |
|--------|------|------|
| `IdleStateHandler` | Netty 内置 | 全连接 **30 秒**无读写则触发 idle 事件（`Client.userEventTriggered` 里处理）。 |
| `PacketCodec` | `CombinedChannelDuplexHandler` | 入站 **`PacketDecoder`**、出站 **`PacketEncoder`**，把字节流 ↔ `InPacket`/`Packet`。 |
| `SendPacketLogger` / `ReceivePacketLogger` | 插在 `Client` 两侧 | 出站/入站包日志（运维抓包、审计）。 |
| `Client` | `ChannelInboundHandlerAdapter` | **业务入口**：收到解码后的 `InPacket` 后进入 `channelRead`。 |

摘录（上游 `ServerChannelInitializer`）：`initPipeline` 内生成 IV → `ProtocolFactory` → `writeInitialUnencryptedHelloPacket`；`setUpHandlers` 依次挂 `IdleStateHandler`、`PacketCodec`、`Client`，并在 `Client` 两侧插入收发包 Logger。

### 5.4 收包主路径：`Client.channelRead`

`Client` **既是** Netty 的 **`ChannelInboundHandlerAdapter`**，**又是** Maple 语义上的「会话」：持有账号、角色、世界、频道、脚本引擎、锁等。

`channelRead` 的核心逻辑（务必读源码）：

1. 若不是 `InPacket`，记 warn 并丢弃。  
2. **`short opcode = packet.readShort()`**：客户端收包的第一个逻辑字段被当作 **RecvOpcode 数值**。  
3. **`packetProcessor.getHandler(opcode)`**：在 **本连接绑定的** `PacketProcessor` 的稀疏数组里取 `PacketHandler`。  
4. 可选：若打开 `use_debug_show_rcvd_packet`，打印 opcode 与 payload（排除忽略列表）。  
5. 若 `handler != null` 且 **`handler.validateState(this)`** 为真：  
   - `ThreadLocalUtil.setCurrentClient(this)`：让深层调用链能通过 ThreadLocal 找到当前会话（脚本、日志、部分工具类常用）。  
   - `MonitoredChrLogger.logPacketIfMonitored(...)`：对「被监控角色」额外记包。  
   - **`handler.handlePacket(packet, this)`**：真正业务。  
   - `catch Throwable`：打 warn（含角色名、地图名），并 **`enableActions()`** 解除客户端假死（说明作者预期 Handler 内可能抛错，不能卡死整条连接）。  
   - `finally`：`ThreadLocalUtil.removeCurrentClient()`。  
6. **`updateLastPacket()`**：刷新「最后一次收包时间」，供防挂机、统计或踢线逻辑使用。

若 `handler == null` 或 `validateState` 为假：**静默丢弃**（不抛异常），这是 Maple 系服务端常见策略——未知 opcode 或状态不允许时直接忽略。

摘录（上游 `org.gms.client.Client#channelRead`）：读 `short opcode` → `packetProcessor.getHandler(opcode)` → `validateState` → `ThreadLocalUtil` 设当前会话 → `handlePacket` → `finally` 清 ThreadLocal；异常时 `enableActions()`；末尾 `updateLastPacket()`。

### 5.5 `PacketProcessor`：opcode → Handler 表是如何建起来的

- 构造时先扫 **`RecvOpcode.values()`** 找最大 opcode 值，分配 **`handlers[max+1]`** 数组。  
- **`getProcessor(world, channel)`** 用字符串 `"world channel"` 做 key **缓存** `PacketProcessor` 实例；同一逻辑频道共用一张表。  
- **`reset(int channel)`**：清空数组后，根据 **`ServerConstants.VERSION`**（例如 83）调用：  
  - **`registerCommonHandlers()`**：所有线都有的，如 `PONG`、`CUSTOM_PACKET`。  
  - 若 **`channel < 0`**：`registerLoginHandlers()`（登录 opcode 全集）。  
  - 否则：`registerChannelHandlers()`（游戏 opcode 全集，体积极大）。  

因此：**登录连接与频道连接用的是不同 `PacketProcessor` 实例、不同 Handler 表**；读 Handler 时先看连接是 `Type.LOGIN` 还是 `Type.CHANNEL`。

```108:136:BeiDou-Server/gms-server/src/main/java/org/gms/net/PacketProcessor.java
    public synchronized static PacketProcessor getProcessor(int world, int channel) {
        final String processorId = world + " " + channel;
        PacketProcessor processor = instances.get(processorId);
        if (processor == null) {
            processor = new PacketProcessor();
            processor.reset(channel);
            instances.put(processorId, processor);
        }
        return processor;
    }

    public void reset(int channel) {
        handlers = new PacketHandler[handlers.length];

        switch (ServerConstants.VERSION) {
            case 83:
                registerCommonHandlers();

                if (channel < 0) {
                    registerLoginHandlers();
                } else {
                    registerChannelHandlers();
                }
                break;
```

### 5.6 断线与异常：`channelInactive` / `exceptionCaught`

- **`channelInactive`**：连接从 Netty 角度已断开 → `closeMapleSession()`。  
- **`closeMapleSession`**：按 `Type` 调用 `SessionCoordinator.closeLoginSession` 或 `closeSession`；若非「切线中」则 `disconnect(...)`，最后 **`ioChannel.close()`**。  
- **`exceptionCaught`**：  
  - 若玩家存在且「未在世界登录完成」等条件，会触发 **`sysRescue.setMapChange(player)`**（解救卡地图）。  
  - 若是 **`InvalidPacketHeaderException`**：认为协议头非法，**强制关会话**。  
  - 若是 **`IOException`**：走 `closeMapleSession()`。

## 6. `Client` 会话对象：字段在语义上各代表什么（精读）

下面按**职责分组**解释 `Client` 中常见字段（非 exhaustive，读源码时以实际定义为准）。

### 6.1 身份与连接

| 字段 / 概念 | 含义 |
|-------------|------|
| **`type`**（`LOGIN` / `CHANNEL`） | 决定断线时走 `SessionCoordinator` 哪条清理路径、以及部分 Handler 是否允许执行。 |
| **`sessionId`** | 本 TCP 连接在进程内的递增编号，便于区分同 IP 多开、查日志。 |
| **`packetProcessor`** | 本条连接绑定的 opcode→Handler 表；登录与频道实例不同。 |
| **`ioChannel`** | Netty `Channel`，用于 **`writeAndFlush`** 发包、主动关连接。 |
| **`remoteAddress`** | 字符串形式客户端 IP（与限流、封禁、SessionCoordinator 一致）。 |
| **`inTransition`** | 是否处于「登录服 ↔ 频道服」等切换中间态；为 true 时 `channelInactive` 里避免重复 `disconnect`（历史 bugfix 注释有说明）。 |

### 6.2 账号与鉴权状态

| 字段 | 含义 |
|------|------|
| **`accId` / `accountName`** | 当前会话绑定的账号；登录过程中逐步填充。 |
| **`loggedIn` / `serverTransition`** | 是否已完成某阶段登录、是否处于服务器要求的「过渡」状态（与 `validateState` 强相关）。 |
| **`loginattempt` / `pin` / `pic` 等** | 登录安全流程中的计数与缓存（防爆破、PIN/PIC 流程）。 |
| **`gmlevel`** | GM 权限级别，影响可用命令与部分校验。 |
| **`macs`** | 收集到的机器标识集合（反作弊、审计）。 |
| **`hwid`**（类型 `Hwid`） | 与 **`SessionCoordinator`** 协同：多开限制、登录与游戏会话衔接。 |

### 6.3 游戏内状态

| 字段 | 含义 |
|------|------|
| **`player`**（`Character`） | 进频道并选角成功后才有；绝大多数游戏 Handler 操作的是 `Character`（地图、技能、任务、背包等）。 |
| **`world` / `channel`** | 逻辑世界与频道号；频道 `Client` 与 `Server.getChannel(world, channel)` 一致才有意义。 |
| **`lastPong` / `lastPacket`** | 心跳与收包时间；idle 检测与防挂机逻辑会用到。 |

### 6.4 并发与脚本

| 字段 | 含义 |
|------|------|
| **`actionsSemaphore` / `lock` / `encoderLock` / `announcerLock`** | 限制并发动作、保护发包顺序、广播公告等；Maple 系服务端常见「防客户端并发点穿」。 |
| **`engines`** | 每个会话可挂多个 `ScriptEngine`（NPC/任务/事件脚本）；key 为脚本上下文标识。 |

### 6.5 与 `PacketHandler.validateState` 的关系

- **`AbstractPacketHandler`** 默认要求 **`c.isLoggedIn()`** 为真才处理——因此大量「频道内」Handler 继承它即可自动拒绝未登录连接。  
- **登录线**上的 Handler 往往**自己实现** `PacketHandler` 或重写 `validateState`，否则会被默认逻辑挡掉。

## 7. `SessionCoordinator`：登录与游戏会话如何防多开、如何清理

该类是**单例**（`getInstance()`），内部维护：

- **`onlineClients`**：`accountId → Client`，表示**已占用账号槽位**的会话（注意注释：关登录会话时不能误删已在游戏的会话）。  
- **`onlineRemoteHwids`**：当前在线 HWID 集合。  
- **`loginRemoteHosts`**：`remoteHost 字符串 → Client`，用于同一主机上登录会话互斥。  
- **`hostHwidCache`**：主机/IP 与 HWID 的缓存与过期清理。

**`canStartLoginSession`**（配置项 `deterred_multi_client` 为 true 时生效）：  
通过 `SessionInitialization` 对 `remoteHost` 做临界区保护，然后检查「已知 HWID 是否已在线」「同 remoteHost 是否已有登录会话」，通过则 **`loginRemoteHosts.put(remoteHost, client)`** 并放行。

**`closeLoginSession`**：清 `loginRemoteHosts`、若曾设置 HWID 则从 `onlineRemoteHwids` 移除，并在**会话 id 匹配**时才从 `onlineClients` 移除（避免删掉已进游戏的会话）。

**`attemptLoginSession` / `attemptGameSession`**：在真正验证账号密码成功后、或从登录切到游戏时，进一步协调 HWID、数据库中的 HWID 关联表、以及「同 HWID 是否已在线」等逻辑。

精读建议：先读 **`canStartLoginSession` → `closeLoginSession` → `attemptGameSession`** 三条路径，再下钻 `SessionInitialization`、`LoginStorage`、`SessionDAO`。

## 8. `Server` 单例：游戏世界运行时「总枢纽」（边界说明）

`org.gms.net.server.Server` 文件极长，不适合线性通读。从**架构**上把它理解成：

1. **进程内全局状态容器**：`World` 列表、`Channel` 映射、公会/联盟缓存、排榜、各类 `Task` 调度入口等。  
2. **与 Spring 的桥接**：通过 **`ServerManager.getApplicationContext().getBean(...)`** 拉取 `AccountService`、`CharacterService` 等——说明游戏逻辑会**间接调用**你在管理端熟悉的那套 Service（注意事务边界与线程模型）。  
3. **时间源**：`currentTime` / `serverCurrentTime` / `update_interval` 配置，供游戏内时间与定时任务对齐。  
4. **在线标志**：`volatile boolean online`（Lombok `@Getter/@Setter`），`Client.channelActive` 首行检查 **`Server.getInstance().isOnline()`**（Lombok 对 `boolean` 生成 `isOnline()`），若服未标记在线则直接关连接——即：**Netty 可以已 bind，但业务上仍可选择拒绝新连接**。

**`getChannel(world, channel)`**：从 `worlds` 读锁取世界，再取频道；任何一步 NPE 都吞掉返回 `null`——因此 **`ChannelServerInitializer`** 里必须显式判断 `null` 关连接，否则玩家会连进「无频道对象」的僵尸状态。

## 9. REST 管理 API（`gms-ui` ↔ `controller` ↔ Security ↔ Service ↔ DB）

本节描述 **HTTP 管理端**（与 Netty 游戏连接并列跑在同一 `gms-server` 进程内）的请求如何被鉴权、如何落到业务、如何访问数据库。读代码时建议打开：`SpringSecurityConfig`、`AuthTokenFilter`、`AuthController`、`AccountService`、`AccountsMapper`。

### 9.1 URL 版本前缀

`ApiConstant.LATEST` 当前为 **`v1`**。各 `*Controller` 的路径形如：

- `/auth/v1/login`
- `/account/v1/info`
- `/server/v1/...` 等

因此 **gms-ui** 里请求的 base path 必须带 **`/v1`**（或你们若改常量需前后端一起改）。

### 9.2 Spring Security 总规则：`SpringSecurityConfig`

`SpringSecurityConfig#filterChain` 要点：

| 配置项 | 含义 |
|--------|------|
| **CORS** | `Customizer.withDefaults()`，允许浏览器跨域时的预检等（具体规则由 Spring 默认 + 项目其它 CORS 配置叠加）。 |
| **CSRF 关闭** | 典型 **JWT 无状态 API** 写法；依赖 HTTPS、Token 与业务校验防跨站。 |
| **无 Session** | `SessionCreationPolicy.STATELESS`：不在服务端存 HTTP Session，身份只靠 JWT（或放行路径）。 |
| **放行路径** | `/auth/**`、`/swagger-ui/**`、`/v3/api-docs/**`、静态首页与 `/assets/**` **匿名可访问**。 |
| **其余路径** | `anyRequest().authenticated()`：必须存在已认证的 `Authentication`（由 `AuthTokenFilter` 填入，见下）。 |
| **`AuthTokenFilter` 位置** | `addFilterBefore(..., UsernamePasswordAuthenticationFilter.class)`：每个请求先尝试从 `Authorization` 解析 JWT 并注入 `SecurityContext`。 |

**注意**：`/auth/**` 在 Security 层已 `permitAll()`，但 **`AuthTokenFilter` 内对 `/auth/` 前缀直接 `doFilter` 返回**，避免对登录接口再套一层无意义校验。

### 9.3 JWT 注入身份：`AuthTokenFilter`

对每个非 `/auth/` 请求：

1. **`parseJwt`**：读 `Authorization` 头，若以 `Bearer ` 开头则取后半段为 raw JWT。  
2. **Swagger 测试后门（务必重视）**：若 SpringDoc 与 Swagger UI **均启用**，且 token 字面量等于 **`swagger`**，则**不校验 JWT**，直接加载用户名为 **`admin`** 的 `UserDetails` 并写入 `SecurityContext`。注释写明生产须关 Swagger，否则「裸奔」。  
3. **正常路径**：`jwtUtils.validateJwtToken(jwt)` 为真时，从 token 解 **`subject` 作为用户名**，`userDetailsService.loadUserByUsername(username)`，构造 **`UsernamePasswordAuthenticationToken`** 放入 **`SecurityContextHolder`**。

若以上分支都未设置 `Authentication`，则对于需 `authenticated()` 的接口，后续会走 **`AuthEntryPointJwt`** 返回 401（未把 `AuthTokenFilter` 标为认证过滤器时，常见表现是「无 Authentication → 入口拒绝」）。

### 9.4 JWT 本身：`JwtUtils` 与 `AuthService`

- **`JwtUtils.generateJwtToken(username)`**：`sub = username`，签发时间 + 过期时间由配置 **`jwt.duration`**，算法 **HS512**，密钥 **`jwt.secret`**。  
- **`AuthService.getToken(name, password)`**：  
  1. `accountService.findByName(name)` 查 **`AccountsDO`**。  
  2. `RequireUtil.requireFalse(account == null || !accountService.checkPassword(password, account), ...)`：账号不存在或密码错则抛业务异常（由全局异常处理转成统一错误响应）。  
  3. 成功则返回 `Map`：`{"token": "<jwt>"}`。  
- **`refreshToken`**：若 `Authorization` 为 `Bearer ...`，去掉前缀后解析用户名，再查库存在则签发**新 JWT**。

**与游戏登录密码算法的关系**：`AccountService.checkPassword` 同时支持 **BCrypt**、**明文**、**SHA-1/SHA-512 十六进制** 等历史形态；`encryptPassword` 则根据 `GameConfig` 的 **`bcrypt_migration`** 决定新密码写 BCrypt 还是 `BCrypt.hashpwSHA512`。管理端 REST 与游戏端共用 **`accounts` 表**时，密码形态必须一致，否则一端能登另一端不能。

### 9.5 「谁能登录管理后台」：`UserDetailsServiceImpl` 与 `webadmin`

`UserDetailsServiceImpl#loadUserByUsername`：

- 用 `AccountsMapper` 按名查 `AccountsDO`。  
- **仅当** `user.getWebadmin() != null && user.getWebadmin() == 1` 时，赋予 **`ROLE_ADMIN`** 并返回 **`UserDetailsImpl`**。  
- 否则返回 **`null`**。

这意味着：**游戏账号默认不能靠 JWT 变成「已认证用户」去调受保护的 REST**，除非该账号在数据库里被标成 **Web 管理员**（`webadmin=1`）。与「先登录拿 token」并不矛盾：登录接口只验证账号密码发 JWT；**后续带 JWT 的请求**仍要能在 `loadUserByUsername` 里还原出 `UserDetails`，否则 `SecurityContext` 为空，接口 401。

实践中：给需要进 **gms-ui** 的账号在库里开 **`webadmin`**，再用 `/auth/v1/login` 取 token。

### 9.6 HTTP 业务闸（第二层）：`ServerFilter`（与 4 节呼应）

`ServerFilter` 在 **Spring Security 之后**执行（见该类注释）：解析真实 IP → **`AccountService.isBanned(ip)`**（查 **`ipbans` 表计数**）→ **全局限流** `RateLimitUtil`。  
这与 **Netty 连接级** `RateLimitUtil` 是两套调用点：REST 按 Servlet 请求 IP；游戏按 Socket 远端 IP。

另：`ServerFilter` 对非 multipart 请求使用 **`CachedHttpServletRequest`** 缓存 body，便于后面 Controller/Advice 多次读取 JSON。

### 9.7 典型 Controller：`AuthController` 与 `AccountController`

**`AuthController`（`/auth`）**

- `POST /auth/v1/login`：请求体为 **`SubmitBody<Map<String,String>>`**，从中取 `username`、`password` 调 `AuthService.getToken`。  
- `DELETE /auth/v1/logout`：当前实现直接 `ResultBody.success()`（无服务端 session 时多为占位；前端删 token 即可）。  
- `GET /auth/v1/refreshToken`：请求头 `Authorization: Bearer <旧token>`，`AuthService.refreshToken`。

**`AccountController`（`/account`）**

- `GET /account/v1/info`：**当前登录用户**信息 → `accountService.getCurrentUser()`。  
- `GET /account/v1`：分页 + 多条件筛选账号列表 → `getAccountList`（未传 `size` 时默认为 **`Integer.MAX_VALUE`**，管理端列表慎用）。  
- `POST/PUT/DELETE`：注册、用户自助改资料、GM 改资料、删号、重置在线、封停/解封等，全部**薄封装**到 `AccountService`。

Controller 层几乎无业务：参数绑定 + 调 Service + **`ResultBody`** 统一包装。

### 9.8 Service 与游戏世界的交界：`AccountService` 精读片段

除常规 CRUD 外，下列方法体现 **REST 与 Netty 游戏进程共享同一 JVM**：

- **`banAccount(accountId, reason)`**：更新 `accounts` 封禁标记后，**遍历该账号下角色**，若某角色在 **`Server.getInstance().getWorlds().get(world).getPlayerStorage()`** 中在线，则：内存 `setBanned`、**`banMacs`**、插入 **`ipbans`**、**`client.disconnect`**。即：**管理后台一条封禁 API 会直接踢掉游戏内在线角色并写 IP 封禁表**。  
- **`isBanned(String ip)`**：给 `ServerFilter` 用，查 `ipbans` 是否存在该 IP。  
- **`getCurrentUser()`**：从 **`SecurityContextHolder`** 取 `UserDetails.getUsername()`，再按名查库——因此 **JWT 的 subject 必须与 `accounts.name` 一致**。

### 9.9 数据访问层：`AccountsMapper` 等

`AccountService` 通过 **MyBatis-Flex** 的 `*Mapper`（如 `AccountsMapper`）访问表 **`accounts`** 及关联查询；具体 SQL/XML 在 `resources` 下对应 mapper 文件（精读时可从 `AccountsMapper` 接口跳到 XML/注解 SQL）。

### 9.10 与 `gms-ui` 的联调方式（操作向）

1. 确保账号 **`webadmin=1`**，密码能通过 **`checkPassword`**。  
2. `POST /auth/v1/login` 拿 `token`。  
3. 后续请求带 **`Authorization: Bearer <token>`**。  
4. 生产环境：**关闭 Swagger** 或至少关闭 `AuthTokenFilter` 中的 **`swagger` 字面量后门**，并配合 HTTPS。

## 10. 游戏登录：`LoginPasswordHandler` → `Client.login` → 会话与库状态

本节对应 **登录服 Netty 连接**（`Type.LOGIN`）上 **`RecvOpcode.LOGIN_PASSWORD`** 的处理链，与 **第 5、6、7 节**（`Client.channelRead`、`SessionCoordinator`）直接衔接。主类：`LoginPasswordHandler`、`Client#login`、`Client#finishLogin`、`Client#updateLoginState`、`Server#registerLoginState`。

### 10.1 `LoginPasswordHandler`：谁在什么状态下能发登录包

- **`validateState`**：`return !c.isLoggedIn()` —— 只有「尚未在会话层标记为已登录」时才处理下一个登录包；已登录则忽略（避免重复提交）。  
- **`handlePacket` 顺序**（建议对照源码一行行看）：

| 步骤 | 行为 |
|------|------|
| 1 | 若 `remoteAddress` 为字面 `"null"`，发 **`getLoginFailed(14)`** 并返回（无法信任来源 IP）。 |
| 2 | 从包体读 **`login`**、**`pwd`**，并 **`c.setAccountName(login)`**。 |
| 3 | **`p.skip(6)`**：跳过客户端填的 6 字节占位。 |
| 4 | 读 **4 字节 HWID 片段**，用 **`HexTool.toCompactHexString`** 转成 **`Hwid`** 对象。 |
| 5 | **`loginok = c.login(login, pwd, hwid)`**：核心校验与多开协调（见 **10.2**）。 |
| 6 | **自动注册**：若配置 **`automatic_register`** 且 `loginok == 5`（无账号），则 `INSERT INTO accounts` 后再次 `login`。 |
| 7 | **BCrypt 迁移**：若 **`bcrypt_migration`** 且 `loginok` 为 **-10**（待迁移）或 **-23**（未接受 TOS 与迁移组合），则 `UPDATE accounts SET password = ?` 后把 `loginok` 归一成 **0** 或 **23**。 |
| 8 | **封禁与临时封禁**：`hasBannedIP` / `hasBannedMac` → 失败码 **3**；`tempban` 未过期则发临时封禁包。 |
| 9 | **`loginok == 3`**：永久封禁包；**`loginok != 0`**：通用 `getLoginFailed(loginok)`。 |
| 10 | **`c.finishLogin() == 0`**：提交「登录完成」状态（见 **10.3**），然后 **`checkChar(accId)`**（可选踢掉同账号已在线角色，见 **10.4**），最后 **`login(c)`** 发 **`getAuthSuccess`** 并 **`Server.registerLoginState(c)`**。否则发失败码 **7**。 |

`GuestLoginHandler` 会**复用** `new LoginPasswordHandler().handlePacket(p, c)`，即访客入口与正式登录共用同一套校验分支。

### 10.2 `Client.login(login, pwd, hwid)`：数据库、`loginok` 与 `SessionCoordinator`

**`loginattempt`**：每次进入 `login` 自增；**超过 4** 则 `loggedIn = false`、`SessionCoordinator.closeSession(this, false)` 并返回 **6**（防暴力尝试）。

**查库**：`SELECT id, password, gender, banned, pin, pic, characterslots, tos, language FROM accounts WHERE name = ?`。

**`loginok` 含义（读代码归纳，与客户端错误码表对应）**：

| 场景 | `loginok` / 行为 |
|------|-------------------|
| 无行 / 无效 id | `accId` 置 **-3** 等，最终常表现为 **5**（需自动注册时由 Handler 处理） |
| **`banned == 1`** | 直接 **3** |
| 已处于「登录状态」(`getLoginState() > NOTLOGGEDIN`) | **7**（重复登录） |
| **`use_debug` + `no_password`** | 直接 **0**（开发跳过密码） |
| BCrypt 密码匹配 | 若 **`tos == 0`** → **23**（需先接受服务条款）；否则 **0** |
| 明文或 SHA-1/SHA-512 匹配 | 若未接受 TOS：无迁移时 **23**，有 **`bcrypt_migration`** 时 **-23**；已接受 TOS：无迁移 **0**，有迁移 **-10**（触发 Handler 里 UPDATE 密码） |
| 密码错误 | **4**，并 `loggedIn = false` |

当 **`loginok == 0` 或 `loginok == 4`** 时，会调用 **`SessionCoordinator.attemptLoginSession(this, hwid, accId, loginok == 4)`**：

- **`loginok == 4`** 时第四个参数为 **true**（routine check），源码注释指出：在 **`deterred_multi_client == true`** 时，密码错一次也可能映射成 **`REMOTE_REACHED_LIMIT`**，客户端需重开——这是**有意权衡**还是边界问题，运维时要知晓。  
- 返回值映射：`SUCCESS` 保持原 `loginok`；`REMOTE_LOGGEDIN`→**17**；`REMOTE_REACHED_LIMIT`→**13**；`REMOTE_PROCESSING`→**10**；`MANY_ACCOUNT_ATTEMPTS`→**16**；其它→**8**。  
- `SUCCESS` 且密码正确时 **`loginattempt = 0`**。

**与 REST 的差异**：此处密码校验在 **`Client.login`** 内手写 JDBC + `BCrypt.checkpw` / `checkHash`，**不经过** `AccountService.checkPassword`；两套逻辑应对**同一 `accounts.password` 列**的多种历史格式，修改密码算法时需**两处**或抽公共方法对齐。

### 10.3 `finishLogin` 与 `updateLoginState`：`loggedin` 列与内存标志

**`finishLogin`**：在 **`encoderLock`** 内若 `getLoginState() > LOGIN_NOTLOGGEDIN`（已在过渡或已登录），则 **`loggedIn = false`** 并返回 **7**；否则 **`updateLoginState(LOGIN_LOGGEDIN)`** 并返回 **0**。

**`updateLoginState(newState)`**：

1. 若新状态为 **`LOGIN_LOGGEDIN`**：调用 **`SessionCoordinator.updateOnlineClient(this)`**，把本会话登记为「该账号当前在线客户端」（供顶号、防多开等）。  
2. **`UPDATE accounts SET loggedin = ?, lastlogin = ? WHERE id = ?`**：`loggedin` 写库，`lastlogin` 用 **`Server.getInstance().getCurrentTime()`**（而非 SQL `CURRENT_TIMESTAMP`，注释说明避免破坏登录流程）。  
3. 若 **`LOGIN_NOTLOGGEDIN`**：`loggedIn = false`、`serverTransition = false`、**`setAccID(0)`**（彻底登出）。  
4. 其它状态：**`serverTransition = (newState == LOGIN_SERVER_TRANSITION)`**，**`loggedIn = !serverTransition`**（过渡态时内存上视为未完全登录）。

**`getLoginState()`**：从库读 `loggedin`；若值为 **`LOGIN_SERVER_TRANSITION`** 且 **`lastlogin` 为空或早于当前时间 30 秒以上**，认为卡死过渡态，**自动写回 `NOTLOGGEDIN`** 并恢复 `accId`（大量历史 bugfix 注释）。最终根据状态设置内存 **`loggedIn`**。

因此：**`accounts.loggedin` 是跨进程/跨重启的「账号是否卡在登录/游戏中」的粗粒度锁**；与 **`SessionCoordinator` 内存表** 共同约束多开。

### 10.4 `checkChar(accId)`：防止同账号多角色同时在线

若配置 **`use_character_account_check`**：遍历所有 **`World`** 的 **`PlayerStorage`**，若发现 **`chr.getAccountId() == accid`**，则 **`forceDisconnect`**、从存储移除。用于修复「同账号多角色重复登录」类问题。

### 10.5 `login(c)` 私有方法与 `registerLoginState`

成功路径末尾：**`PacketCreator.getAuthSuccess(c)`** 告知客户端鉴权成功；**`Server.getInstance().registerLoginState(c)`** 把该 `Client` 放入 **`inLoginState`** Map，值为 **当前时间 + 10 分钟**——供 **`disconnectIdlesOnLoginState`** 等逻辑踢掉「卡在选角界面过久」的连接。

## 11. 进频道：`PlayerLoggedinHandler`、`Character.loadCharFromDB` 与 Spring `CharacterService`

本节对应 **频道服**（`RecvOpcode.PLAYER_LOGGEDIN`）——玩家从登录流程切到某 **world/channel** 后的 TCP 连接上，服务端把 **角色 id** 对应的数据hydrate 成内存 **`Character`**，并挂到 **`Client.player`**、写入 **`PlayerStorage`** / 地图可见性。与 **第 10 节**衔接：选角成功后库中 `accounts.loggedin` 常为 **`LOGIN_SERVER_TRANSITION`（1）**；本 Handler 末尾 **`updateLoginState(LOGIN_LOGGEDIN)`** 把「真进游戏」落定。

### 11.1 与 `AbstractPacketHandler` 相反的 `validateState`

`PlayerLoggedinHandler` **重写** `validateState` 为 **`return !c.isLoggedIn()`**（与 `AbstractPacketHandler` 默认的「必须 `isLoggedIn()`」相反）。

原因：频道上新建立的 **`Client`** 在内存里 **`loggedIn` 仍为 `false`**（`isLoggedIn()` 读的是内存字段；见 `Client#isLoggedIn`），但库侧已处于**过渡态**，必须允许处理 **`PLAYER_LOGGEDIN`**。若未重写，会被父类挡掉。

### 11.2 并发：`tryacquireClient` 与 `attemptingLoginAccounts`

- 入口 **`c.tryacquireClient()`** 失败：发 **`getAfterLoginError(10)`** 并结束（**`Semaphore` + 锁**防同连接重入）。  
- 静态 **`attemptingLoginAccounts`**：`tryAcquireAccount(accId)` 保证**同一账号**不会并发跑两套「读 `getLoginState` + `updateLoginState`」，避免双登把 `loggedin` 写乱。

### 11.3 解析 world / channel、取已在线角色或加载 DB

1. **`cid = p.readInt()`**：包内角色 id。  
2. 取 **`World wserv`**、**`Channel cserv`**；若频道 null 会尝试回落到 **channel 1**，仍失败则 **disconnect**。  
3. **`player = wserv.getPlayerStorage().getCharacterById(cid)`**：若该角色对象**已在内存**（例如切频道复用），走 **老 `Client` 换绑** 分支；否则 **`newcomer = true`**，需从 DB 加载。  
4. **HWID**：若 `player == null`，**`SessionCoordinator.pickLoginSessionHwid(c)`** 从登录阶段缓存取出；否则从已在线角色的 **`Client`** 取。然后 **`c.setHwid(hwid)`**，与 **第 7 节 `attemptGameSession`** 的「登录 HWID 与进游戏一致」衔接。  
5. **`server.validateCharacteridInTransition(c, cid)`**：当 **`use_ip_validation`** 为 true 时，要求 **`Server.transitioningChars`** 里该 **remoteHost** 登记的角色 id 与 **cid** 一致（**`setCharacteridInTransition`** 在选角/切服流程中先前写入）；校验通过即从 map **remove** 消费掉。关闭配置则恒 true。

### 11.4 `Character.loadCharFromDB` → `CharacterService.loadCharFromDB`

`Character.loadCharFromDB(cid, c, true)` 是薄委托：内部 **`characterService.loadCharFromDB`**（`Character` 静态字段指向 Spring **`CharacterService`** bean）。

**`CharacterService.loadCharFromDB` 要点**（`channelServer == true` 时）：

| 阶段 | 行为 |
|------|------|
| 主表 | **`findById(cid)`** → `CharactersDO`，**`Character.fromCharactersDO(..., client)`** 得到领域对象骨架。 |
| 地图 | **`MapManager.getMap(mapId)`**，null 则落到 **`MapId.HENESYS`**；按 **`initialSpawnPoint`** 取 **Portal**，修正 **`position`**。 |
| 社交 | 从 **World** 挂 **Party**、**Messenger**（若库里有有效 id）。 |
| 状态 | **`chr.setLoggedIn(true)`**（角色对象层已登录标志，与 `Client.loggedIn` 不同）。 |
| 任务/技能 | `questService` 拉 **QuestStatus**；`skillsMapper` 填 **`SkillFactory` + SkillEntry**；冷却、疾病等读表后部分转存 **`PlayerBuffStorage`** 再删表行。 |
| 其它 | 技能宏、键位、保存点位置、好友、**账号仓 `Storage`**（`world.loadAccountStorage`）、**`reapplyLocalStats`**、**`changeHpMp`**。 |

若 **`channelServer == false`**（例如仅查角色展示），只做到 **`fromCharactersDO`** 即返回，不绑频道地图工厂。

### 11.5 绑定 `Client`、校验 `LOGIN_SERVER_TRANSITION`、写回 `LOGIN_LOGGEDIN`

1. **`c.setPlayer(player)`**、**`c.setAccID(player.getAccountId())`**。  
2. 在 **`tryAcquireAccount(accId)`** 保护下读 **`state = c.getLoginState()`**（**读库**，见第 10.3 节）：  
   - 若 **`state != LOGIN_SERVER_TRANSITION`** 或业务不允许：清空 **`setPlayer(null)`、`setAccID(0)`**；若已是 **`LOGIN_LOGGEDIN`** 则 **disconnect**，否则发 **`getAfterLoginError(7)`**。  
   - 通过则 **`c.updateLoginState(LOGIN_LOGGEDIN)`**（再次写库、**`SessionCoordinator.updateOnlineClient`**、内存 **`loggedIn = true`**）。  
3. **`!newcomer`**：`player.newClient(c)` —— 把 **`Character.client`** 指到新 `Client`，从 **`c.getChannelServer().getMapFactory()`** 重取地图与出生点（**`Character#newClient`**）。**`newcomer`** 则角色已在 `loadCharFromDB` 里绑好地图。

### 11.6 进图与大量「登录后同步」

成功通过状态校验后，Handler 执行一长串**与客户端状态机对齐**的发包与服务器侧登记（摘逻辑块，便于你对照源码）：

- 可选 **`HpMpAlertService`** + **`PacketCreator.updateClientSettings`**（仅本人）。  
- **`cserv.addPlayer` / `wserv.addPlayer`**：进入 **频道与世界级玩家存储**。  
- **`PlayerBuffStorage`** 取 Buff / Disease，**`silentGiveBuffs` / `silentApplyDiseases`**。  
- **`getCharInfo`**、GM 隐身、**键位/快捷栏/宏**、**`addPlayer` 到 `MapleMap`**、好友在线同步、家族/公会/联盟、**`noteService.show`**、组队 HUD、**穿戴装备 `equippedItem`**、技能/宠物/坐骑、公告、定时任务注册（buff/disease/cooldown 等）、倍率/优惠券、婚礼伴侣通知、事件实例 **`EventRecallCoordinator`**、**`setNPCScriptable`**、**`setLoginTime`** 等。  
- **`finally`**：**`c.releaseClient()`** 释放入口信号量。

### 11.7 与第 10 节的时序关系（一句话）

**登录服**：账号密码 OK → **`finishLogin`** 把库标成「已登录/待过渡」并发 **`getAuthSuccess`**。  
**频道服**：带 **cid** 的 **`PLAYER_LOGGEDIN`** → 校验 **过渡态 + IP/角色 id** → **`loadCharFromDB`** → **`updateLoginState(LOGIN_LOGGEDIN)`** → 进 **`PlayerStorage` + 地图** 并下发全套登录包。

## 12. `Channel`、`PlayerStorage` 与 `MapleMap`：频道内玩家在何处登记、地图如何见人见物

### 12.1 `Channel`：一个逻辑频道实例里有什么

`org.gms.net.server.channel.Channel` 表示「某 **world** 下的第 **channel** 号频道」的运行时容器，构造函数里会做几件大事：

| 成员 / 初始化 | 作用 |
|----------------|------|
| **`port` / `ip`** | **`BASE_PORT 7575` + `(channel-1) + world*100`** 得到监听端口；`ip = serviceProperty.getWanHost() + ":" + port`，发给客户端连接用。 |
| **`MapManager mapManager`** | **`new MapManager(null, world, channel)`**：本频道内所有 **`MapleMap`** 实例的工厂/缓存入口（具体加载 WZ/Reactor 等在 `MapManager` 内）。 |
| **`ChannelServer channelServer`** | **`initServer`** → **`new ChannelServer(port, world, channel).start()`**，即本节 Netty **bind 的端口**与 **`Channel`** 字段一致（与 **精读第 5 节** Initializer 一致）。 |
| **`PlayerStorage players`** | 本频道**当前在线角色**索引（按 id / 名字小写）。 |
| **`eventSM`、`services`、`hiredMerchants`、`dungeons`** 等 | 事件脚本、频道服务、雇佣商店、小黑屋等业务状态。 |

精读 **`PlayerLoggedinHandler`** 时你已看到 **`cserv.addPlayer(player)`**：对应 **`Channel.addPlayer`** —— **`players.addPlayer(chr)`** 并 **`chr.sendPacket(serverMessage)`**（全频道公告条）。 **`getMapFactory()`** 即 **`mapManager`**，供 **`CharacterService.loadCharFromDB`**、`changeMap` 等取 **`MapleMap`**。

### 12.2 `PlayerStorage`：频道级「谁在线」

`org.gms.net.server.PlayerStorage`：

- **`storage`**：`id → Character`，**`nameStorage`**：`name.toLowerCase() → Character`。  
- 所有读写用 **`ReentrantReadWriteLock`** 包 **`wlock` / `rlock`**。  
- **`addPlayer`** / **`removePlayer`** / **`getCharacterById`、`getCharacterByName`**：供 **`Channel`**、组队查找、管理员命令等使用。  

**与世界级索引的关系**：同名角色**不应**同时出现在同一 `World` 下两个频道，但数据结构上是 **频道各有一份 `PlayerStorage`**；跨频道找人由 **`World.getPlayerStorage`**（世界级 **`players`**）再 **`Character.getClient().getChannelServer()`** 配合（见 **`World.removePlayer`** 会先 **`cserv.removePlayer`** 再找错频道兜底）。

### 12.3 `World.addPlayer` / `World.removePlayer`（与频道配合）

- **`World.addPlayer(chr)`**：仅 **`players.addPlayer(chr)`**（世界级注册）。  
- **`World.removePlayer(chr)`**：先按 **`chr.getClient().getChannelServer()`** 从**当前频道** `removePlayer`；若不在（状态不一致），**遍历所有 `Channel`** 尝试移除；最后在 **world 级 `PlayerStorage`** 里 **`removePlayer(chr.getId())`**。

因此：**频道 `PlayerStorage` = 快速列本频道在线**；**世界 `PlayerStorage` = 只要在任意频道在线就有一份引用**（与实现细节一致时需结合 `removePlayer` 路径理解）。

### 12.4 `MapleMap.addPlayer`：进图时发生什么（可见性的核心）

`MapleMap#addPlayer(chr)` 是大函数，可分块理解：

1. **`chrWLock`**：把 **`chr` 加入本图 `characters` 集合**（**本图在线玩家列表**），并更新 **组队在本图的成员索引**（`addPartyMemberInternal`）。首张图到会启动 **掉落监控、_aggroCoordinator_** 等。  
2. **`chr.setMapId(mapid)`**、**`chr.updateActiveEffects()`**；按 **字段伤害图**向 **World** 注册/移除 **HP 持续减少**监听。  
3. **`MapScriptManager`**：若配置了 **`onFirstUserEnter`** / **`onUserEnter`**，跑脚本（例如保存回城点、竞技场逻辑）。  
4. 特殊地图：**传送船/迷你地城/竞技场**时钟与 **`changeMap` 调度**；**坐骑限制**等与 **`FieldLimit`** 相关Buff取消。  
5. **对他人可见**：若 **`chr.isHidden()`**，走 **`broadcastGMSpawnPlayerMapObjectMessage`**（**仅 GM** 客户端收到 **`spawnPlayerMapObject`**，并辅以暗系外显Buff包）；否则 **`broadcastSpawnPlayerMapObjectMessage(chr, chr, true)`** —— **同图其他每一个 `Character`**（除 `source` 自己）调用 **`PacketCreator.spawnPlayerMapObject`**。  
6. **`sendObjectPlacement(chr.getClient())`**：**给刚进图的玩家**遍历 **`mapobjects`**，对 **`PLAYER/NPC/HIRED_MERCHANT/...` 等非范围类型**调 **`sendSpawnData`** —— **你看到的地图静态/动态物件**主要来自这里（其它类型如召唤兽有额外距离判断）。  
7. **`objectWLock`**：把 **`chr` 以其 `objectId` 塞进 `mapobjects`**（**地图实体表**，与 `characters` 集合互补：`characters` 专指玩家，`mapobjects` 含怪、掉落、NPC 等）。  
8. **玩家商店龙、坐骑、Summon**：按状态 **`addMapObject` / `spawnSummon` / `updateMapObjectVisibility`**。

**要点**：「**别人看见我**」主要靠 **`broadcastSpawnPlayerMapObjectMessage`**；「**我看见地图与别人**」靠 **`sendObjectPlacement`**（及后续其它 enter 发包）。GM 隐身走 **`broadcastGM*`** 分支，与普通玩家分叉。

### 12.5 `MapleMap.removePlayer`：离图广播

**`removePlayer(chr)`**：从 **`characters`** 移除、拆掉组队在本图索引、迷你地城 unregister、**`removeMapObject(chr.getObjectId())`**；然后 **`broadcastMessage` 或 `broadcastGMMessage(PacketCreator.removePlayerFromMap(...))`** 与 **`isHidden()`** 一致；再 **`chr.leaveMap()`**、处理召唤兽/龙的移除广播。

### 12.6 广播 helpers：`broadcastMessage` 家族

地图上大量 **`broadcastMessage` / `broadcastPacket` / `broadcastGMMessage`** 重载：有的**全图**、有的带 **`source` + 距离平方**（**可见范围**，与 `MapleMap` 内常量相关）。精读某技能/怪物掉落时，可从此处下钻 **「是否 ranged、是否 repeatToSource」**。

## 13. 会话过渡与 IP 校验：`setCharacteridInTransition` 调用链（衔接第 11 节）

第 11 节 **`PlayerLoggedinHandler`** 里，若 **`GameConfig.use_ip_validation`** 为 true，会调用 **`Server.validateCharacteridInTransition(c, cid)`**：只有 **`transitioningChars`** 里登记的「本客户端将要登录的角色 id」与包内 **cid** 一致才放行，并从 Map **remove** 消费掉。本节说明：**谁写入这条登记**、key 是什么、还有哪些路径复用同一机制。

### 13.1 `Server` 侧数据结构与方法

- **`transitioningChars`**：`Map<String, Integer>`，**value** 为角色 **character id**。  
- **`setCharacteridInTransition(Client client, int charId)`**：**key** = **`SessionCoordinator.getSessionRemoteHost(client)`**（见下）。  
- **`validateCharacteridInTransition`**：`use_ip_validation` 为 **false** 时直接 **`return true`**（**不读、不删** Map）；为 true 时 **`remove(key)`** 并与 **charId** 比较。  
- **`freeCharacteridInTransition`**：**remove** 并返回值 —— **`SessionCoordinator.fetchInTransitionSessionClient`** 等路径用于异常恢复。  
- **`hasCharacteridInTransition`**：是否仍存在 key（用于 **`disconnect`** 时决定是否把库 **`loggedin` 清回 `NOTLOGGEDIN`**，见 **`Client.disconnect`** 分支）。

### 13.2 Map 的 key：不是裸 IP

**`SessionCoordinator.getSessionRemoteHost(client)`**：

- 若 **`client.getHwid() != null`**：**`remoteAddress + "-" + hwid.hwid()`**  
- 否则：**`remoteAddress`**

因此 **NAT 下多用户同出口 IP** 时，只要 HWID 不同，过渡登记 **不会互相覆盖**（比单纯 IP 更稳）。

### 13.3 入口：`Client.setCharacterOnSessionTransitionState(int cid)`

该方法依次：**`updateLoginState(LOGIN_SERVER_TRANSITION)`**（写库过渡态）、**`inTransition = true`**、**`Server.getInstance().setCharacteridInTransition(this, cid)`**。

三件事：**库 `accounts.loggedin` 写成过渡态**、内存 **`inTransition = true`**（断开连接时走 **`serverTransition`** 分支避免误清全局登录）、**在 `Server` 登记「该远端会话将要登录的角色 id」**。

### 13.4 登录服：选角后发 **`getServerIP`**（最常见路径）

下列 Handler 在 **校验账号拥有该角色、世界未满、`getInetSocket` 成功** 之后，普遍顺序为：**`server.unregisterLoginState(c)`**（退出「卡在选角界面计时」）→ **`c.setCharacterOnSessionTransitionState(charId)`** → **`PacketCreator.getServerIP(..., charId)`**。

| Handler | 备注 |
|---------|------|
| **`CharSelectedHandler`** | 先发 **`attemptGameSession`**，再校验 **`haveCharacterEntry`** |
| **`CharSelectedWithPicHandler`** | 先 **`checkPic`**，再 **`attemptGameSession`**（顺序与上一类略有不同） |
| **`ViewAllCharSelectedHandler` / `WithPicHandler`** | 全角色视图下的选角 |
| **`RegisterPicHandler` / `ViewAllCharRegisterPicHandler`** | 带 PIC 注册流程 |

客户端拿到 **`getServerIP`** 后拨 **频道服** TCP；第一条业务包常为 **`PLAYER_LOGGEDIN`**（含 **cid**），频道端 **`validateCharacteridInTransition`** 对上登记即可。

### 13.5 频道内：`Character.setSessionTransitionState()`（切频道 / 现金商店返回等）

**`Character.setSessionTransitionState()`** 内部即 **`client.setCharacterOnSessionTransitionState(this.getId())`**。

1. **`Client`** 里 **换频道**逻辑（约 **`changeChannel`**）：从本地图 **`removePlayer`**、**频道 `PlayerStorage` remove**、存盘后 **`player.setSessionTransitionState()`**，再 **`getChannelChange`** 指向下一段 TCP。  
2. **`ChangeMapHandler.enterFromCashShop`**：**现金商店关闭**、`getInetSocket` 成功后 **`chr.setSessionTransitionState()`** + **`getChannelChange`**。  
3. **GM `WarpWorldCommand`**：跨世界传送类命令同样要走过渡登记。

这些场景的共同点：**同一逻辑账号要从当前频道 TCP 断到新频道 TCP**，频道端的 **`PlayerLoggedinHandler`** 仍需 **`cid` 与登记一致**（在 **`use_ip_validation`** 开启时）。

### 13.6 配置 **`use_ip_validation`** 为 false 时的行为

- **`validateCharacteridInTransition`** **恒 true**：任意 **cid** 不会因 Map 缺失被拒（仍以 **`haveCharacterEntry`**、**`attemptGameSession`** 等为其它防线）。  
- **`transitioningChars`** **仍会被 `setCharacteridInTransition` 写入**（key 可能重复覆盖）；**不会在 validate 里删除**，依赖后续 **`setCharacter...`** 覆盖或 **`freeCharacteridInTransition`** 清理。运维关闭校验时需知晓：**过渡 Map 不再是硬门禁**。

### 13.7 与 **`disconnect` / `hasCharacteridInTransition`**（顺带）

**`Client.disconnect`**：若在 **`serverTransition`** 流程中断线且 **`!Server.hasCharacteridInTransition(this)`**，会把 **`loggedin` 清回 `NOTLOGGEDIN`**，避免账号永久卡在过渡列。

## 14. `Character.changeMap` 与 Portal（从光圈到 `MapleMap.addPlayer`）

第 12 节写过 **`MapleMap.addPlayer`** 进图后的可见性与脚本；本节补全**玩家是怎么被送进那张地图的**：**客户端发包 → `ChangeMapHandler` → `Portal` → `Character.changeMap*`，最终在 `changeMapInternal` 里 `removePlayer` / `addPlayer`**。

### 14.1 Portal 在代码里是什么

- **`org.gms.server.maps.Portal`**：接口，常量 **`TELEPORT_PORTAL` / `MAP_PORTAL` / `DOOR_PORTAL`** 区分类型；提供 **`getId`、`getName`、`getPosition`、`getTarget`（目标图上 portal 名）、`getTargetMapId`、`getScriptName`、`enterPortal(Client)`** 等。  
- **`GenericPortal`**：`MapPortal` 的父类，承载 WZ 里解析出的字段；**`MapleMap`** 用 **`portals`**（`id → Portal`）管理，另可按名字查找（见下）。  
- **脚本型 portal**：**`scriptName != null`** 时 **`enterPortal` 委托 `PortalScriptManager.executePortalScript`**（每 portal 一把 **`ReentrantLock`**，避免并发重入同脚本）。  
- **数据型 portal**：无脚本且 **`targetMapId != NONE`** 时，解析 **`MapleMap to`**、目标 **`Portal pto = to.getPortal(getTarget())`**（缺失则 **`getPortal(0)`**），再 **`chr.changeMap(to, pto)`**。  

**`MapleMap#getPortal(String)`**：遍历 **`portals.values()`** 按 **`getName()`** 匹配；**`getPortal(int)`** 直接 **`portals.get(portalid)`**。名字来自地图定义，须与 **`ChangeMapHandler`** 读到的 **`portalName`** 一致。

### 14.2 客户端「踩光圈」：`ChangeMapHandler`

**`org.gms.net.server.channel.handlers.ChangeMapHandler`**（注释：**玩家通过光圈切换地图触发**）主线：

1. **已在换图中或封号**：若 **`chr.isChangingMaps()`**，打日志并发 **`enableActions()`** 返回（防重复 warp）。  
2. **`p.available() == 0`**：**从现金商店回频道** → **`enterFromCashShop`**（与第 13 节 **`setSessionTransitionState`** 衔接）。  
3. 否则读包：**死亡 / 普通 portal**、`targetMapId`、**`portalName`**、若干字节；可选 **GM 追逐模式**改坐标。  
4. **`targetMapId != -1`** 分支：**死亡复活**（Wheel / `revivePlayer` / **`respawn`**）或 **剧情/教程白名单式直连换图**（大量 **`divi` + `targetMapId` 判断**；**GM** 可直接 **`getWarpMap` + `changeMap(to, portal0)`**）。  
5. **`portal.getPortalStatus()`** 为 false：**`blockedMessage` + `enableActions`**，不进图。  
6. **`portal != null`**：**与 portal 坐标距离平方 ≤ 400000**（约 **632px**）才允许 **`portal.enterPortal(c)`**；否则只 **`enableActions`**（防远距离伪造 portal）。  
7. **`portal == null`**：**`enableActions`**（客户端报了不存在的 portal 名等情况）。

**要点**：普通地图跳转的最终执行往往在 **`GenericPortal.enterPortal` → `Character.changeMap`**；**`targetMapId != -1`** 那块更多是**死亡与新手剧情**特例。

### 14.3 `getWarpMap` vs `getMap(mapid, showMsg)`

- **`getWarpMap(int map)`**：换图解析「**去哪张 MapleMap 实例**」。若在 **`EventInstanceManager`** 内则用 **`eim.getMapInstance(map)`**；若在怪物嘉年华事件图则用嘉年华实例；否则 **`MapFactory.getMap(map)`**。事件内外可能对应**不同 MapleMap 对象**（同 map id 不同实例）。  
- **`getMap(int mapid, boolean showMsg)`**：频道 **`MapFactory.getMap`**；**null** 且 **`showMsg`** 时 **`dropMessage` + `enableActions`**。  
- **`changeMapInternal`** 里再一次 **`getMap(to.getId(), true)`** 校验：防止目标图未加载或无效。

### 14.4 `changeMap` 重载：一律落到 `changeMapInternal`

对外入口较多，关系可以记成：

| 入口 | 行为摘要 |
|------|-----------|
| **`changeMap(int map, Object pt)`** | 若有 **`EventInstanceManager`** 用 **`eim.getMapInstance`**，否则 **`getMap(map, true)`**；**`pt`** 为 **`null` → 随机玩家出生点**，**`Integer`/`String`/`Portal`** 解析 portal；否则 **`getPortal(0)`**。 |
| **`changeMap(MapleMap to, int portal)`** | **`to.getPortal(portal)`** 再交给 **`changeMap(to, Portal)`**。 |
| **`changeMap(MapleMap target, Portal pto)`** | **`canWarpCounter++`** → **`eventChangedMap`** → **`getWarpMap(target.getId())`**（事件内可能替换实际 **`to`**）→ **`pto == null` 则用 `to.getPortal(0)`** → **`changeMapInternal(to, pto.getPosition(), getWarpToMap(to, pto.getId(), …))`** → 维护 **`canWarpMap` / `canWarpCounter`** → **`eventAfterChangedMap`**。 |
| **`changeMap(MapleMap target, Point pos)`** | 同上但 warp 包用 **`0x80` + 坐标**（精确坐标落点，不绑定 portal id）。 |
| **`forceChangeMap(MapleMap target, Portal pto)`** | **绕过 `getWarpMap` 的「目标 id 再解析」**：用于必须进**参数所指 MapleMap 实例**的场景（如事件图切换）；内含 **`exitPlayer` / `registerPlayer`** 与 **`changeMapInternal(target, …)`**。 |
| **`changeMapBanish`** | 可选 **`SPIKES` 装备跳过驱逐**；记录 **banish 起点**后 **`changeMap`**，再 **`setBanishPlayerData`**。 |

**`canWarpMap` / `canWarpCounter`**（字段注释：**一次调用链路里只允许一个真正的 warp 生效**）：  
- **`changeMapInternal`** 开头 **`if (!canWarpMap) return`**。  
- 外层 **`changeMap`** 在进入 internal 前 **`canWarpMap = false`**，嵌套调用用 **`canWarpCounter`** 计数，**归零后 `canWarpMap = true`**。  
- **`warpAhead(int map)`**（供 **OnUserEnter** 等脚本在「仍在两图之间」时排队）：设 **`newWarpMap`**；**`changeMapInternal` 末尾**若 **`newWarpMap != -1`**，则 **`canWarpMap = true`** 再递归 **`changeMap(temp)`**。

### 14.5 `changeMapInternal`：真正离图 / 进图

1. **`mapTransitioning.set(true)`**（与 **`ChangeMapHandler` 的 `isChangingMaps`** 门禁一致）、**`clearTeleportDistanceContext`** 等清理。  
2. **交互收尾**：**`unregisterChairBuff`**、**`clearBanishPlayerData`**、**`Trade.cancelTrade`**、**`closePlayerInteractions`**。  
3. **`sendPacket(warpPacket)`**，然后 **`map.removePlayer(this)`**。  
4. **若频道 `PlayerStorage` 仍能按 id 找到本角色**（防 stuck）：**`map = to`、`setPosition(pos)`、`map.addPlayer(this)`、`visitMap(map)`**；否则 **warn + `disconnect`**。  
5. **组队静默更新**（**`prtLock`** 内 **`mpc.setMapId`、`SILENT_UPDATE`、`updatePartyMemberHPInternal`**）、**`silentPartyUpdateInternal`**。  
6. **`notifyMapTransferToPartner`**；若 **`newWarpMap`** 已设置则递归换图；否则 **事件门 **`recoverOpenedGate`****、**`environmentMoveList`** 等。

**`setMapTransitionComplete`** 把 **`mapTransitioning`** 置 false（通常在客户端确认进图后的 handler 里调用；具体可与 **`RecvOpcode`** 对齐精读）。

### 14.6 与第 12 节的衔接

**`changeMapInternal`** 在 **`sendPacket` + `removePlayer` + `addPlayer`** 这一条上与 **第 12 节 `MapleMap.removePlayer` / `addPlayer`** 直接串联：**Portal 只负责算出 `(to, portal)` 或跑脚本**；**统一的副作用（贸易取消、过渡标记、组队地图 id）集中在 internal**。

## 15. `getInetSocket`、`getServerIP` 与 `getChannelChange`（登录进频道 / 切频道）

登录服在选角通过后要让客户端**新开一条 TCP 连到频道服**；频道内**换频道**、**现金商店回频道**也同样要先拿到「频道服监听地址」再通知客户端去连。本节对应 **`Server#getInetSocket`** 与 **`PacketCreator#getServerIP` / `getChannelChange`** 的语义与包布局（基于 **`BeiDou-Server`** 上游 **`gms-server`**）。

### 15.1 两个层次：`String[]` 与下行包

| 层次 | 位置 | 作用 |
|------|------|------|
| **`getInetSocket(Client, world, channel)`** | **`org.gms.net.server.Server`** | 根据 **世界 + 频道** 取出配置的 **`"host:port"`**，再按玩家登录 IP 是否本机/局域网**改写 host**，得到 **`String[]{host, port}`**。 |
| **`getServerIP(InetAddress, port, clientId)`** | **`org.gms.util.PacketCreator`** | _opcode_ **`SendOpcode.SERVER_IP`（`0x0C`）**，告诉客户端：**下一个 TCP 连哪里**，并带上 **角色 id**（见包体）。 |
| **`getChannelChange(InetAddress, port)`** | **`PacketCreator`** | _opcode_ **`SendOpcode.CHANGE_CHANNEL`（`0x10`）**，**切频道 / CS 回来**时用；**包里没有 clientId**，角色身份仍靠会话上的 **`setCharacterOnSessionTransitionState`**（第 13 节）与随后的 **`PLAYER_LOGGEDIN`**。 |

### 15.2 频道端口与 `"host:port"` 从哪来

**`org.gms.net.server.channel.Channel`** 构造里：

- **`BASE_PORT = 7575`**  
- **`port = BASE_PORT + (channel - 1) + (world * 100)`**（频道编号从 1 起）。  
- **`this.ip = serviceProperty.getWanHost() + ":" + port`**  

即：**默认对外宣告的是配置项 `gms.service.wan-host` + 推导端口**。  
**`Server`** 里 **`channels`**（world → channel → 字符串）在 **`addChannel`** 等路径写入 **`channel.getIP()`**，所以 **`getInetSocket`** 本质读的是这份 **`wanHost:port`**，再按客户端来源改写 host。

**`ServiceProperty`**（`gms.service`）另有 **`localhost`、`lanHost`**：**本机或局域网玩家**不能去拿「公网域名」，否则客户端会连不到 —— 这就是 **`getInetSocket`** 里 **`IpAddresses.isLocalAddress` / `isLanAddress`** 分支的意义。

### 15.3 `Server#getInetSocket` 逐步语义

1. **`remoteIp = client.getRemoteAddress()`**  
2. **`hostAddress = getIP(world, channel).split(":")`** → 期望长度为 2：**`[0]` host，**`[1]`** port 字符串**。  
3. 若 **`IpAddresses.isLocalAddress(remoteIp)`**：**`hostAddress[0] = serviceProperty.getLocalhost()`**  
4. 否则若 **`IpAddresses.isLanAddress(remoteIp)`**：**`hostAddress[0] = serviceProperty.getLanHost()`**  
5. **`return hostAddress`**（外围 **`try/catch`** 失败则 **`null`**）

**`socket == null`** 时：例如 **`CharSelectedHandler`** 走 **`getAfterLoginError(10)`**；**`Client#changeChannel`** 则发 **`serverNotice`** 提示频道不可用 —— 本质是 **拿不到合法频道地址串** 或解析异常。

### 15.4 `PacketCreator#getServerIP` 包体字段（顺序）

**opcode**：**`SERVER_IP`（`0x0C`）**。

写入顺序（与 **`InetAddress#getAddress()`** 字节序一致，IPv4 为 **4 字节**）：

1. **`writeShort(0)`** — 固定占位（历史协议字段）。  
2. **`writeBytes(inetAddr.getAddress())`** — IP，**IPv4 为 4 字节**（若误传 IPv6 **16 字节**，会与旧客户端假设不一致，通常部署上应用 IPv4）。  
3. **`writeShort(port)`** — 频道 **`Channel`** 监听端口。  
4. **`writeInt(clientId)`** — 此处传入的是 **角色 id**（**`charId`**），见 **`CharSelectedHandler`**：`getServerIP(..., charId)`。  
5. **`writeBytes(new byte[]{0, 0, 0, 0, 0})`** — **5 字节填零**（协议尾部占位）。

客户端收到后应 **关闭或使用登录连接的逻辑栈**，按给定地址新建连接并发 **`PLAYER_LOGGEDIN(cid)`**；服务端则用第 13 节的 **`validateCharacteridInTransition`**（若开启）对齐 **`cid`**。

### 15.5 `PacketCreator#getChannelChange`（对照）

**opcode**：**`CHANGE_CHANNEL`（`0x10`）**。

1. **`writeByte(1)`**  
2. **`writeBytes(inetAddr.getAddress())`** — 同上，IPv4 四字节。  
3. **`writeShort(port)`**  

与 **`getServerIP`** 相比：**无 short(0)、无 clientId、无 5 字节尾**；**角色 id 不带在本包里**，因为玩家已在频道上线过，过渡关系在第 13 节 **`setSessionTransitionState`** 已写入 **`transitioningChars`**。

### 15.6 典型调用链（可与前几节对照）

| 场景 | 调用大致顺序 |
|------|----------------|
| **选角后进游戏** | **`CharSelectedHandler`**（及带 PIC / ViewAll 等同类 Handler）→ **`getInetSocket`** → **`unregisterLoginState`** → **`setCharacterOnSessionTransitionState(charId)`** → **`getServerIP(..., charId)`** |
| **频道服换频道** | **`Client#changeChannel`**：校验、`removePlayer`、存档、`setSessionTransitionState()` → **`getChannelChange`** |
| **现金商店回频道** | **`ChangeMapHandler#enterFromCashShop`**：同样在 **`getInetSocket`** 成功后 **`setSessionTransitionState`** + **`getChannelChange`** |
| **GM 跨世界等** | 如 **`WarpWorldCommand`** 也会对目标 world 调 **`getInetSocket`** |

读懂本节后再回看 **第 13 节**：**`getServerIP`/`getChannelChange` 解决「客户端 TCP 连哪」；`transitioningChars` 解决「这条连接声称要登录的 cid 是否合法」。**

## 16. `PLAYER_LOGGEDIN` 拆包与「进图完成」：`PLAYER_MAP_TRANSFER` → `setMapTransitionComplete`

第 11 节已从产品视角写过 **`PlayerLoggedinHandler`** 与 **`loadCharFromDB`**；本节补：**频道第一个业务包 `RecvOpcode.PLAYER_LOGGEDIN` 在读什么**、**与第 15 节 `getServerIP` 里 `writeInt` 的对应关系**，以及 **`Character.mapTransitioning` 何时解除**（换图门禁 **`ChangeMapHandler`** 依赖它）。

### 16.1 注册位置与 opcode

- **`PacketProcessor`**（频道）：**`registerHandler(RecvOpcode.PLAYER_LOGGEDIN, new PlayerLoggedinHandler(...))`**  
- **`RecvOpcode.PLAYER_LOGGEDIN`**：**`0x14`**（枚举注释：玩家已登录）

### 16.2 `PlayerLoggedinHandler#handlePacket` 入口：包体第一个 int

方法第一行有效读取：**`final int cid = p.readInt()`**（源码注释写明待核实是否与 **`PacketCreator#getServerIP`** 里 **`writeInt(clientId)`** 为同一概念）。

与 **第 15 节** 对齐：**选角后下发的 **`getServerIP`** 在 **`CharSelectedHandler`** 等处传入 **`charId`**；客户端连上频道后应把这个 **角色 id** 带进 **`PLAYER_LOGGEDIN`**，服务端 **`cid`** 即用于：

1. **`validateCharacteridInTransition(c, cid)`**（第 13 节）  
2. **`Character.loadCharFromDB(cid, ...)`**（若世界级 **`PlayerStorage`** 尚无该实例）  
3. **BuffStorage**：**`getBuffsFromStorage(cid)` / `getDiseasesFromStorage(cid)`**

后续 **`InPacket`** 是否还有字段取决于客户端版本；当前 Handler **未再读 `p`**，其余逻辑不依赖包尾。

### 16.3 `validateState`：只允许「尚未 logged-in」的会话

**`validateState(Client c)`** 返回 **`!c.isLoggedIn()`**。频道处理器共性：**已登录会话不应再走登录入口**。

### 16.4 并发与其它门卫（与第 11 节呼应）

按大致顺序（中间省略若干 disconnect）：

1. **`c.tryacquireClient()`**：拿不到则 **`getAfterLoginError(10)`**（与 **`attemptingLoginAccounts`** 等共同防止同会话乱序）。  
2. **`World` / `Channel` 非空**，否则 **disconnect**。  
3. **`Character player = wserv.getPlayerStorage().getCharacterById(cid)`**：若 **null**（新开 TCP，尚未在本 world 内存里），用 **`SessionCoordinator.pickLoginSessionHwid(c)`** 补齐 **HWID**，失败则 **disconnect**。  
4. **`validateCharacteridInTransition`**，失败 **disconnect**。  
5. **`player == null`** 则 **`Character.loadCharFromDB(cid, c, true)`**，仍 **null** 则 **disconnect**。  
6. **`c.setPlayer(player)`、`setAccID`**。  
7. **`tryAcquireAccount(accId)`** 内检查 **`getLoginState()`** 须为 **`LOGIN_SERVER_TRANSITION`**，否则 **`setPlayer(null)`** 等并按状态 **`getAfterLoginError(7)`** 或 **disconnect**；成功则 **`updateLoginState(LOGIN_LOGGEDIN)`**。  
8. 若角色已在 world 在线（**`!newcomer`**）：**`player.newClient(c)`** 换绑会话。  
9. **`cserv.addPlayer`、`wserv.addPlayer`**，**`getCharInfo`、`addPlayer(map)`** 及一长串同步包（好友、家族、公会、技能、任务定时器等）——与 **第 11 / 12 节**一致。  
10. **`finally { c.releaseClient(); }`**

**要点**：**`PLAYER_LOGGEDIN` 不承担「地图加载完毕」**；它负责 **把角色挂进频道与世界、塞进地图并发初始下行**。真正的 **「换图中」标志**见下。

### 16.5 `mapTransitioning` 默认值与 `changeMapInternal`

**`Character`** 字段：**`mapTransitioning`** 初始为 **`true`**（注释：客户端正在换图或正在登录进游戏地图）。

- **`changeMapInternal`** 开头：**`mapTransitioning.set(true)`**（每次服务端发起 warp 再置一次）。  
- **`ChangeMapHandler`** 开头：若 **`chr.isChangingMaps()`**（即 **`mapTransitioning`**），直接 **`enableActions()`** 返回 —— **在客户端确认「过渡结束」之前，踩光圈等会被拒**。

因此：**刚进频道若客户端不发「过渡完成」包，会一直视为 changing maps**。解除标志在下面。

### 16.6 `PlayerMapTransitionHandler`：`RecvOpcode.PLAYER_MAP_TRANSFER`

- **`PacketProcessor`**：**`registerHandler(RecvOpcode.PLAYER_MAP_TRANSFER, new PlayerMapTransitionHandler())`**  
- **`RecvOpcode.PLAYER_MAP_TRANSFER`**：**`0xCF`**（枚举注释：玩家地图转换）

**`handlePacket(InPacket p, Client c)`**：

1. **`chr.setMapTransitionComplete()`** → **`mapTransitioning.set(false)`**，允许 **`ChangeMapHandler`** 等继续处理。  
2. **Homing Beacon**：若有 **`BuffStat.HOMING_BEACON`**，取消并发 **`giveBuff`** 同步客户端。  
3. **非隐身玩家**：遍历当前地图怪物 —— 对 **原控制者为本角色** 的怪 **`stopControllingMonster`、`sendDestroyData`、`aggroRemoveController`**；其它分支 **`sendDestroyData`**；再 **`sendSpawnData`、`aggroSwitchController`** —— **换图后刷新怪物控制器与可见状态**，避免冰冻等状态错乱（注释中致谢多条 bug 报告）。

即：**该包 = 客户端告知「地图实例切换的这一帧已就绪」**；服务端先解锁 **`mapTransitioning`**，再修怪物视图。

### 16.7 与登录 / 切频道串起来的最小时间线

| 阶段 | 服务端侧标志 / 包 |
|------|-------------------|
| 登录服选角 | **`setCharacterOnSessionTransitionState`**，`LOGIN_SERVER_TRANSITION`，**`getServerIP`** |
| 频道 TCP 建立 | **`PLAYER_LOGGEDIN`** → **`LOGIN_LOGGEDIN`**，**`map.addPlayer`** |
| 客户端加载地图完毕 | **`PLAYER_MAP_TRANSFER`（0xCF）** → **`setMapTransitionComplete`**，`mapTransitioning == false` |
| 之后踩光圈 | **`CHANGE_MAP`**，不再被 **`isChangingMaps`** 挡住 |

若 **`PLAYER_MAP_TRANSFER`** 丢失或客户端不发：**`mapTransitioning`** 长期为 **true**，表现为 **换图相关包被拒绝**（与 **`ChangeMapHandler`** 日志场景一致）。

## 17. `RecvOpcode.CHANGE_MAP` 收包结构与 `ChangeMapHandler` 分支（交叉第 14 节）

**`PacketProcessor`**：**`registerHandler(RecvOpcode.CHANGE_MAP, new ChangeMapHandler())`**。  
**`RecvOpcode.CHANGE_MAP`**：**`0x26`**（枚举注释：更改地图）。

Handler 注释：**玩家通过光圈切换地图触发**。下面按 **`ChangeMapHandler#handlePacket`** 源码顺序拆开：**何时几乎不读包**、**普通路径读了哪些字段**、**死亡 / GM / 教程与白名单**、**最后为何仍可能走 `portal.enterPortal`**。

### 17.1 前置门卫（与第 16 节衔接）

1. **`chr.isChangingMaps()`**（**`mapTransitioning`**）或 **`chr.isBanned()`**：打日志（若在换图中）、**`enableActions()`**、**return**。  
2. **`chr.getTrade() != null`**：**`Trade.cancelTrade(..., UNSUCCESSFUL_ANOTHER_MAP)`**。  
3. **`p.available() == 0`**：**视为「空包」——从现金商店回频道**，转 **`enterFromCashShop(c)`**（第 13 / 15 节：**`setSessionTransitionState` + `getChannelChange`**），本函数 **return**。  
4. **`chr.getCashShop().isOpened()`**：**`disconnect(false, false)`** —— 频道侧认为不应在 CS 开着时走常规 **`CHANGE_MAP`**。

### 17.2 非空包：字段读取顺序（与实现一致）

以下均在 **`try { ... }`** 内按顺序 **`read`**（末尾 **`catch` 只 `printStackTrace`**，异常时可能半截读包）：

| 顺序 | 读取 | 含义（源码注释或推断） |
|------|------|------------------------|
| 1 | **`byte`** | **死亡 vs 普通**：注释 **`1 = from dying`，`0 = regular portals`**（服务端当前**未分支使用该字节**，仅消耗）。 |
| 2 | **`int`** | **`targetMapId`**；**`-1`** 表示「无客户端侧直达目标」一类语义（见 17.4）。 |
| 3 | **`String`** | **`portalName`** → **`chr.getMap().getPortal(portalName)`**。 |
| 4 | **`byte`** | **未注释**，消费占位（常见于客户端对齐版本）。 |
| 5 | **`byte`**（作布尔） | **`wheel`**：**`> 0`** 表示携带「复活轮」意图（见死亡分支）。 |
| 6 | **`byte`** | **GM 追逐模式前置**：若 **`== 1`** 且 **`chr.isGM()`** 且 **`p.available() == 8`**（两个 int），则读 **`Point`**。 |

**GM chasing**：**`chr.setChasing(true)`**，**`chr.setPosition(new Point(x, y))`** —— 服务端信任 GM 包内坐标（需在 **`RecvOpcode.MOVE_PLAYER`** 精读里再对照是否一致）。

### 17.3 `targetMapId != -1`：死亡复活 / GM 直飞 / 教程白名单

该块与 **`portal.enterPortal`** **并列执行**：前面可能 **`changeMap`** 过了，后面仍会跑 **portal 状态 / 距离 / `enterPortal`**（见 17.5）。

**角色已死亡（`!chr.isAlive()`）**

- **`wheel && haveItem(WHEEL_OF_FORTUNE)`**：扣道具、**`updateHp(50)`**、**在同图 **`changeMap(map, findClosestPlayerSpawnpoint(...))`** —— 原地复活类处理。  
- 否则：若 **`EventInstanceManager != null`**，先试 **`revivePlayer`**；否则 **`respawn(map.getReturnMapId())`**。

**角色存活且 `targetMapId != -1`**

- **GM**：**`getWarpMap(targetMapId)`** + **`changeMap(to, to.getPortal(0))`** —— **无视 portal 名**，直达目标图 portal 0。  
- **非 GM**：**`divi = chr.getMapId() / 100`**，仅在若干 **新手 / 剧情地图区间**内，若 **`targetMapId`** 命中白名单，则 **`changeMap(to, to.getPortal(0))`**（含蘑菇村、战神/骑士团引导、魔女塔入口等分支 —— 防擅自改 **`targetMapId`** 乱飞）。

若 **`targetMapId == -1`**：**本块整体跳过**，直接进入下一小节。

### 17.4 Portal 闭合与世界观：`targetMapId == -1` 的常见路径

通常客户端 **踩光圈**会配合 **`targetMapId == -1`**（或与你所用客户端版本等价语义）：服务端 **不信任**客户端声称的跨图目标，而依赖 **`GenericPortal`** 里配置的 **`targetMapId` / 脚本**（**第 14.1 节**）。

### 17.5 Portal 校验与 `enterPortal`（与第 14.2 节一致，此处强调顺序）

在 **`targetMapId`** 块之后**总是**继续：

1. **`portal != null && !portal.getPortalStatus()`**：**`blockedMessage(1)`** + **`enableActions()`**，**return**。  
2. **健身 / OLA 终点图**：重置次数类副作用。  
3. **`portal != null`**：  
   - **`distanceSq(position) > 400000`**（约 **632px**）：**`enableActions()`**，**return**。  
   - 否则 **`portal.enterPortal(c)`**（脚本或 **`changeMap(to, pto)`**，见 **14.1**）。  
4. **`portal == null`**：**`enableActions()`** —— 常见於 **`portalName`** 拼错、或 **`targetMapId`** 分支已 **`changeMap`** 后人已在其它图导致 **`getPortal` 仍用旧图引用等边界情况**（具体依赖客户端发包）。

### 17.6 与 **`CHANGE_MAP_SPECIAL`（0x64）** 的边界

**`RecvOpcode.CHANGE_MAP_SPECIAL`** 由 **`ChangeMapSpecialHandler`** 处理，**不是**本节 **`ChangeMapHandler`**。传送符、道具 **`InnerPortal`** 等在 **`RecvOpcode` / `PacketProcessor`** 另有路由；精读传送道具时请换 Handler。

### 17.7 调试提示

- **卡在光圈**：先看 **`PLAYER_MAP_TRANSFER`（16.7）** 是否发出 **`mapTransitioning`**。  
- **教程图无故_disconnect**：核对 **`CashShop.isOpened`** 分支与 **`p.available()`**。  
- **GM / 脚本不一致**：核对 **`targetMapId`** 与 **`portalName`** 谁在 **`GenericPortal.enterPortal`** 里说了算。

## 18. `RecvOpcode.MOVE_PLAYER`、`AbstractMovementPacketHandler` 与 **`setChasing`**

**`PacketProcessor`**：**`registerHandler(RecvOpcode.MOVE_PLAYER, new MovePlayerHandler())`**。  
**`RecvOpcode.MOVE_PLAYER`**：**`0x29`**（枚举注释：移动玩家）。

本节说明：**收包头固定跳过 9 字节**、**位移解析如何更新服务端坐标**、**如何把原始位移字节转给其它客户端**，以及 **GM「追逐」标志如何写进 `SET_FIELD`（换图包）并在落地后清掉**。

### 18.1 `MovePlayerHandler` 流水线

**`handlePacket`** 步骤：

1. **`p.skip(9)`** — 协议前导（未在该类逐字段命名；版本演进时常为 **tick / 随机 / 掩码** 等，**服务端不参与逻辑判断**）。  
2. 记录 **`movementDataStart = p.getPosition()`**。  
3. **`updatePosition(p, c.getPlayer(), 0)`** — 见 **18.2**，内部消费 **一整段 movement command 列表** 并 **`setPosition` / `setStance`**。  
4. **`movementDataLength = p.getPosition() - movementDataStart`** — **单纯字节计数**：后面要把这段 **原样拷进下行包**。  
5. **`p.seek(movementDataStart)`** — 读指针回到 movement 起点，供 **`rebroadcastMovementList` 按字节复制**。  
6. **`chr.getMap().movePlayer(chr, chr.getPosition())`** — **按更新后的坐标**刷新 **可见物件**（生成/销毁 spawn 数据），见 **18.3**。  
7. **`broadcastGMMessage` 或 `broadcastMessage`**：**`PacketCreator.movePlayer(chrId, p, movementDataLength)`**，把移动动画同步给附近玩家（GM 隐身走 **`broadcastGMMessage`**）。

**`EmptyMovementException`**：**空 movement** 或 **`AbstractMovementPacketHandler`** 遇到 **未实现的 command** 会抛；**`MovePlayerHandler` 静默吞掉** —— **不向客户端 `enableActions`**，异常路径等于 **本帧移动丢弃**。

### 18.2 `AbstractMovementPacketHandler#updatePosition`：服务端相信什么

与 **`parseMovement`**（另一条解析路径）不同，**`updatePosition`** 专为 **服务端状态** 服务：

- **列表形态**：先 **`byte numCommands`**，每条 **`byte command`** 再分支读字段。  
- **绝对位移（0 / 5 / 17）**：读 **`xpos, ypos`**（short）直接 **`setPosition`**；**`xwobble/ywobble/fh` 共 6 字节 `skip`** —— 注释写明 **「只有绝对位移对服务端重要」**。  
- **相对位移（1、2、6、12、13、16、18、19、20、22）**：读 **`deltaX, deltaY`**，对 **`Character`** 用 **`estimateRelativeMovePosition(before + delta)`** 更新坐标；注释说明目的：**不做精确物理**，只为 **攻击距离校验不要长期落在过时坐标**。  
- **瞬移类（3 / 4）**：**`handleTeleportMove`** —— 读落点并 **`markTeleportLikeMove(before, after)`**（**`Character`** 上 **传送前后双坐标上下文**，给后续攻击距离校验用）。  
- **突进类（7 / 8 / 9）**：**`handleDashLikeMove`** —— 同步坐标与姿态。  
- **椅子（11）**：**`handleChairMove`** —— **只 `setStance`，不改坐标**（与注释一致）。  
- **Jump down（15）**：**`handleJumpDownMove`** —— 更新坐标并 **`markRegularMove`**。  
- **case 14 / 21**：**`skip` 固定长度**，不更新坐标。  
- **default**：**`log.warn` + `EmptyMovementException`**。

**`recordRegularMove`**：在若干分支里对 **`Character`** 调 **`markRegularMove(before, after)`** —— **普通移动短时窗双坐标**，与 **`markTeleportLikeMove`** 同属 **`Character` 文末一段「攻击距离上下文」**（源码 Javadoc 已写明）。

**反作弊边界**：**此处没有速度上限、没有路径合法性（foothold）校验**；服务端假设 **客户端发来的位移序列在协议上可解析**。若要找「移速 / 飞天」类校验，需继续精读 **攻击 Handler**、**特定技能**或其它模块是否在别处限速。

### 18.3 `MapleMap#movePlayer`：坐标之后的可见性

**`player.setPosition(newPosition)`**（注意：**与 `updatePosition` 内坐标应一致**，此处再写一次是历史接口形态）。

随后：

- 遍历 **`player.getVisibleMapObjects()`**，若物件仍在 **`mapobjects`** 里则 **`updateMapObjectVisibility`**，否则 **`removeVisibleMapObject`**。  
- 再按 **`getMapObjectsInRange(player.getPosition(), getRangedDistance(), rangedMapobjectTypes)`** 给玩家 **`sendSpawnData`** / **`addVisibleMapObject`**。

即：**MOVE_PLAYER 不仅改坐标，还驱动「我看见谁 / 谁看见我」在范围内的增量更新**。

### 18.4 下行 `PacketCreator.movePlayer`：原样转发位移字节

**`movePlayer(int chrId, InPacket movementPacket, long movementDataLength)`**：

- **`SendOpcode.MOVE_PLAYER`**  
- **`writeInt(chrId)`**、**`writeInt(0)`**  
- **`rebroadcastMovementList`**：循环 **`movementDataLength`** 次 **`readByte` → `writeByte`** —— **与客户端上行 payload 一致**（注释亦提到「长度由客户端带」的隐含假设）。

他人客户端据此渲染 **同一套 movement command**；**不负责校验是否合理**。

### 18.5 `setChasing`：与 **`CHANGE_MAP`**、**`getWarpToMap`** 的闭环（回扣第 17 节）

**`ChangeMapHandler`**（第 **17.2**）：GM 满足条件时 **`chr.setChasing(true)`** 且 **`setPosition(Point)`** —— **在进入 `changeMapInternal` 之前就改了服务端坐标**。

**`PacketCreator.getWarpToMap(MapleMap to, int spawnPoint, Character chr)`**（**`SendOpcode.SET_FIELD`**）：

- 写完地图 id、portal、HP 后：**`writeBool(chr.isChasing())`**  
- 若 **`isChasing()`**：**立刻 `chr.setChasing(false)`**，再 **`writeInt(position.x)`、`writeInt(position.y)`** —— 把 **GM 追逐落地坐标**塞进换图包；**标志一次性消费**。

**`getWarpToMap(..., Point spawnPosition, ...)`**（坐标spawn重载）：写 **`writeBool(true)`** 与 **`spawnPosition`**，用于明确坐标落点，**不经过 `isChasing` 分支**。

因此：**「追逐」不是 MOVE_PLAYER 里的状态机**，而是 **CHANGE_MAP → 下一帧 SET_FIELD 携带布尔 + 坐标 → 自动清除**；与 **18.1** 的日常 **`MOVE_PLAYER`** 并列存在。

### 18.6 小结表

| 话题 | 结论 |
|------|------|
| **包头 9 字节** | **固定 skip**，不参与校验。 |
| **坐标来源** | **`updatePosition`** 按 movement command **解析或推断**；**无 foothold / 速度校验**。 |
| **他人看到的动作** | **`rebroadcastMovementList`** **字节级转发**。 |
| **`setChasing`** | **`CHANGE_MAP` 置位 + `getWarpToMap` 写入 SET_FIELD 并清除**。 |

## 19. 攻击伤害：`AbstractDealDamageHandler`、攻击框与 **`teleport` / `movement` 距离上下文**

近战 / 远程 / 魔法普攻与技能伤害在 **`PacketProcessor`** 中分别注册：

- **`RecvOpcode.CLOSE_RANGE_ATTACK`（`0x2C`）** → **`CloseRangeDamageHandler`**  
- **`RecvOpcode.RANGED_ATTACK`（`0x2D`）** → **`RangedAttackHandler`**  
- **`RecvOpcode.MAGIC_ATTACK`（`0x2E`）** → **`MagicDamageHandler`**  

三者均继承 **`AbstractDealDamageHandler`**：`handlePacket` 里 **`parseDamage(...)`** 拆包并 **`applyAttack(...)`** 结算伤害与 **`AutobanFactory`** 记分。本节聚焦：**移动/传送上下文如何喂给距离判定**，以及 **`DISTANCE_HACK` 与攻击框的关系**。

### 19.1 `Character` 上的两段「位移补偿」上下文（字段级）

在 **`markTeleportLikeMove` / `markRegularMove`**（第 **18.2** 节 **`AbstractMovementPacketHandler`**）写入前，需满足 **位移幅度阈值**：

| 上下文 | 过期时间 | 最多几次攻击消费 | 最小位移平方 |
|--------|-----------|------------------|----------------|
| **Teleport** | **1200 ms** | **2** | **≥ 1600**（约 **40px**） |
| **Movement** | **350 ms** | **1** | **≥ 400**（约 **20px**） |

生效条件（**`isTeleportDistanceContextActiveLocked` / `isMovementDistanceContextActiveLocked`**）：**context mapId == 当前地图**、**未过期**、**剩余次数 > 0**、**before/after 点非 null**。  
**`clearTeleportDistanceContext()`**（例如 **`changeMapInternal`**）：清空 **传送 + 移动** 两套上下文并把 **`lastTeleportLikeMoveTime`** 置 0。

**读接口**：**`getTeleportBeforePositionForDistanceCheck()`** / **`getMovementBeforePositionForDistanceCheck()`** —— 若已失效会在 getter 内 **`clear*Locked`** 并返回 **null**。

**消费接口**：**`consumeTeleportDistanceCheckContext()`** / **`consumeMovementDistanceCheckContext()`** —— **`applyAttack`** 若开头 **`getTeleportBeforePositionForDistanceCheck` / `getMovementBeforePositionForDistanceCheck`** 曾得到 **非 null**（在同一方法开头缓存到局部变量），则在 **`applyAttack` 末尾各 `consume` 一次**，递减次数。**注意**：这与「本轮是否真的打到怪」无关——只要进了 **`applyAttack` 且 getter 当时有效，就会消费一次**（极端路径下可能未造成伤害也已扣次数）。

### 19.2 `parseDamage` 与攻击间隔（顺带）

**`parseDamage(InPacket p, Character chr, boolean ranged, boolean magic)`** 开头：**`detectionAttackInterval(chr, ret)`** —— 对非 **SKIP_SKILL_ID_SET**（如 **暴风箭雨 / 金属风暴**）技能，若两次攻击间隔 **< 250 ms** 记 **`AutobanFactory.ATTACK_INTERVAL` point**，**< 350 ms** 走 **`alert`**；并 **`setLastAttackTime`**。

其后按 **`ranged` / `magic`** 分支跳过不同字节，计算 **`calcDmgMax`**，再按 **`numAttacked`** 读每个怪的 **`oid`、伤害列表**；伤害超过 **`maxWithCrit * 1.5`** 告警、**`* 5`** 记 **`DAMAGE_HACK` point**（与距离外挂并列的另一条防线）。

### 19.3 `applyAttack`：距离判定的入口

对每个 **`attack.allDamage`** 中的 **`Monster`**：

1. 计算 **`distanceToDetect`**：**基础 `200000`（距离平方）**，**远程 +400000**、**魔法 +200000**，**Aran 职业链 +200000**，再按 **技能 id** 微调（如 **全屏类、龙之怒、链闪电** 等有加码）。  
2. **`skipDistanceHack`**：**全屏豁免**（Genesis / Blizzard / Meteor 等）或 **无空间攻击框语义**的技能（如 **Energy Charge** 且 **StatEffect 无 bbox**）。  
3. **`teleportBeforePosForDistanceCheck`**、**`movementBeforePosForDistanceCheck`** 在 **循环外**各取一次（同一 **`applyAttack` 调用内复用）。  
4. 若 **`isWithinAttackBox(..., teleportAlt, movementAlt)`** 为 **true**：**视为技能框已盖住怪物**，**不进入**下面的 **`DISTANCE_HACK` 距离比较**。  
5. 否则若 **`hasReliableDistanceGeometry`**（技能有框 **或** 怪物有 **bbox**）：**`chooseBestDistanceCheckSample`** 在 **`player.getPosition()`**、**传送前坐标**、**移动前坐标** 中选 **距怪最近**的一点，若 **`distanceSq > distanceToDetect`** 且为本轮最差，则记录 **`distanceHackWorst*`**。  

**链式闪电**：仅 **第一个目标**做玩家距离检测，后续目标 **`checkDistance = false`**（注释：目标来自跳跃链）。

循环结束后：

- 若开头缓存的 **`teleportBeforePosForDistanceCheck != null`**：**`consumeTeleportDistanceCheckContext()`**  
- 若 **`movementBeforePosForDistanceCheck != null`**：**`consumeMovementDistanceCheckContext()`**  
- 若有 **`distanceHackWorstMonster`**：**`AutobanFactory.DISTANCE_HACK.addPoint`** + **`log.warn`**（带 **`buildBboxInfo`**）。

### 19.4 `isWithinAttackBox`：三种「玩家站位」试朝向

**`intersectsAnyAttackBox`** 会用 **`StatEffect.calculateBoundingBox(playerPos, facingLeft)`** 与 **`getMonsterBounds(monster)`** 求交；**`playerPositions`** 依次为：**当前坐标**、**`teleportBefore`**、**`movementBefore`**（null 跳过，与当前点重复跳过）。

**朝向**：先试 **`attack.direction`**（**`direction == 0` → 朝左**），再试 **`attack.stance`** 奇偶，再试 **`player.isFacingLeft()`** —— 减少 **包内朝向与内存 stance 短暂不一致** 的误判。

### 19.5 `chooseBestDistanceCheckSample`：为何需要「备用坐标」

传送或大步移动后，**服务端 `player.getPosition()`** 可能已是 **落点**，而客户端本次攻击仍按 **起手位置** 判定命中；**在上下文窗口内**允许用 **更小** 的 **`distanceSq`**（相对怪）作为样本，避免 legit 被判远距离。**`usedTeleportContext` / `usedMovementContext`** 会写入 **`DISTANCE_HACK` 日志**，便于区分是否走了补偿分支。

### 19.6 与第 18 节的衔接（数据流）

```text
MOVE_PLAYER → updatePosition → markRegularMove / handleTeleportMove → markTeleportLikeMove
       ↓
CLOSE_RANGE / RANGED / MAGIC → parseDamage → applyAttack
       → getTeleportBeforePositionForDistanceCheck / getMovementBeforePositionForDistanceCheck
       → isWithinAttackBox / chooseBestDistanceCheckSample
       → consumeTeleportDistanceCheckContext / consumeMovementDistanceCheckContext
```

### 19.7 局限（读代码时的清醒认知）

- **`MOVE_PLAYER` 仍无速度校验**；距离检测是 **事后**、且依赖 **WZ 攻击框 / 怪物 bbox** 是否齐全。  
- **`distanceToDetect` 为固定平方阈值 + 技能补丁**，不是物理引擎。  
- 异常 **`parseDamage`** 路径若未进入 **`applyAttack`**，**不会消费**位移上下文（次数可能保留到超时）。

## 20. 召唤兽：`MOVE_SUMMON`、`SUMMON_ATTACK`、`DAMAGE_SUMMON` 与地图广播

与 **第 18 节** 玩家 **`MOVE_PLAYER`** 类似：召唤兽位移同样走 **`AbstractMovementPacketHandler#updatePosition`** + **`rebroadcastMovementList`** 转发载荷；但 **收包头字段不同**，且 **不承担 `Character` 的 `markRegularMove` / 传送距离上下文**（目标对象是 **`Summon`**）。攻击路径 **`SummonDamageHandler`** **不入 `AbstractDealDamageHandler.applyAttack`**，因此 **没有第 19 节的攻击框 / `DISTANCE_HACK` / `teleport`·`movement` 上下文**，仅有 **伤害上限裁剪**。

### 20.1 Opcode 与注册（易混：`SEND` / `RECV` 各自编号）

**收包 `RecvOpcode`**（频道 **`PacketProcessor`**）：

| RecvOpcode | 值 | Handler |
|------------|-----|---------|
| **`MOVE_SUMMON`** | **`0xAF`** | **`MoveSummonHandler`** |
| **`SUMMON_ATTACK`** | **`0xB0`** | **`SummonDamageHandler`** |
| **`DAMAGE_SUMMON`** | **`0xB1`** | **`DamageSummonHandler`** |

下行 **`PacketCreator.moveSummon`** 使用 **`SendOpcode.MOVE_SUMMON`**（枚举值 **`0xB1`**）—— **与 `RecvOpcode.DAMAGE_SUMMON` 数值相同，命名空间不同**，阅读日志时不要混在一起。

### 20.2 地图侧：`spawnSummon` 与「可见性」

**`MapleMap#spawnSummon`**：**`spawnAndAddRangedMapObject(summon, c -> spawnSummon(...), null)`** —— 与怪物等一致走 **范围广播**，不是全图无条件 **`broadcastMessage`**（见 **第 12 节**「ranged map objects」）。地图还可 **`allowSummonState`** 关闭召唤（部分传送/FieldLimit 逻辑）。

### 20.3 `MoveSummonHandler`：收包 → 更新 `Summon` → 范围广播

1. **`oid = readInt`**（召唤物 **`objectId`**）。  
2. **`startPos = (readShort, readShort)`** —— 包内起点；**下行 `PacketCreator.moveSummon` 会原样 `writePos(startPos)`**。  
3. 在 **`player.getSummonsValues()`** 里按 **`oid`** 找 **`Summon`**；找不到则 **静默忽略**。  
4. **`updatePosition(p, summon, 0)`** —— 与 **`MOVE_PLAYER`** 共用 **同一套 movement command 解析**（绝对位移更新坐标，相对位移对 **`AnimatedMapObject`** **`afterPos` 可能为 null**，见 **18.2** 分支注释）。  
5. 记录 **`movementDataLength`**，`seek` 回起点后：  
   **`broadcastMessage(player, PacketCreator.moveSummon(playerId, oid, startPos, p, len), summon.getPosition())`** —— **`summon.getPosition()` 已在 `updatePosition` 后刷新**；**`broadcastMessage` 的 ranged 语义**决定哪些客户端收到 **`MOVE_SUMMON`**（与 **第 12 节**范围物件一致）。

**与 `MovePlayerHandler` 的差异**：**无 `map.movePlayer`** —— **不触发玩家可见物件表的增量刷新**；召唤兽自身坐标更新后，依赖 **`broadcastMessage` 的 ranged 语义**让周边客户端收到 **`MOVE_SUMMON`**。

**`EmptyMovementException`**：**静默忽略**，本帧不广播。

### 20.4 `SummonDamageHandler`（`SUMMON_ATTACK`）：拆包、广播、伤害上限

1. **`oid`** → 在 **`getSummonsValues()`** 找 **`Summon`**；**无 summon / 玩家死亡** → **return**。  
2. **`summonSkill` / `summonEffect`** 来自 **`summon.getSkill()` / `getSkillLevel()`**。  
3. **`p.skip(4)`**。  
4. **`direction = readByte`**，**`numAttacked = readByte`**，**`p.skip(8)`**（源码注释：略过 mob/summon 坐标类字段）。  
5. 循环 **`numAttacked`**：**`monsterOid`**，**`skip(18)`**，**`damage`** → **`SummonAttackEntry` 列表**。  
6. **`broadcastMessage(..., PacketCreator.summonAttack(...), summon.getPosition())`** —— 先同步动画/数字给他人。  
7. **`isOwnershipRestricted`** → **return**（不占伤害）。  
8. **`magic = (summonEffect.getWatk() == 0)`**，**`maxDmg = calcMaxDamage(...)`** —— 物理侧用 **手持武器类型** 参与 **`calculateMaxBaseDamage`**（无武器则按 **单手剑**），再乘 **`summonEffect.getWatk()`** 与分段系数；魔法侧用 **`calculateMaxBaseMagicDamage` × `matk` 比例**。  
9. 对每个目标：**`damage > maxDmg`** → **`DAMAGE_HACK.alert`** + **log**，**`damage = maxDmg`**；可选 **怪物 stati**；**`map.damageMonster(player, target, damage)`**。  

**缺口**：**未校验** summon 与怪的 **距离**、**攻击次数**是否与 **`StatEffect`** 一致（与 **第 19 节** 完整近战/魔法链对比鲜明）。

**特例**：**`Outlaw.GAVIOTA`** 攻击后 **`player.cancelEffect(summonEffect, ...)`** 取消召唤。

### 20.5 `DamageSummonHandler`（`DAMAGE_SUMMON`）：召唤兽挨打

1. **`oid`**，**`skip(1)`**，**`damage`**，**`monsterIdFrom`**。  
2. **`player.getMap().getMapObject(oid)`** 若为 **`Summon`**：**`addHP(-damage)`**；**HP ≤ 0** 且与傀儡相关则 **`cancelEffectFromBuffStat(PUPPET)`**。  
3. **`broadcastMessage(..., PacketCreator.damageSummon(...), summon.getPosition())`**。

**信任模型**：**伤害值基本采信客户端**（无 **`maxDmg`** 裁剪），与 **`SummonDamageHandler`** 不同。

### 20.6 与前几节对照（一行）

| 维度 | 玩家普攻/技能（第 19 节） | 召唤兽攻击（本节） |
|------|---------------------------|---------------------|
| 距离 / 攻击框 | **`isWithinAttackBox`、`DISTANCE_HACK`** | **无** |
| `teleport`/`movement` 上下文 | **有** | **无**（不走 **`applyAttack`**） |
| 伤害 | **`calcDmgMax`、暴击与多段线** | **`calcMaxDamage` 上单段封顶** |

## 21. `ITEM_PICKUP`、`MapItem` 归属与掉落可见性（衔接第 12 节）

**`RecvOpcode.ITEM_PICKUP`**：**`0xCA`** → **`ItemPickupHandler`**。

本节串起：**拾取包怎么读**、**服务端距离门禁**、**`MapItem` 归属 / 组队 / FFA**、**`pickupItem` 与 `pickItemDrop`**，以及 **`spawnItemDrop` 为何「范围 spawn + 全图广播」双下发**。

### 21.1 `ItemPickupHandler`

**`handlePacket`** 顺序：

1. **`readInt()`** — 注释 **Timestamp**  
2. **`readByte()`**  
3. **`readPos()`** — 注释 **`cpos`**（客户端声称坐标；**当前未用于校验**）  
4. **`oid = readInt()`** — 地图物件 id  

**`chr.getMap().getMapObject(oid)`**：**null** 则 **return**（物件已消失或 oid 伪造）。

**距离**：若 **`|dx| > 800`** 或 **`|dy| > 600`**（相对 **`Character.getPosition()`** 与 **`ob.getPosition()`**）：**`AutobanFactory.DISTANCE_HACK.addPoint`** + **`log.warn`**，**return**（注释掉的 **`alert`** 保留在源码）。  

通过则 **`chr.pickupItem(ob)`** —— 参数类型是 **`MapObject`**，实际分支在 **`instanceof MapItem`**。

### 21.2 `MapItem`：掉落实体与客户端归属字段

**`MapObjectType.ITEM`**。构造常见字段：

- **`character_ownerid`** / **`party_ownerid`** / **`partyDrop`**  
- **`playerDrop`**、**`type`（`droptype`）**、**`meso`** 或 **`item`**  
- **`dropTime`**：**`setDropTime`** 在 **`spawnItemDrop` / `spawnMesoDrop` / `spawnDrop`** 等处设为服务端当前时间  

**归属判定**：

- **`hasExpiredOwnershipTime()`**：落地超过 **15 秒** → 视同过期，他人可捡。  
- **`isFFADrop()`**：**`type == 2 || type == 3`** **或** 已过归属时间 —— **全队 / 个人锁失效**。  
- **`canBePickedBy(Character chr)`**：**`character_ownerid <= 0`** 或 **`isFFADrop()`** → **true**；否则仅限 **本人 / 同队**（含运行时把 **`party_ownerid`** 补上的分支）。

**`getClientsideOwnerId()`**：无队时报 **角色 id**；有队时报 **`party_ownerid`** —— 与客户端显示归属一致（源码注释致谢 bug 修复）。

### 21.3 `MapItem.sendSpawnData`：任务掉落与其它掉落的差异

**`sendSpawnData(Client)`**：仅当 **`chr.needQuestItem(questid, getItemId())`** 时才 **`dropItemFromMapObject`**。

含义：**「后进图、跑 `sendObjectPlacement`」路径下**，未接到 **`spawnItemDrop` 全图广播**的玩家，**只有需要做该任务的人才收到这件地上的可见包**；普通掉落主要靠 **`spawnAndAddRangedMapObject` 的 packetbakery** 与下面的 **`broadcastItemDropMessage`**（见 **21.4**）。

### 21.4 `spawnItemDrop`：范围写入 `mapobjects` + 全图二次广播

**`spawnItemDrop(...)`**（非 **`DROP_LIMIT`** 地图）：

1. **`MapItem`** **`setDropTime`**。  
2. **`spawnAndAddRangedMapObject(mdrop, packetbakery)`**：  
   - **_oid** 分配、`**mapobjects**` 登记；  
   - **仅当 **`chr.getPosition().distanceSq(dropPos) ≤ getRangedDistance()`** 的玩家 **`addVisibleMapObject`**；  
   - **`packetbakery`** 对这些玩家发 **`dropItemFromMapObject(..., mod 1)`**。  
3. **`broadcastItemDropMessage(mdrop, ..., mod 0)`**：**无距离过滤**（**`rangeSq == ∞`**），对 **本图全部 **`characters`** 再发 **`dropItemFromMapObject`**。  

因此：**服务端「可见物件表」仍按范围维护**（与 **第 12 节**、`MovePlayer` 增量一致）；**全图玩家仍会收到一次 mod 0 掉落包**（远端客户端表现与 **`visibleMapObjects`** 是否同步取决于客户端如何利用两包）。

**`spawnMesoDrop`**：只做 **`spawnAndAddRangedMapObject` + `instantiateItemDrop`**，**没有**第三步全图广播 —— **远距离玩家仅靠是否在掉落瞬间处于 **`getRangedDistance`** 内**收到 **`packetbakery`**。

**`instantiateItemDrop`**：**地图 **`item_limit_on_map`** 超限时踢旧掉落（**`makeDisappearItemFromMap`**）；**`droppedItems`** 登记过期时间 **`item_expire_time`**（**`everlast`** 图可为 **`Long.MAX_VALUE`**）。

### 21.5 `Character#pickupItem`：时间锁、组队分钱、`pickItemDrop`

入口判断：

- **`now - dropTime < 400`** **或** **`!canBePickedBy(this)`**：**`enableActions`** **return**（防落地瞬间抢捡、防归属外捡）。  

**`lockItem`** 后 **CAS 式**：若 **`isPickedUp`**：**`showItemUnavailable`**。  

成功路径组装 **`PacketCreator.removeItemFromMap`**（宠物分支 **`mod 5`**），再走 **金币平分组队成员 / `gainMeso` / `addFromDrop` / NX / 任务道具等**，最后 **`getMap().pickItemDrop(pickupPacket, mapitem)`**。

### 21.6 `MapleMap#pickItemDrop`

**`broadcastMessage(pickupPacket, mdrop.getPosition())`** —— **按掉落位置的 ranged 规则**广播移除（他人客户端删掉地面物件）。  

然后：**`droppedItemCount--`、`removeMapObject`、`setPickedUp(true)`、`unregisterItemDrop`**。

### 21.7 组队变动与静默刷新（顺带）

**`updatePlayerItemDropsToParty`** / **`updatePartyItemDropsToNewcomer`**：**`silentRemoveItemFromMap`** + **`updateMapItemObject`**，把 **`party_ownerid`** 变更同步给需要的客户端（与 **第 12 节**组队摘要一致）。

## 22. 怪物掉落调度：`dropFromMonster`、`MobLootEntry` 与 `use_spawn_loot_on_animation`

在 **`killMonster` 成功击杀且 `withDrops`** 时：**`dropOwner = monster.killBy(chr)`**（可能 **null** 则回退攻击者），再 **`dropFromMonster(dropOwner, monster, false)`**（**`useBaseRate == false`** 表示使用角色 **`getDropRate` / `getBossDropRate`** 等）。本节说明：**掉落表来源**、**`MobLootEntry#run` 顺序**、**延时掉落定时器**、以及与 **`dropItemsFromMonster`** 的差异。

### 22.1 `dropFromMonster`：droptype、倍率与表分拣

**`dropFromMonster(Character chr, Monster mob, boolean useBaseRate)`**：

1. **`dropsDisabled()` 或 `!dropsOn`** → **return**。  
2. **`droptype`**：**爆炸_reward → 3**；否则 **FFA → 2**；否则 **有组队 → 1**；否则 **0**（对应 **`MapItem` 归属**，见 **第 21 节**）。  
3. **`chRate`**：非 Boss **`chr.getDropRate()`**，Boss **`getBossDropRate()`**；叠加 **`SHOWDOWN`**、家族 **`chr.getFamilyDrop()`**；**`useBaseRate`**（友好怪 **`dropFromFriendlyMonster`**）则 **`chRate = 1`**。  
4. **条目**：**`use_spawn_relevant_loot`** → **`mob.retrieveRelevantDrops()`**，否则 **`MonsterInformationProvider.retrieveEffectiveDrop(mobId)`**；**全局表 **`getRelevantGlobalDrops(mapId)`****。  
5. **`sortDropEntries`**：**普通** / **`visibleQuestEntry`**（**`chr.needQuestItem`**）/ **`otherQuestEntry`**。  
6. **`lootEntry` 空** → **return**。  
7. **`registerMobItemDrops(...)`** → **`MobLootEntry`** 立即 **`run()`** 或进入 **`mobLootEntries`**（**22.3**）。

### 22.2 `MobLootEntry#run`：顺序与散落偏移

**`run()`**：

1. **`dropItemsFromMonsterOnMap(dropEntry)`** —— 普通  
2. **`dropGlobalItemsFromMonsterOnMap(globalEntry)`** —— 全局（随机 **`de.chance`**，**不乘 **`chRate`**）  
3. **`dropItemsFromMonsterOnMap(visibleQuestEntry)`**  
4. **`dropItemsFromMonsterOnMap(otherQuestEntry)`** —— **最后一趟返回值丢弃**

**`dropItemsFromMonsterOnMap`**：**shuffle**；命中则改写 **`pos.x`** 相对 **`mobpos` 一字排开（**25px**，**`droptype==3`** 用 **40px**）**；金币 **`spawnMesoDrop`**，装备/道具 **`spawnDrop`**（含 **`questid`**）。**`dropChance`**：**`de.chance * chRate * chr.getCardRate(de.itemId)`** 与 **999999** 比较。

### 22.3 `use_spawn_loot_on_animation`

**`registerMobItemDrops`**：**若配置 true**，**`mobLootEntries.put(mle, currentTime + (long)(0.42 * mob.getAnimationTime("die1")))`**；否则 **`mle.run()`**。

**`startItemMonitor`**：首张图起监控时，若配置开启：**`mobLootEntries.clear()`**，**`mobSpawnLootTask`** **每 200ms `spawnMobItemDrops`**。到期条目 **`remove` 后 `run()`**。  
**`stopItemMonitor`**：**cancel `mobSpawnLootTask`**。

### 22.4 `dropItemsFromMonster`（公开）

**不经 **`MobLootEntry`****：**`droptype`** 仅 **组队 → 1 / 否则 0**；**`chRate = 1000000`**（极高）；直接 **`dropItemsFromMonsterOnMap`** —— 供脚本/强制掉落场景。

### 22.5 与第 21 节

落地仍走 **`spawnMesoDrop` / `spawnDrop` / `spawnItemDrop`** 与拾取 **`ITEM_PICKUP`**。

## 23. `killMonster`、`removeKilledMonsterObject` 与 **`Monster.killBy` → 掉落归属

本节衔接 **第 22 节**：**谁作为 `Character owner` 写进 `MapItem`**，来自 **`dropFromMonster(dropOwner, …)`** 的 **`dropOwner`**；而 **`dropOwner`** 又由 **`Monster.killBy(killer)`** 的返回值（再 **`null` 则回退 `killer`**）决定。

### 23.1 入口：`damageMonster` → **`killMonster(monster, chr, true)`**

**`MapleMap#damageMonster(chr, monster, damage)`**：怪仍 **`isAlive()`** 时 **`monster.damage(chr, damage, false)`**；若 **`damage` 返回 lastHit（致死）** 或触发自爆阈值，则 **`killMonster(monster, chr, true[, animation])`**。

因此 **`killMonster` 的 `chr`** 通常是 **最后一击（或来自地图逻辑的击杀发起者）**，不一定是 **`killBy` 最终选用的掉落主人**。

### 23.2 `killMonster(monster, chr, withDrops, animation)` 分支

**`monster == null`** → **return**。

**`chr == null`**（环境杀、脚本清怪等）：

- **`removeKilledMonsterObject(monster)`** 成功则：**`dispatchMonsterKilled(false)`**（**无 killer** 语义）、**`broadcast killMonster`**、**`aggroSwitchController(null)`**。  
- **不掉落**（无 **`dropFromMonster`**）。

**`chr != null`**：

1. **`removeKilledMonsterObject`**：**`monster.getHp() < 0`** → **false**（防止同一死亡二次移除）；否则 **`spawnedMonstersOnMap--`、`removeMapObject`、`disposeMapObject`**（**`hp.set(-1)`**）、Boss 血条清理。  
2. **`try`** 内：等级差 **`>= 30`** 非 GM **Autoban alert**；CPQ **`gainCP`**；全图 Buff 物品；扎昆手臂等特殊逻辑。  
3. **`Character dropOwner = monster.killBy(chr)`**；若 **`withDrops && !dropsDisabled`**：**`dropOwner == null` 则 `dropOwner = chr`**，再 **`dropFromMonster(dropOwner, monster, false)`**。  
4. **`finally`**：**`dispatchMonsterKilled(true)`** + **`broadcast killMonster`**。

**顺序要点**：**掉落先结算**，**死亡广播在 `finally`**（即使 **`try` 抛异常**也会 **`broadcast`**）。

### 23.3 `Monster#killBy(Character killer)`：经验与 **`looter`**

1. **`distributeExperience(killer != null ? killer.getId() : 0)`** —— 经验分配逻辑独立于下文 **`looter`**（含组队分摊等，不在此展开）。  
2. **`aggroRemoveController()`**，记下 **`lastController`**：复活列表 **`revives` 非空**时，会在 **`die1` 时长后 **`spawnMonster`**，并把 **`aggro`** 交回 **`lastController`**（道场音效、时间怪公告、扎昆臂连带 **`killMonster`** 等分支见源码定时任务）。  
3. **掉落归属**：  
   **`Character looter = map.getCharacterById(getHighestDamagerId());`**  
   **`return looter != null ? looter : killer;`**

即：**优先本图 **`takenDamage`** 累计伤害最高的角色**；若解析不到在线角色（最高伤玩家已离场、`getHighestDamagerId` 为 **0** 等），**回退最后一击 `killer`**。

### 23.4 `takenDamage` 与 **`getHighestDamagerId`**

**`applyDamage`**（真实伤害，非 **`fake`**）：按攻击者 **`from.getId()`** 累加 **`trueDamage`** 到 **`takenDamage`**。

**`getHighestDamagerId`**：遍历 **`takenDamage`**，取 **累计伤害最大** 的 **character id**（并列时后遍历到的 id 会覆盖，属实现细节）。

**`retrieveRelevantDrops`**（第 **22** 节 **`use_spawn_relevant_loot`**）：用 **`takenDamage.keySet()`** 在线玩家去 **`LootManager.retrieveRelevantDrops`** —— **掉落表也可能随「谁打过这只怪」收缩**，与 **`dropOwner`** 不必是同一人，但同源 **`takenDamage`**。

### 23.5 `processMonsterKilled` 与 **`takenDamage` 清理**

**`dispatchMonsterKilled(true)`** → **`processMonsterKilled(true)`**：**不**走 **`dispatchRaiseQuestMobCount`**（有经验击杀路径另行计数）；**`aggroClearDamages()`** 清空 **`takenDamage`**，listener / stati 清理。

因此：**`killBy`** 必须在 **`dispatchMonsterKilled`** 之前读完 **`takenDamage`** —— 当前 **`killMonster`** 顺序满足这一点。

### 23.6 与 **`killMonsterWithDrops(int mobId)`**（顺带）

取 **`mapChars.get(mob.getHighestDamagerId())`** 作为 **`chr`**，若无则 **默认第一个地图玩家**，再 **`killMonster(mob, chr, true)`** —— **最后一击参数被设为「最高伤」**，与 **`killBy` 内层逻辑一致倾向**，但若 **`killBy` 仍回退 `killer`**，两者在边界情况下仍可能不完全相同。

## 24. `distributeExperience`：按伤害比例、`partyParticipation` 与白字判定

**`Monster#killBy`** 第一件事 **`distributeExperience(killerId)`**（第 **23** 节）。本节说明：**谁吃到经验**、**组队如何拆**、**「吸经验 / 等级区间」**与 **`giveExpToCharacter`** 里的 Buff 叠乘。

### 24.1 前置：`isAlive()` 门禁与 **`maxHpPlusHeal`**

**`distributeExperience`** 开头：**若 `isAlive()` 则 return**。

正常 **`killMonster`** 路径里 **`removeKilledMonsterObject`** 会先 **`disposeMapObject()` → `hp.set(-1)`**，因此 **`killBy` 内调用时怪已判死**，经验分支会执行。

**经验总量归一化分母**：**`totalDamage = maxHpPlusHeal.get()`** —— 出生时设为 **初始 HP**，**怪物回血**时 **`heal`** 会 **`maxHpPlusHeal.addAndGet(hpHealed)`**。经验按 **「对该怪造成的实际扣血 + 统计到的治疗增量」** 缩放，而不是单纯的 **`stats.getHp()`**。

**`expPerDmg = mobExp / totalDamage`**，其中 **`mobExp = getExp()`**。

### 24.2 参与者分组：**`partyExpDist` / `soloExpDist`**

遍历 **`takenDamage`**（仍在 **`killBy` 早期**，尚未 **`aggroClearDamages`**）：

- **`chr = mapPlayers.get(cid)`** 在线在同一 **`map.getMapAllPlayers()`** 索引里 → 按 **是否有 **`Party`** **塞进 **`partyParticipation.put(chr, damage)`** 或 **`soloExpDist.put(chr, damage)`**。  
- **`chr == null`**（离线或不在本图字典）：**`totalEntries += 1`** 仍计数 —— 当作「独立参与方」影响下文 **白字阈值** 的 **`totalEntries`**。

同一 **`Party`** 的多名输出合并进 **`partyExpDist` 的一个 **`Map<Character, Long>`****。

### 24.3 个人占比与白字（黄字经验）

对 **每个 solo**：**`ratio = damage / totalDamage`** → **`personalRatio`** + **`entryExpRatio`**。  

对每个 **party 桶**：每个成员各自 **`personalRatio`**；桶的 **`entryExpRatio`** 元素为 **队内所有人 ratio 之和**（整队占怪经验池的比例）。

**`sdevRatio = calcExperienceStandDevThreshold(entryExpRatio, totalEntries)`**：对 **`entryExpRatio`** 求均值 + 标准差，用作阈值。

**`isWhiteExpGain(chr)`**：**`personalRatio.get(chr.id) >= sdevRatio`** 则为 **白字**（源码注释链到 AyumiLove 等公式说明）。

### 24.4 Solo：**`distributePlayerExperience`**

**`exp = damage * expPerDmg`**（单人 **100%** 把自己那份池吃完）。

**`distributePlayerExperience(chr, exp, partyBonusMod=0, totalPartyLevel=chr.getLevel(), highestPartyDamager=true, white, hasPartySharers=false)`**：

- **`playerExp = (exp_split_common_mod * level / totalPartyLevel) * exp`**；若 **MVP** 再加 **`exp_split_mvp_mod`**（solo 始终 MVP）。  
- **`bonusExp = partyBonusMod * playerExp`** —— solo 为 **0**。  
- **`giveExpToCharacter(chr, playerExp, bonusExp, white, false)`**。

### 24.5 Party：**`distributePartyExperience`**

**`partyDamage`** = 队内 **`takenDamage`** 之和；**`participationMvp`** = **partyDamage 最大者**。  

**吸经验等级窗**：**`IntervalBuilder`** 并 **怪物等级 ± `exp_split_level_interval`**，再对每个参与者 **± `exp_split_leech_interval`**；若 **`use_enforce_mob_level_range`**：只对 **`getPartyMembersOnSameMap()`** 里 **落在并集区间内**的成员 **`expMembers`** 分经验；**越级 / 过低**进 **`underleveled`**，事后 **`showUnderLeveledInfo`**。

**`participationExp = partyDamage * expPerDmg`** —— **整队从怪的总经验池里刨出的基础量**。  

**`membersSize > 1`** 时 **`partyBonusMod = 0.05 * membersSize`**（源码注释：组队奖励）。

对每个 **`expMembers`**：**`distributePlayerExperience(..., participationExp, partyBonusMod, sum(levels), mc==participationMvp, white, hasPartySharers)`**。

### 24.6 **`giveExpToCharacter`**：Holy Symbol、Showdown、倍率与 **`gainExp`**

仅 **`attacker.isAlive()`** 时：

- **`personalExp`**：**`getStatusExpMultiplier`**（**Holy Symbol**：**`use_full_holy_symbol`** 与否决定是否在单人时打折；**Showdown** 乘在怪身上剩余 stati）→ **`expRate * mobExpRate`**。  
- 叠加 **`EXP_INCREASE`（加法）**、**`EXP_BUFF`（×2）**、**家族 Buff**。  
- **`partyExp`** 分支同样乘 **倍率** 与 **`party_bonus_exp_rate`（配置浮点）**。  
- **`gainExp(_personalExp, _partyExp, …, white)`**，**`increaseEquipExp`**，**`raiseQuestMobCount(getId)`**。

### 24.7 `killBy` 末尾的 **`eim.monsterKilled`**

**`distributeExperience`** 块尾：若 **`EventInstanceManager != null`**，且 **`mapPlayers.get(killerId)`** 非 null，则 **`eim.monsterKilled(chr, this)`**（参数顺序：**击杀者角色 + 怪物实例**）。  
这与 **`dispatchMonsterKilled` → `eim.monsterKilled(this, hasKiller)`**（参数顺序：**怪物 + 布尔**）是 **另一套重载** —— 同一次击杀链路里 **两处都可能触发**，脚本接口签名不同需注意。

### 24.8 与 **`dropOwner`**（第 23 节）的对照

| 维度 | 经验 | 掉落归属 |
|------|------|----------|
| 核心分配 | **所有 **`takenDamage`** 在线参与者按伤害比例 / 组队规则** | **`getHighestDamagerId()` → `killer` 回退** |
| 组队边缘成员 | **`expMembers`** 可因等级区间被排除并提示 | **`MapItem.canBePickedBy`** 仍按组队 / 15 秒（第 **21** 节） |

## 25. `gainExp` → `gainExpInternal` → **`levelUp`**：诅咒、溢出与等级上限

承接 **第 24 节**：**`giveExpToCharacter`** 末尾 **`gainExp(personal, party, …)`**。本节说明：**入口参数如何折叠成一条原子增量**、**何时不涨经验**、**连升与经验溢出递归**、**`levelUp` 做什么**，以及 **`getMaxLevel` / 事件实例例外**。

### 25.1 重载与 **`gainExp(int gain, int party, …)`**

常用路径：**`gainExp(gain, party, show, inChat, white)`**。

1. **`CURSE`**：**`gain`、`party` 各 ×0.5**。  
2. **负值溢出补丁**：**`gain < 0`** → **`gain = Integer.MAX_VALUE`**（注释 **integer overflow**）；**`party`** 同理。  
3. **`equip`**：**`(gain / 10) * pendantExp`** 封顶 **`Integer.MAX_VALUE`** —— **吊坠等加成下的「装备经验条」份额**，与角色 **`exp`** 不同通道，仍打进 **`gainExpInternal`**。  
4. **`gainExpInternal(gain, equip, party, …)`** —— **`synchronized`**（注释：**MedicOP** 发现的并发需求）。

### 25.2 `gainExpInternal`：门禁、合并、`ExpTable` 循环与 **`leftover`**

**`total = max(gain + equip + party, -exp.get())`** —— **允许扣到 0，不会把当前经验扣成负数**（死亡扣经验可走 **`loseExp` → `gainExpInternal(-loss,…)`**）。

**门禁**：**`level < getMaxLevel()`** 且 **（`allowExpGain || getEventInstance() != null`）** —— **关闭吃经验时，只要在事件实例内仍可涨**（PQ 等）。

**加经验**：

- **`nextExp = exp + total`**，若超过 **`Integer.MAX_VALUE`**：本次只加到上限，**`leftover = nextExp - MAX`** 递归处理（**`gainExpInternal(leftover, …, show=false)`**，避免重复飘字）。  
- **`updateSingleStat(EXP)`**、**`totalExpGained += total`**。  
- **`show`**：**`announceExpGain` → `PacketCreator.getShowExpGain`**（个人 / 组队数字、**`white`** 控制颜色）。

**连升**：**`while (exp >= ExpTable.getExpNeededForLevel(level))`** → **`levelUp(true)`**。  

- 全局公告：**`use_announce_global_level_up`** 且非 GM。  
- **`level == getMaxLevel()`**：**`setExp(0)`** 并 **`break`**。  
- **`use_level_up_protect`**：**升一级后 `break`** —— **单次调用最多升 1 级**（防一口气跨多级）。  

**收尾**：无 **`leftover`** 时 **`lastExpGainTime`**，可选 **`use_exp_gain_log`** 写 **`ExpLogger`**，**`totalExpGained` 清零**。

### 25.3 `getMaxLevel()` 与职业上限

**`getMaxClassLevel()`**：骑士团 **120**，其它 **200**。  

**`getMaxLevel()`**：若 **`use_enforce_job_level_range`** 且非 GM 职业：**`GameConstants.getJobMaxLevel(job)`**；否则 **`getMaxClassLevel()`**。

### 25.4 `levelUp(boolean takeexp)` 主干（节选）

**AP**：新手 **`use_auto_assign_starters_ap`** 自动加点；否则 **`level_up_ap_gain`**，骑士团低级有额外 AP。

**HP/MP**：按 **`job`** 随机区间 + **强化血量/魔法** 被动技能加成；可选 **`use_randomize_hpmp_gain`** 按智力再加 MP。

**`takeexp` 为 true**：**`exp -= ExpTable.getExpNeededForLevel(level)`**（注意：**先扣当前等级所需经验，再 `level++`**），**`exp` 负则归 0**。

**`level++`**；若 **`level >= getMaxClassLevel()`**：**经验清零**、等级钳到上限；**200 级非 GM** 可 **`PlayerNPC.spawnPlayerNPC`** + 世界公告 **`LEVEL_200`**。

**客户端**：**`recalcLocalStats`**、**`changeHpMp`**、**`PacketCreator.updatePlayerStats`**；地图 **`showForeignEffect(..., 0)`**；组队 **`silentPartyUpdate`**；公会 **`levelUpMessage`**。

**周期性**：每 **20 级** 可选 **`use_add_slots_by_level` / `use_add_rates_by_level`**；**30 级 Perfect Pitch**；**10 级** 异步 **`leaveParty`** 新手队提示等。**`guildUpdate`**、**家族声望**、**`updateMobExpRate`**。

### 25.5 与 **`increaseEquipExp`**（第 24 节脚注）

**`giveExpToCharacter`** 在 **`gainExp` 之后**调用 **`increaseEquipExp(_personalExp)`** —— **同样受 **`allowExpGain`** 控制是否执行**；遍历 **`getUpgradeableEquipList`** 调用 **`gainItemExp`**。

### 25.6 边界一览

| 情况 | 行为 |
|------|------|
| **`allowExpGain == false` 且无 EIM** | **`gainExpInternal` 整体不执行** |
| **经验单次溢出 int** | **截断 + `leftover` 递归** |
| **`use_level_up_protect`** | **单次最多升 1 级** |
| **死亡扣经验** | **`loseExp` → `gainExpInternal` 负增量** |

## 26. `StatEffect.applyTo` → **`registerEffect`** → **`recalcLocalStats`**（Buff 快照与本地属性）

承接 **第 25 节**：升级后会 **`recalcLocalStats`**；平时施放 Buff / 吃药也同样走「**服务端状态写入「快照」→ 重算 `local*` → 必要时发包同步**」。本节把 **`StatEffect`**（技能或道具解析出的**数值模型**）如何落到 **`Character`** 上的 **`effects`** / **`BuffStatValueHolder`**，以及 **`reapplyLocalStats` 读哪些 Buff** 串起来。

### 26.1 `StatEffect` 是什么（不是「运行时 Buff」本身）

类 **`org.gms.server.StatEffect`** 持有从 WZ / 数据解析出来的一坨字段：**`watk` / `matk`、`duration`、`statups`（`List<Pair<BuffStat, Integer>>`）**、召唤类型、范围框 **`lt`/`rb`**、怪物状态 **`monsterStatus`** 等。它是 **「这一次技能/道具效果的规格说明书」**；同一条规格可以 **`applyTo` 多次**（自己、队友、地图对象）。

### 26.2 `applyTo` 主路径（谁在何时吃到效果）

对外常用：**`applyTo(Character chr)`** → **`applyTo(chr, chr, primary=true, …)`**。

核心 **`private applyTo(applyfrom, applyto, primary, pos, useMaxRange, affectedPlayers)`** 大致顺序：

1. **GM 隐身**：若 **`skill && sourceid == GM.HIDE / SuperGM.HIDE`**，则 **`applyto.toggleHide(false)`** 并返回（与其它 Buff 分支无关的特殊短路）。  
2. **群体治疗前置**：若 **`primary && isHeal()`**，先 **`affectedPlayers = applyBuff(applyfrom, useMaxRange)`** —— **后面算每个人 **`calcHPChange`** 时用 **`affectedPlayers` 分摊百分比治疗**。  
3. **`calcHPChange` / `calcMPChange`**：叠加固定值、百分比、`hpCon`/`mpCon`（含元素增幅、无限魔力、专注等 MP 修正）、复活时对 **`primary==false`** 的目标特殊处理等。  
4. **`applyto.applyHpMpChange(...)`** 失败则 **`enableActions`** 并 **`return false`**（MP 不足等）。  
5. **`moveTo`**：传送类道具/技能 → **`changeMap`**；同图 **`moveTo == mapId`** 时 **`return false`**。  
6. **`Shadow Claw`**：消耗背包飞镖等硬条件分支。  
7. **`overTime || isCygnusFA() || summonMovementType != null`**：**`applyBuffEffect`** —— **发包 + `registerEffect`**（见 **26.4**）。  
8. **`primary && overTime`**：**`applyBuff`** —— **组队 Buff**：在矩形（或 **`useMaxRange` 时整图矩形）里找队友，对每个 **`applyTo(applyfrom, affected, primary=false, …)`**，并广播 **`showBuffEffect`**。  
9. **`primary && isMonsterBuff()`**：**`applyMonsterBuff`** —— 对范围内怪物 **`monster.applyStatus`** 等。  
10. 坐骑疲劳、**召唤 **`spawnSummon`**、魔法门、迷雾、Time Leap、嘉年华 **`gainCP`** / 对敌 **`giveDebuff`**、**`cureDebuffs`**、对自身或他人 **`giveDebuff`** 等 **长尾分支**，都在 **`return true`** 之前逐个消化。

要点：**「瞬时 HP/MP、传送、召唤实体」**与 **「持续 Buff（`overTime`）」**在同一次 **`applyTo`** 里按顺序组合执行；**组队扩散**是 **`primary` 侧第二次 **`applyTo(..., false)`** 调用**，目标角色的 **`primary=false`** 路径不再递归 **`applyBuff`**（避免重复圈人）。

### 26.3 `applyBuffEffect`：取消叠加规则 → 发包 → **`registerEffect`**

**`applyBuffEffect(applyfrom, applyto, primary)`**：

1. **`cancelEffect(this, true, -1)`**（除非骑宠 / 抵用券 / 神秘门（且非 Hyper Body）/ Combo 等特殊类型）—— **同类持续效果先卸一层**，减少客户端与服务端状态分叉。  
2. 准备 **`localstatups`**、**`localDuration`**（**`getBuffLocalDuration()`**：受 **`use_buff_everlasting`** 影响可变成 **`Integer.MAX_VALUE`**）、骑宠形态下把 **`localsourceid`** 换成坐骑物品 ID 等。  
3. **`primary`**：**`alchemistModifyVal`** 拉长持续时间（炼金术被动）、广播 **`showBuffEffect(..., 1, 3)`**。  
4. 若 **`StatEffect.isActive(applyto)`**（与 **`CardItemupStats`** 地图区间、是否组队狩猎有关）：构造 **`PacketCreator.giveBuff` / `givePirateBuff`** 等 **发给本人**；另有 **`giveForeign*`** 发给地图他人（Dash、WK Charge、隐身外观、Combo 数字、骑宠展示等分支）。  
5. **`applyto.registerEffect(this, starttime, starttime + localDuration, false)`**。  
6. 战船 HP 等收尾。

**`silentApplyBuff`**：**不发 **`giveBuff`** 的标准路径**（或只处理召唤等特殊），但仍 **`registerEffect(..., true)`** —— **用于登入还原、脚本静默补 Buff**：客户端状态或由 **`writeBuffStats`** 等别处对齐。

### 26.4 `registerEffect`：`effects` 快照与 **`use_buff_most_significant`**

**`Character.registerEffect(effect, starttime, expirationtime, isSilent)`** 前半段按 **`StatEffect`** 类型挂 **定时治疗 / Beholder / Recovery / 椅子回血 / map chair** 等 **周期性任务**（与「面板数值」并行存在）。

后半段在 **`prtLock` → `effLock` → `chrLock`** 下：

- 把 **`effect.getStatups()`** 转成 **`appliedStatups: Map<BuffStat, BuffStatValueHolder>`**，每个 holder 带 **`effect`、`startTime`、`value`**。  
- **`use_buff_most_significant`**：**同一 **`BuffStat`** 若已有更高 **`value`**（或同值但 **`statups` 更少**），则新 Buff **可能不进 **`effects`****，但 **`addItemEffectHolder`** 仍会记录物品层叠次数；并通过 **`propagateBuffEffectUpdates`** 把 **可能被挤掉的旧来源** 再同步给客户端 —— **避免「喝了两次药只显示强者」但客户端图标错乱**。  
- 默认分支：**`effects.putAll`**（受 **`effect.isActive`** 过滤：卡片类 Buff 在不符合地图/组队条件时 **`toDeploy` 为空**，**服务端也不登记 **`effects`****，与 **`giveBuff` 是否发送**一致）。

**结束时无条件 **`updateLocalStats()`**（见 **26.6**），因此 **任意 Buff 登记都会触发一次本地属性重算 + 可能的 **`updatePlayerStats`****。

**读取快照**：**`getBuffedValue(BuffStat)`** / **`getBuffEffect(BuffStat)`** 在 **`effLock` + `chrLock`** 下读 **`effects`** 映射 —— **战斗公式、命中率、**`reapplyLocalStats`** 里加的 **`localwatk`/`localmagic`** 等都从这里拿「当前生效的一层」**。

### 26.5 `reapplyLocalStats`：基础四维 / 装备 → **Buff 修正「快照」**

**`reapplyLocalStats`**（**`effLock` → `chrLock` → `statWlock`**）：

1. **`localMaxHp = getMaxHp()`、`localMaxMp = getMaxMp()`**，四维取自 **`getDex/getInt/...`**，**`localwatk = 0`**，**`localmagic = localint_`**，再 **`recalcEquipStats()`** 叠装备（含卷轴、星星攻击力等）。  
2. **`localmagic = min(localmagic, 2000)`**。  
3. **百分比体：** **`BuffStat.HYPERBODYHP` / `HYPERBODYMP`** → **`localMaxHp/Mp += (buff%/100) * 当前 local 上限`**，再 **钳到 30000**。  
4. **代表性加法：** **`ARAN_COMBO`** 加 **`combo.getX()`**、能量满 **`ENERGY_CHARGE`** 加 **`watk`**、**`MAPLE_WARRIOR`** 按比例抬高四维显示值、**`WATK`/`MATK`** Buff 直接加到 **`localwatk`/`localmagic`**，弓箭手 **`expert`** 被动 **`getX()`**，祝福 **`10000000 * jobType + 12`**，以及远程武器 **子弹 `watk`** 等。

注意：**`SPEED`/`JUMP` Buff 在源码里被注释掉**，若客户端仍能显示移速图标，**服务端 **`localspeed`** 可能未完全跟随 Buff** —— 排查手感差异时可对照此处。

### 26.6 `recalcLocalStats` / `updateLocalStats`：锁顺序与 HP 比例同步

- **`recalcLocalStats`**：在已有锁内调用 **`reapplyLocalStats()`**，若开启 **`use_fixed_ratio_hpmp_update`**，比较 **`localMaxHp/Mp`** 与旧值，按 **当前 HP/MP 比例或 `transientHp/Mp` 浮点比例** 生成 **`Pair<Stat, Integer>`** 列表（供 **`updatePlayerStats`** 只改数值不切条）。  
- **`updateLocalStats`**：**`prtLock` → `effLock` → `statWlock`** 调 **`recalcLocalStats()`**，**`enforceMaxHpMp()`**，若有 HP/MP 更新则 **`sendPacket(updatePlayerStats(..., true, this))`**；若 **`localMaxHp`** 变了 **`updatePartyMemberHP()`**（注释：**避免 party HP 死锁**）。

**继承字段**：**`effLock`/`statWlock`** 来自 **`AbstractCharacterObject`**，与 **`Character`** 里 **`chrLock`/`prtLock`** 组合使用 —— **不要在 holding **`chrLock`** 的情况下去拿 **`AbstractCharacterObject`** 外侧另一套锁**，新增代码时应沿用 **`updateLocalStats` / `registerEffect` 已固定的顺序**。

### 26.7 与 **第 25 节** 的对照

| 场景 | 触发的重算路径 |
|------|----------------|
| **`levelUp`** | 直接 **`recalcLocalStats()`** + **`updatePlayerStats`**（升级分支内） |
| **施放持续 Buff / 吃药** | **`applyBuffEffect` → `registerEffect` → `updateLocalStats`** |
| **仅同步面板、不改 Buff 表** | 部分路径只调 **`recalcLocalStats()`**（例如 **`cancelBuffStats`** 末尾已有 **`registerEffect` 之外的调用点** 时需对照具体方法） |

### 26.8 边界一览

| 情况 | 行为 |
|------|------|
| **`applyHpMpChange` 失败** | **不发 Buff 包**、**不 `registerEffect`** |
| **`use_buff_most_significant`** | **同 Stat 保留更强一层**，物品叠层仍记账 |
| **`use_buff_everlasting`** | **持续时间视为极大**，与 **`silentApplyBuff`** 组合时注意客户端表现 |
| **卡片 Buff **`isActive`** 为 false** | **`effects` 不写入**、通常 **本人也无 **`giveBuff`**** |

## 27. `cancelEffect`、`buffExpireTask`、`PlayerBuffStorage` 与 **`silentGiveBuffs`**（衔接第 26 节）

第 **26** 节写了 **`registerEffect` → `effects`/`buffEffects`** 与 **`updateLocalStats`**。本节说明：**如何卸下**、**定时过期**、**跨场景暂存**（商城 / 自由市场 / 登出），以及 **登入时如何用「已消耗时间」对齐截止时间**。精读文档里早前写的 **`writeBuffStats`** 在本分支源码中**未出现**；Buff 外观与属性主要靠 **`SET_FIELD`（`getCharInfo`）里的角色快照** + 必要时 **`giveBuff`/`cancelBuff`**。

### 27.1 三张表：`effects`、`buffEffects`、`buffExpires`

| 结构 | 键 | 语义 |
|------|-----|------|
| **`effects`** | **`BuffStat`** | **当前对每个 BuffStat「生效的那一层」**（读 **`getBuffedValue`**、**`reapplyLocalStats`** 时主要看这里） |
| **`buffEffects`** | **`buffSourceId`（技能或道具效果 ID）** | **同一来源下的多条 **`BuffStat -> BuffStatValueHolder`****（叠Buff、按来源整批移除） |
| **`buffExpires`** | **`buffSourceId`** | **该来源整批效果的绝对过期时间戳**（毫秒，与 **`Server.getInstance().getCurrentTime()`** 对齐） |

**`addItemEffectHolder`**： **`buffEffects.put(sourceid, statups)`** 且 **`buffExpires.put(sourceid, expirationtime)`**。  
**`registerEffect`** 末尾统一 **`addItemEffectHolder(sourceid, expirationtime, appliedStatups)`** 再 **`effects.putAll(toDeploy)`**（与 **`use_buff_most_significant`、`isSilent`** 的细节组合见源码 **4170–4244** 行）。

### 27.2 `cancelEffect`：`cancelEffectInternal` → **`updateLocalStats` → `updateEffects`**

公开入口 **`cancelEffect(StatEffect effect, boolean overwrite, long startTime)`**：先 **`prtLock` → `effLock`** 调私有重载；若 **`effect.isMagicDoor()`** 且返回 true，再二次加锁检查 **`hasBuffFromSourceid(MYSTIC_DOOR)`**，没有则 **`Door.attemptRemoveDoor`**。

私有 **`cancelEffect(..., firstCancel)`**：

1. **`dropBuffStats(cancelEffectInternal(...))`**：若卸下的 holder 标了 **`bestApplied`**，在 **`chrLock`** 下 **`fetchBestEffectFromItemEffectHolder`**，把 **同一 BuffStat 下一条更强的来源** 写回 **`effects`**（叠 Buff 时的「顶栈」逻辑）。  
2. **`updateLocalStats()`**：重算 **`local*`** 并发 **`updatePlayerStats`**（见第 **26** 节）。  
3. **`updateEffects(removedStats)`**：对已移除的 **`BuffStat`** 再 **`fetchBestEffectFromItemEffectHolder`**，然后 **`propagateBuffEffectUpdates`**。

**`cancelEffectInternal`** 如何选 **`buffstats`**（要从 **`buffEffects` 里摘掉哪一批）：

- **`overwrite == false`**（正常移除某来源）：**`extractCurrentBuffStats(effect)`** —— **`buffEffects.remove(effect.getBuffSourceId())`**，同步 **`buffEffectsCount`**。  
- **`overwrite == true`** 且效果含 **singleton BuffStat**：按 **singleton** 找到当前 **`effects`** 里的 **`StatEffect`**，再 **`extractCurrentBuffStats`** 那一整来源（用于 **`applyBuffEffect` 里同类顶掉**，见第 **26** 节 **`cancelEffect(this, true, -1)`**）。  
- 否则：**`extractLeastRelevantStatEffectsIfFull`** —— 统计每个 **`BuffStat`** 被多少来源占用；超过 **`max_monitored_buff_stats`**（非 singleton）时拆掉 **数值最小** 的那条来源（防 Buff 表爆炸）。

**`deregisterBuffStats`**：对 **`effects` 里来源匹配的条目 **`effects.remove`**；顺带 **`recoveryTask` / 召唤 / Beholder 定时器 / DragonBlood / HPREC·MPREC 额外回复任务** 等与 **具体 BuffStat** 绑定的副作用在此撤销。**`buffEffects` 为空条目时清 **`buffExpires`****。

### 27.3 `propagateBuffEffectUpdates` 与 **`cancelPlayerBuffs`**

**`propagateBuffEffectUpdates`**：

1. **`cancelInactiveBuffStats`**：**已在 **`effects` 里消失的 **`BuffStat`**** 发 **`cancelBuff` / `cancelForeignBuff`**（他人图标）。  
2. 对 **`retrievedStats`**（例如叠 Buff 被卸一层后仍需刷新剩余层）按 **`StatEffect` 拓扑排序，循环计算 **`WATK`/`MATK`** 是否对该职业「 relevant」，再 **`StatEffect.updateBuffEffect`** —— **按剩余时间重发 **`giveBuff`/`givePirateBuff`****，避免客户端倒计时与面板不一致。  
3. **priority 香水类**（玫瑰等 **`isPriorityBuffSourceId`**）另有 **`propagatePriorityBuffEffectUpdates`** 补丁。

**`cancelPlayerBuffs(List<BuffStat>)`**：**仅在 **`PlayerStorage` 仍能按 ID 找到本角色时** —— **`updateLocalStats`** → **`cancelBuff`/`cancelForeignBuff`**。用于 **`cancelBuffStats`** 末尾等路径。

### 27.4 `buffExpireTask`：每 1.5 秒扫 **`buffExpires`**

**`buffExpireTask()`**（仅 **`buffExpireTask == null` 时注册一次**）：**`TimerManager.register(..., 1500)`**。  
每tick：**复制 **`buffExpires`****，若 **`curTime >= expiration`**，从 **`buffEffects.get(sourceid)`** 取 **第一个 **`BuffStatValueHolder`****（注释 **rofl**，即 **按来源取一条代表**），收集后对每条 **`cancelEffect(mbsvh.effect, false, mbsvh.startTime)`**。

**`PlayerLoggedinHandler`** 在角色登录流程末尾调用 **`player.buffExpireTask()`**（与 **`diseaseExpireTask`、`skillCooldownTask`** 等并列），保证 **暂存 Buff 还原后仍会过期**。

### 27.5 `PlayerBuffStorage`：内存暂存，按角色 ID **取出即删除**

**`Server` 单例**持有 **`PlayerBuffStorage`**。**`addBuffsToStorage(cid, getAllBuffs())`** 的典型调用点：**进商城**、**进 MTS**、**`Client` 侧登出保存**等。**`getBuffsFromStorage(cid)`** 为 **`buffs.remove(chrid)`** —— **单次消费**，避免重复应用。

**`getAllBuffs`**：遍历 **`buffEffects`**，按 **`buffSourceId` 去重**，**`usedTime = curtime - startTime`**（已流逝毫秒），装入 **`PlayerBuffValueHolder`**。**Debuff** 另 **`getAllDiseases` → `addDiseasesToStorage`**（剩余时间写入 **`Pair` 左值**）。

### 27.6 登入还原：**`getLocalStartTimes` → `silentGiveBuffs` → `getCharInfo`**

**`PlayerLoggedinHandler`**（角色已进入 **`PlayerStorage`/世界** 之后）：

1. **`getBuffsFromStorage(cid)`**，若非 null：**`getLocalStartTimes`**：**`localStart = curServerTime - pb.usedTime`**，按 **`localStart` 排序**后 **`player.silentGiveBuffs(timedBuffs)`**。  
2. **`getDiseasesFromStorage`** → **`silentApplyDiseases`**（只写 **`diseases`/`diseaseExpires`**，外观包在 **`newcomer`** 分支另发 **`giveDebuff`**）。  
3. **`c.sendPacket(PacketCreator.getCharInfo(player))`**：`SET_FIELD` + **`addCharacterInfo`** —— **客户端从进场快照里看到 Buff 图标与外观**。  
4. 稍后 **`player.buffExpireTask()`** 接上服务端 **`buffExpires`**。

**`silentGiveBuffs`**：**`StatEffect.silentApplyBuff(this, localStartTime)`** → **`registerEffect(..., isSilent=true)`**。与 **`applyBuffEffect`** 的差别：**不发 **`giveBuff`/`giveForeign`* **（依赖 **`getCharInfo`** 或后续 **`propagateBuffEffectUpdates`** 对齐）；**仍 **`registerEffect` → `updateLocalStats`****，服务端 **`local*`** 与 **`effects`** 一致。

**`StatEffect.silentApplyBuff`**：另处理 **非站桩召唤**（地图上 **`spawnSummon` 不在此路径**——静默还原只在 **`chr.addSummon`** + **`addHP`**）、**战船 HP 通告**。

### 27.7 `cancelBuffStats` / `cancelAllBuffs` / `empty`

- **`cancelBuffStats(BuffStat)`**：扫描 **`buffEffects`** 中含该 Stat 的所有来源，**`extractBuffValue`**（按来源摘掉这一条 Stat），**`dropBuffStats(deregisterBuffStats(...))`**，再 **`cancelPlayerBuffs(singleton(stat))`**。  
- **`cancelAllBuffs(true)`**：软清 —— 卸召唤/木偶/Combo，**`effects.clear()`**，遍历 **`removeItemEffectHolder`**。  
- **`cancelAllBuffs(false)`**：把 **`buffEffects`** 里所有 **`StatEffect+startTime`** 逐个 **`cancelEffect`**。  
- **`Character.empty`**：**`cancelBuffExpireTask()`** 等，断开时不再扫过期。

### 27.8 边界一览

| 情况 | 行为 |
|------|------|
| **`buffExpireTask` 与真实下线** | **`empty` 取消任务**；再上线依赖 **`PlayerBuffStorage`** 是否曾 **`addBuffsToStorage`** |
| **叠 Buff 卸一层** | **`fetchBestEffectFromItemEffectHolder` + `propagateBuffEffectUpdates`** 刷新剩余层倒计时 |
| **`silentGiveBuffs` 无配套 `getCharInfo`** | 客户端可能 **缺图标**；本仓库登录路径是先静默注册再 **`getCharInfo`** |
| **路线图曾写 `writeBuffStats`** | 当前精读基于的上游 **无此符号**；以 **`getCharInfo` + Buff 包** 为准 |

## 28. **`giveDebuff`、`diseaseExpires`、`diseaseExpireTask` 与 `dispel*`**（衔接第 27 节）

服务端把 **异常状态（Debuff）** 放在 **`Character`** 的另一套路：**`diseases` + `diseaseExpires`**，与 Buff 的 **`effects`/`buffEffects`** **并行**；发包 opcode 与 Buff **共用 `GIVE_BUFF` / `CANCEL_BUFF` / `GIVE_FOREIGN_BUFF` / `CANCEL_FOREIGN_BUFF`**，靠 **掩码（双 long）** 区分 BuffStat 与 Disease。

### 28.1 **`Disease` 与掩码**

枚举 **`org.gms.client.Disease`** 每项有 **`long getValue()`**（位掩码），并可选绑定 **`MobSkillType`**（**`getBySkill`** 用于从怪物技能反查）。  
**`PacketCreator.writeLongMaskD`**：对每个 **`Pair<Disease, Integer>`**，若 **`disease.isFirst()`**（在本枚举里恒为 **false**）则并入 **第二段 mask**，否则并入 **第一段** —— 实际 **全部 Disease 走第二段 `secondmask`**。

### 28.2 **`giveDebuff(Disease, MobSkill)`**：名额、圣盾、发包

前提：**`!hasDisease(disease) && getDiseasesSize() < 2`** —— **同一 Disease 不叠**，且 **身上最多 2 个 Debuff**（实现上的硬上限）。

**主教 `HOLY_SHIELD`**：**`SEDUCE` 与 `STUN` 不受拦截**；其余 Disease 若 **`hasActiveBuff(HOLY_SHIELD)`** 直接 **`return`**。

生效时 **`chrLock`** 下：

- **`diseaseExpires.put(disease, curTime + skill.getDuration())`**  
- **`diseases.put(disease, Pair(DiseaseValueHolder(curTime, duration), skill))`**  

**`SEDUCE`** 且未坐椅子：**`sitChair(-1)`**。

发包：**本人 `PacketCreator.giveDebuff`**；地图他人 **`giveForeignDebuff`**。**`SLOW` 走单独分支**：本人仍 **`giveDebuff`**（同一 **`GIVE_BUFF` + mask**），他人 **`giveForeignSlowDebuff`**（固定 **`writeLongMaskSlowD`**，与普 Debuff 编码不同）。**`giveDebuff` 包体**（本人）：双 long 掩码 → 每个 statup：**`short(x)`、`writeMobSkillId`、`int(duration)`** → 固定 **`short(0), short(900), byte(1)`**。他人 **`giveForeignDebuff`**：**POISON** 时多写 **`short(x)`**；只写 mob skill id，**不写 duration int**（与本人包差异）。

### 28.3 **`dispelDebuff` / `dispelDebuffs` / `purgeDebuffs`**

**`dispelDebuff(Disease)`**：若持有该病：**`cancelDebuff(mask)`**（**`CANCEL_BUFF`**：**第一 long=0，第二 long=mask，`byte(0)`** —— 与取消 Buff 的 **`cancelBuff(List)`** 尾部 **`byte(1)`** 区分）→ 他人 **`cancelForeignDebuff`**，**`SLOW` 用 `cancelForeignSlowDebuff(cid)`** → **`chrLock`** 下 **`diseases`/`diseaseExpires` remove**。

**`dispelDebuffs()`**：驱散 **诅咒、黑暗、毒、封印、虚弱、减速** —— 注释写明 **`ZOMBIFY` 不在此列表**（需 **`purgeDebuffs`**）。  

**`purgeDebuffs()`**：**魅惑、僵尸、混乱** + **`dispelDebuffs()`**，等价「更接近全清异常」。

**`dispel()`**（注意名字）：清的是 **Buff** —— 遍历 **`getAllStatups()`**，**`StatEffect.isSkill()`** 且 **来源不是 `Aran.COMBO_ABILITY`** 则 **`cancelEffect`**；若 **`use_undispel_holy_shield`** 且身上有 **`HOLY_SHIELD`** 则 **整条 `dispel()` 不执行**。与 **`dispelDebuffs`** 不同对象。

### 28.4 **`diseaseExpireTask`**：与 **`buffExpireTask` 对称**

**`diseaseExpireTask()`**：**`TimerManager.register(..., 1500)`**，每 tick **`chrLock`** 下扫描 **`diseaseExpires`**，**`expTime < curTime`** 的记入 **`toExpire`**，锁外逐个 **`dispelDebuff(d)`** —— **过期路径也会发取消包**。

**`PlayerLoggedinHandler`** 在 **`buffExpireTask()`** 旁边调用 **`diseaseExpireTask()`**。  
**`Character.empty`** / **频道切换前清理**：**`cancelDiseaseExpireTask()`**；**`Client` 登出、切频道**里在 **`addDiseasesToStorage`** 之后会 **`cancelAllDebuffs()`** 清空内存（病状态由 **`PlayerBuffStorage`** 带走时再 **`silentApplyDiseases`** 还原，见第 **27** 节）。

### 28.5 **静默登录与广播：`silentApplyDiseases`、`announceDiseases`、`collectDiseases`**

**`silentApplyDiseases`**：只写 **`diseaseExpires`** / **`diseases`**（**`Pair` 内 `DiseaseValueHolder(startTime, length)`**），**不发包**。

**`PlayerLoggedinHandler`**：**`newcomer`** 时对 **`diseases`** 遍历发 **`giveDebuff`**（本人可见）；非 newcomer 若骑战船另有 HP 通告。

**`announceDiseases`**：**`isLoggedInWorld()`** 时对本地 **`diseases`** 向地图 **`giveForeignDebuff` / `giveForeignSlowDebuff`** —— 注释：**中毒伤害可见性、跨图后他人可见 debuff**。

**`collectDiseases`**：对 **地图上所有玩家** 拉取其 **`getAllDiseases()`**，向 **本角色客户端** 发送他人 **`giveForeignDebuff`** —— 用于 **新进图的人补齐视野**。

**`Server.runAnnouncePlayerDiseasesSchedule`**：**`registerAnnouncePlayerDiseases`** 入队后，下一 tick 对队列中 **`Client`** 依次 **`announceDiseases()` + `collectDiseases()`**（需 **`isLoggedInWorld()`**）。

### 28.6 与 **`playerDead`**

**`playerDead`**：**`cancelAllBuffs(false)`** 后 **`dispelDebuffs()`** —— 死亡清 **技能 Buff** 与 **常规可驱散 Debuff**；**魅惑/僵尸/混乱** 不在 **`dispelDebuffs`** 内，是否仍存在取决于后续是否另有 **`purge`** / **`cancelAllDebuffs`** 调用路径。

### 28.7 边界一览

| 情况 | 行为 |
|------|------|
| **已有 2 个 Debuff** | **`giveDebuff` 直接不满足前置** |
| **圣盾** | **拦普通异常；不拦 `SEDUCE`/`STUN`** |
| **`cancelAllDebuffs`** | **只清 Map，不发取消包**（用于登出/切频道前内存收尾） |
| **Buff 的 `dispel()` vs Debuff 的 `dispelDebuffs`** | **前者卸技能 Buff；后者卸异常 bitmask** |

## 29. **`MobSkill`、`Monster.applyStatus` / `applyMonsterBuff` 与 `MobStatusService`**（衔接第 28 节）

第 **28** 节写的是 **玩家身上的 `Disease`**。本节写 **怪物**：怪物用的 **`MobSkill`（WZ `MobSkill.img`）** 如何把 **`MonsterStatus`** 或 **`Character.giveDebuff`** 打到场上；以及 **玩家技能** 如何通过 **`Monster.applyStatus`** 把 **`MonsterStatusEffect`** 挂到怪身上并由 **`MobStatusService`** 定时撤销 / DOT。

### 29.1 **`MobSkill` 数据：`MobSkillFactory` 与字段语义**

**`MobSkillFactory`** 从 **`MobSkill.img`** 懒加载 **`MobSkill`**，键为 **`MobSkillType + level`**（**`MobSkillId`**）。构造字段主要包括：**`mpCon`**、**`hp/x/y/count`**（含义随 **`MobSkillType`** 变）、**`duration`/`cooltime`**、**`prop`（命中率）**、范围框 **`lt`/`rb`**、**`limit`**（如召唤上限）、**`toSummon`**（召唤怪 ID 列表）等。

**`makeChanceResult()`**：**`prop == 1.0` 或 `random < prop`**，失败则 **`applyEffect` 直接返回**。

### 29.2 **`MobSkill.applyEffect`**：`MonsterStatus` vs **`Disease`**

入口 **`applyEffect(Character player, Monster monster, boolean skill, List<Character> banishPlayersOutput)`**：

1. **`makeChanceResult()`**。  
2. **`switch (MobSkillType)`** 填充 **`EnumMap<MonsterStatus, Integer> stats`**，或设置 **`Disease disease`**，或走 **`HEAL_M` / `DISPEL` / `BANISH` / `AREA_POISON`（迷雾）/ `SUMMON`** 等分支。  
   - **加攻防速免疫反射等**：写入 **`stats`**，末尾 **`applyMonsterBuffs`**。  
   - **封印、黑暗、虚弱、晕、诅咒、毒、减速、魅惑、混乱、僵尸等**：映射到 **`Disease`**，末尾 **`applyDisease`**。  
   - **`DISPEL`**：**单体 `player.dispel()`**；**AOE（`lt/rb` 且 `skill`）**对范围内玩家 **`Character::dispel`** —— 卸的是 **玩家 Buff**（第 **28** 节 **`dispel()`**），不是 **`dispelDebuffs`**。  
   - **`BANISH`**：把目标玩家填入 **`banishPlayersOutput`**（调用方再处理传送）。  
   - **`AREA_POISON`**：**`spawnMonsterMist`**（毒雾 **`Mist`**，持续时间 **`x * 100`**）。  
   - **`SUMMON`**：**`summonMonsters`**（地图怪物数 **< 80**、`countAvailableMobSummons` 限制；道场 **`skillLimit`** 放宽等）。

3. **AOE vs 单体**：**`applyMonsterBuffs`** / **`applyDisease`** 内若 **`lt != null && rb != null && skill`**，则用 **`calculateBoundingBox(monster.getPosition())`** 找范围内 **怪物** 或 **玩家**；否则只对 **`monster`/`player` 单体**生效。

4. **`applyDisease`（怪物→人）**：AOE 时对 **`getPlayersInRange`**：**无圣盾**才 **`giveDebuff`**；**`SEDUCE`** 最多 **`count`** 个玩家依次施加。单体分支直接 **`player.giveDebuff`**（圣盾判断在 **`giveDebuff`** 内，见第 **28** 节）。

### 29.3 **`Monster.applyMonsterBuff`**（怪物侧 Buff / 反射）

**`applyMonsterBuff(stats, x, duration, skill, reflection)`**：

- 构造 **`MonsterStatusEffect(stats, null, skill, monsterSkill=true)`**。  
- **`PacketCreator.applyMonsterStatus(oid, effect, reflection)`** —— **`reflection`** 用于 **反伤类**技能带的 **`List<Integer>`**。  
- **`broadcastMonsterStatusMessage`**：地图广播 + **当前控制器若看不见怪也会补一包**（避免控怪客户端缺状态）。  
- **`stati` map**：每个 **`MonsterStatus` → 该 `MonsterStatusEffect`**，并记入 **`alreadyBuffed`**。  
- **`MobStatusService.registerMobStatus(mapId, effect, cancelTask, duration)`**：到期 **`cancelMonsterStatus`** 并从 **`stati` 移除。

### 29.4 **`Monster.applyStatus`**（玩家技能→怪物异常）

**`applyStatus(Character from, MonsterStatusEffect status, boolean poison, long duration, boolean venom)`** —— **`MonsterStatusEffect`** 通常来自 **`StatEffect.getMonsterStati()` + 玩家 **`Skill`**（**`monsterSkill=false`**）。

**门禁**：

- **`getMonsterEffectiveness(skill.getElement())`**：**`IMMUNE`/`STRONG`/`NEUTRAL` → false**；仅 **`NORMAL`/`WEAK` 继续**。  
- **FP/IL 毒雾合成、夜行者毒等**：另有 **毒/冰抗性** 特殊判断。  
- **`poison && hp<=1`**：**false**。  
- **Boss**：除非 **`status` 同时包含 **`SPEED` + `NINJA_AMBUSH` + `WATK`** 三个键（源码字面条件），否则 **拒绝**大部分玩家上状态 —— 等价于 **Boss 几乎只吃特定组合**，其它 **`MonsterStatus`** 会被挡。

**顶替同一 `MonsterStatus`**：**`statiLock`** 下若已有 **`oldEffect`**，**`removeActiveStatus`**；若 **`oldEffect` 空**，**`MobStatusService.interruptMobStatus`** 取消旧定时任务。

**广播**：**`broadcastStatusEffect`** → **`animationTime = skill.getAnimationTime()`**。

**持续伤害**：  
- **`poison`**：按技能等级算 **`poisonDamage`**，写入 **`MonsterStatus.POISON`**，**`DamageTask`** **`overtimeDelay=1000`**。  
- **`venom`**：夜行者/标飞/刀飞分支算伤害，**`VENOMOUS_WEAPON` + POISON**，同样 **`DamageTask`**。  
- **忍者伏击**：**`NINJA_AMBUSH`**，`DamageTask` **`type=2`**，delay **1000**。  
- 其它：**仅动画广播**。

最后 **`registerMobStatus(..., duration + animationTime - 100, overtimeAction, overtimeDelay)`** —— 总时长 **含动画微调 −100ms**；若有 **`overtimeAction`**，**`MobStatusScheduler`** 按 **`mob_status_monitor_proc`** 周期触发 DOT（见 **29.5**）。

**`DamageTask.run`**：**HP≤1** 或 **伤害钳制到留 1 血**（视 **`type`**）会 **`interruptMobStatus`**；否则 **`applyDamage(from, damage, ...)`** 并视类型 **`damageMonster` 广播**。

### 29.5 **`MobStatusService`**：分片调度、DOT、打断

**`MobStatusService`** 按 **`channel_locks`** 对 **`mapId` 哈希分片**，每片一个 **`MobStatusScheduler`（`BaseScheduler`）**。

- **`registerMobStatus`**：可选注册 **`MobStatusOvertimeEntry`**（**`ceil(overtimeDelay / mob_status_monitor_proc)`** 次 tick 执行一次 **`overtimeStatus`**）。  
- **`interruptMobStatus`**：**顶替或 DOT 终止**时取消 **`MonsterStatusEffect`** 对应的调度项。

### 29.6 **怪物使用技能：`canUseSkill` / `usedSkills` / `MobClearSkillService`**

**`canUseSkill(MobSkill toUse, boolean apply)`**：**封印技能 `SEAL_SKILL`**、**反射类互斥**、**本怪 `usedSkills` 冷却表**、**MP 不足** → **false**。**`apply==true`** 时 **`usedSkill`**：**扣 MP**、**登记冷却**，**`MobClearSkillService.registerMobClearSkillAction`** 到期 **`clearSkill`**。

（攻击位 **`canUseAttack`/`usedAttack`** 同理，走 **`MonsterInformationProvider` 攻击 MP 与冷却**。）

### 29.7 **`dispelSkill` / `debuffMob`**（玩家驱散怪物 Buff）

- **`Monster.dispelSkill(MobSkill skill)`**：遍历 **`stati`**，**`mse.getMobSkill().getType() == skill.getType()`**（**不比 level**）则 **`debuffMobStat`**。  
- **`debuffMob(skillid)`**：**祭司驱散 / Crash / 金钱投掷**等分支，摘掉 **`WEAPON_ATTACK_UP` 等**，可选 **`use_anti_immunity_crash`** 卸 **无敌**。

### 29.8 与 **第 28** 节的对照

| 维度 | **怪物→玩家（`MobSkill`）** | **玩家→怪物（`applyStatus`）** |
|------|------------------------------|--------------------------------|
| 状态枚举 | **`Disease` → `giveDebuff`** | **`MonsterStatus` + `MonsterStatusEffect`** |
| 发包 | **`GIVE_BUFF` / `GIVE_FOREIGN_BUFF`（人）** | **`applyMonsterStatus` / `cancelMonsterStatus`（怪）** |
| 调度 | **`diseaseExpireTask`（人）** | **`MobStatusService`（怪）** |

### 29.9 边界一览

| 情况 | 行为 |
|------|------|
| **`MobSkill.prop` 失败** | **整段 `applyEffect` 不执行** |
| **Boss + 玩家 `applyStatus`** | **绝大多数状态被拒**（除非满足三字键组合） |
| **怪物 `DISPEL` AOE** | **对玩家 `dispel()`**，**不清 Debuff** |
| **毒 DOT 把怪打到 1 血** | **`interruptMobStatus` 终止后续 DOT** |

## 30. **`RecvOpcode.MOVE_LIFE` → `MoveLifeHandler`**：控怪、走位广播、技能 **`applyDelayedEffect`**

承接 **第 29 节**：怪物 **`MobSkill`** 的数值与 **`applyEffect`** 已定；**何时施放、是否与位移包绑定**，主要由 **`MoveLifeHandler`**（客户端 **`MOVE_LIFE`**）驱动：**当前地图上的「控制器」玩家**上报怪的位移与动作帧，服务端校验 MP/冷却/血量阈值后，**立刻或延时**执行 **`MobSkill.applyEffect`**，并向全图广播 **`MOVE_MONSTER`**。

### 30.1 入口与前置：**仅控制器、防串图**

**`handlePacket`**：若 **`player.isChangingMaps()`** 直接 **`return`**（防地图切换 OID 错乱）。  
按 **`objectid`** 取 **`MapObject`**，必须是 **`MONSTER`**。

### 30.2 拆包（与位移广播共用 **`AbstractMovementPacketHandler`**）

前几字段：**`moveId`**、**`pNibbles`**、**`rawActivity`**、**`skillId`/`skillLv`（byte）**、**`pOption`**，再 **`skip(8)`**。  
若 **`rawActivity >= 0`**：**`rawActivity = (byte)(rawActivity & (0xFF >> 1))`** —— 即 **`& 0x7F`**（清最高位）。

**动作分类**：私有方法 **`inRangeInclusive`** 的实现为 **`!(pVal < pMin) || (pVal > pMax)`**，并非字面意义上的闭区间。代入后可简化为：

- **`isAttack`** 等价于 **`rawActivity >= 24`**（对 **`rawActivity < 0`** 需另行看符号扩展，常见位移帧为非负）。  
- **`isSkill`** 等价于 **`rawActivity >= 42`**。

二者同时为真时 **`if (isSkill)` 优先**，因此 **≥42 的帧走技能分支**，**\[24,41)** 走下面的 **`canUseAttack`** 分支。

### 30.3 **技能分支：`hasSkill` → `canUseSkill(true)` → 延时或立刻 `applyEffect`**

若判为 **技能帧**：**`useSkillId/useSkillLevel`** 取自包内 byte。

1. **`monster.hasSkill(useSkillId, useSkillLevel)`** 为真才继续。  
2. **`MobSkillFactory.getMobSkillOrThrow`**。  
3. **`monster.canUseSkill(toUse, true)`**：**`apply==true`** 时会 **`usedSkill`** —— **扣 MP**、记入 **`usedSkills`**、**`MobClearSkillService`** 登记 **`cooltime`** 后 **`clearSkill`**（第 **29** 节）。失败则整段不施放。  
4. **`animationTime = MonsterInformationProvider.getMobSkillAnimationTime(toUse)`**：  
   - 若 **`animationTime > 0` 且类型不是 `BANISH`**：**`toUse.applyDelayedEffect(player, monster, true, animationTime)`** —— **`OverallService.registerOverallAction(mapId, runnable, animationTime)`**，在动画时长后 **`applyEffect(player, monster, true, null)`**（怪仍存活时）。  
   - 否则：**立刻 `applyEffect`**；若需要放逐，准备 **`banishPlayers`** 列表。

**`applyEffect` 第四个参数 `skill` 为 `true`**：怪物技能走 **AOE（`lt/rb`）** 分支（第 **29** 节 **`applyMonsterBuffs`/`applyDisease`**）。

### 30.4 **攻击分支：`canUseAttack` 与非法攻击回退**

非技能帧：**`castPos = (rawActivity - 24) / 2`**，**`monster.canUseAttack(castPos, isSkill)`**。  
若 **`< 1`**：把 **`rawActivity = -1`、`pOption = 0`** —— **后续广播里表现为未打出合法攻击**（省 MP、不占冷却的逻辑在 **`canUseAttack`/`usedAttack`** 内）。

### 30.5 **「下一tick 可能放技能」：`getRandomSkill` 与 `MOVE_MONSTER_RESPONSE`**

**`nextMovementCouldBeSkill = !(isSkill || (pNibbles != 0))`** —— **本轮既不是技能位移，`pNibbles` 又为 0** 时，服务端认为 **下一轮客户端可能插入怪物技能**。

此时若怪 **`hasAnySkill()`**：**`getRandomSkill()`** 均匀抽 **`MobSkillId`**，取出 **`nextUse`**，并校验：

- **`canUseSkill(nextUse, false)`**（**不扣 MP、不登记 used**，只做可否预览）；  
- **血量百分比 ≥ `nextUse.getHP()`**（WZ 里配置的 **HP 阈值**，百分制）；  
- **`mobMp >= mpCon`**。

通过则在 **`MOVE_MONSTER_RESPONSE`** 里附带 **`nextSkillId/nextSkillLevel`**（**`PacketCreator.moveMonsterResponse`**：**`useSkills` 实为 `aggro` 布尔**，字段名历史遗留）；否则 **skill 两项为 0**。客户端据此预判怪物下一步技能；**真正结算仍以客户端上报的技能帧 + 服务端 `canUseSkill(..., true)` 为准**。

### 30.6 **`aggroMoveLifeUpdate`：只有控制器能驱动整条链路**

**`monster.aggroMoveLifeUpdate(player)`**：仅当 **`player` 是该怪当前 `controller`** 时返回 **`Boolean`**（是否 **controllerHasAggro**）；**非控制器返回 `null`** → **`MoveLifeHandler` 直接 `return`** —— **他人发来的 MOVE_LIFE 不会生效**。这与 **`MonsterAggroCoordinator`/`aggroMonsterDamage`** 切换控制器（第 **29** 节引用）配合：**谁能控怪，谁的位移包才算数**。

### 30.7 **广播与地图同步**

1. **`moveMonsterResponse`** → **仅发送给控制器 **`c`****。  
2. **`updatePosition(p, monster, -2)`** 解析剩余字节更新 **`monster`** 坐标（与 **`AbstractMovementPacketHandler`** 一致，**−2** 为脚注里的海绵 Boss 修正）。  
3. **`map.broadcastMessage(..., PacketCreator.moveMonster(...), serverStartPos)`** —— **`MOVE_MONSTER`**：`skillPossible`、`rawActivity`、`useSkillId/Lv`、`pOption`、起点与 movement blob。  
4. **`map.moveMonster(monster, monster.getPosition())`**：**刷新各玩家视野内怪的 spawn/destroy**（**`updateMapObjectVisibility`**）。

### 30.8 **放逐：**`BANISH` **与 **`changeMapBanish`**

若 **`applyEffect`** 填充了 **`banishPlayers`**（**`MobSkillType.BANISH`** 且非延时路径）：处理器末尾对列表 **`chr.changeMapBanish(...)`**，使用怪物 **`getBanish()`** 配置的地图、传送点与提示。

### 30.9 **`applyAnimationIfRoaming` 注释**

**`Monster.canUseSkill` / `canUseAttack`** 内有一段 **`applyAnimationIfRoaming` 调用被注释掉** —— 设计上曾打算用 **`MobAnimationService`** 限制「漫游态不打动画」；当前分支 **不启用**，动画冲突主要靠客户端位移与 **`OverallService` 延时** 对齐。

### 30.10 边界一览

| 情况 | 行为 |
|------|------|
| **非控制器发 MOVE_LIFE** | **`aggroMoveLifeUpdate == null` → 丢弃** |
| **技能动画时间 > 0 且非 BANISH** | **`applyDelayedEffect`**；否则 **当场 `applyEffect`** |
| **`nextUse` 预览失败** | **响应包 skillId/Lv 为 0**；不影响本轮位移广播 |
| **`EmptyMovementException`** | **静默吞掉**（不广播位移） |

## 31. **`MonsterAggroCoordinator`、控制器与傀儡仇恨**（衔接第 30 节）

第 **30** 节说明：**只有当前「控制器」** 的 **`MOVE_LIFE`** 会生效。本节说明：**谁当上控制器**、**挨打后何时换人**、**地图上周期性仇恨表如何维护**，以及 **傀儡（`BuffStat.PUPPET`）如何插队控怪**。

### 31.1 **`Monster` 侧状态（WeakReference + 三个布尔）**

- **`controller`**：**`WeakReference<Character>`**，当前 **客户端 AI 驱动该怪的玩家**。  
- **`controllerHasAggro`**：**怪是否处于「有仇恨」** —— 影响 **`MOVE_MONSTER_RESPONSE`** 里的 **`useSkills` 参数**（实为 aggro 标志）、以及 **`aggroMonsterControl(..., immediateAggro)`**。  
- **`controllerKnowsAboutAggro`**：**控制器客户端是否已被告知「我知道自己有仇恨」** —— **`aggroMoveLifeUpdate`** 里若 **`controllerHasAggro`**，则 **`setControllerKnowsAboutAggro(true)`**（第 **30** 节）。  
- **`controllerHasPuppet`**：**本轮木偶刷新流程是否认定控制器带着傀儡视野**。  
- **`aggroUpdateLock`**：**`aggroSwitchController`/`aggroRemoveController`** 等串行化，避免并发换人炸状态。

**换控制器发包**：**`PacketCreator.controlMonster`**（**`aggroMonsterControl`**）；卸下：**`stopControllingMonster`** + **`Character.stopControllingMonster`**。

### 31.2 **`MapleMap` 上的协调器：何时启动**

每张 **`MapleMap`** 构造 **`MonsterAggroCoordinator aggroMonitor`**。  

**`addPlayer`**：若 **本图人数从 0→1** 且 **`startItemMonitor()`**：顺带 **`aggroMonitor.startAggroCoordinator()`**。  

**物品监视 tick**：若 **`characters.isEmpty()`** 且超时：`stopItemMonitor()` 时 **`aggroMonitor.stopAggroCoordinator()`** —— **无人地图停表**，减少空转。

**定时任务**：**`TimerManager.register`**，周期 **`mob_status_aggro_interval`**，每 tick：**`runAggroUpdate(1)`** + **`runSortLeadingCharactersAggro()`**。启动时会根据停机时长 **`runAggroUpdate(timeDelta)`** 追赶过期。

### 31.3 **`PlayerAggroEntry`：伤害衰减与排序**

**`addAggroDamage(mob, cid, damage)`**（典型来自 **`monster.aggroMonsterDamage`** → **`AbstractDealDamageHandler`**、反射伤害 **`TakeDamageHandler`**、磁铁 **`SpecialMoveHandler`** 等）：

- 全局 **`lock.tryLock()`** 懒建 **`mobAggroEntries`/`mobSortedAggros`** —— **拿不到锁直接 return**（注释：**尽量快、允许丢采样**）。  
- **`PlayerAggroEntry`**：维护 **`averageDamage`、`currentDamageInstances`、`accumulatedDamage`**，以及 **`toNextUpdate`**（下次衰减 tick）。新受伤 **`insertEntryDamage`**：滚动平均；**`damage < 1` 且条目已存在** 时 **直接 return**（不吃 0 伤害刷新）。

**`runAggroUpdate`**：对每个条目 **`expiredAfterUpdateEntryDamage`** —— **每个 interval 扣 `toNextUpdate`**，到期则 **`currentDamageInstances--`**，实例归零则 **移出该玩家**；若 **整张怪的表清空** 且 **非 Boss**：**`mob.aggroResetAggro()`** —— **清控制器身上的仇恨标志**（**不换人**，仅 **`setControllerHasAggro(false)`**）。

**`runSortLeadingCharactersAggro`**：**`insertionSortAggroList`** 按 **`accumulatedDamage` 降序**，并重写 **`entryRank`**。

### 31.4 **`isLeadingCharacterAggro`：谁能抢走控制权**

**`Monster.aggroMonsterDamage`**：先 **`mmac.addAggroDamage`**，再：

- 若 **当前控制器 != 攻击者**：若 **`isLeadingCharacterAggro(this, attacker)`** → **`aggroSwitchController(attacker, true)`**；否则 **原地 `setControllerHasAggro(true)`** + **`aggroUpdatePuppetVisibility()`**。  
- 若 **已是控制器**：**`setControllerHasAggro(true)`** + 傀儡可见性刷新。

**`isLeadingCharacterAggro(mob, player)`** 规则摘要：

1. 若 **`mob.isLeadingPuppetInVicinity()`**（**当前控制器脚下有傀儡圈**）→ **恒 false**，**阻止 DPS 列表抢走控制** —— **傀儡控场优先**。  
2. 否则若 **`mob.isCharacterPuppetInVicinity(player)`**（**攻击者本人在傀儡圈内**）→ **true**。  
3. 否则扫 **`sortedAggro` 前至多 5 名**，找在线 **`Character`**：  
   - 若 **列表里先碰到 `player`** → **true**；  
   - 若碰到 **别的活人** 且其 **`updateStreak < mob_status_aggro_persistence`** → **false**（**领先者「活跃度」保护**）；  
   - 否则继续扫。  
4. 扫完仍无结论 → **false**。

### 31.5 **傀儡：**`isPuppetInVicinity` **与 **`mapPuppetEntries`**

**距离**：**`summon.getPosition().distanceSq(mob.getPosition()) < 177777`**（约 **421px** 量级，具体以距离平方为准）。

**`isCharacterPuppetInVicinity`**：读 **`BuffStat.PUPPET`** → **`getSummonByKey`**；无召唤物则从 **`MonsterAggroCoordinator.removePuppetAggro(cid)`** 清理注册。

**`aggroAddPuppet`/`aggroRemovePuppet`**：**`mapPuppetEntries`** 记录 **当前地图有哪些角色宣称挂了傀儡**；**`aggroUpdatePuppetController`** 在 **控制器失效或失去傀儡圈** 时，遍历该列表 **寻找仍在傀儡范围内的候选人** 并 **`aggroSwitchController(..., 保留原 hasAggro)`**。

**`aggroRefreshPuppetVisibility`**：**对已控多只怪的弓箭手类补丁** —— 逐个 **`stopControllingMonster`** → **`removeSummon`（仅发包）** → **重新 `controlMonster`** → **`spawnSummon`**，迫使客户端 **把仇恨视觉绑回傀儡**（注释 **lame patch**）。

**`aggroUpdatePuppetVisibility`**：若 **`PUPPET` Buff 存在**，异步 **`OverallService.registerOverallAction(..., update_interval)`** 触发上述刷新；否则 **补发 `controlMonster` 复位**。

### 31.6 **无伤害时的控怪：**`AutoAggroHandler` **与 **`getNextControllerCandidate`**

**`AutoAggroHandler`**（客户端主动）：未隐身则 **`monster.aggroAutoAggroUpdate(player)`** —— **无控制器则 `aggroSwitchController(player, true)`**；已是控制器则 **`setControllerHasAggro(true)`**，并按 **`use_auto_aggro_nearby`** 决定是否 **`controlMonster` 再通告。

**`aggroUpdateController`**：当前控制器 **null 或死亡** 时，**`getNextControllerCandidate()`**：  
遍历地图玩家（**未隐身**），**优先第一个「傀儡在圈内」**；否则 **活人里「已控怪数量最少」**；再不行 **死人里同样规则**。

### 31.7 **清空仇恨表：**`aggroClearDamages`

**`processMonsterKilled`**：**`aggroClearDamages()`** → **`removeAggroEntries`**，防止尸体残留表。  

**怪物磁铁**：非 Boss 先 **`aggroClearDamages()`**，再 **`aggroMonsterDamage(chr, 1)`**，最后 **`aggroSwitchController(chr, true)`** —— **强制拉控并重算领先**。

### 31.8 边界一览

| 情况 | 行为 |
|------|------|
| **`addAggroDamage` 拿不到全局 lock** | **静默丢弃本次采样** |
| **仇恨表自然枯竭（非 Boss）** | **`aggroResetAggro`**，**控制器未必卸任** |
| **控制器脚下傀儡生效** | **`isLeadingCharacterAggro` 对他人多为 false** |
| **地图最后一人离开并停机监视** | **`stopAggroCoordinator`** |

## 32. **`Monster.damage` / `MapleMap.damageMonster`、反伤与无敌钳制**（衔接第 31 节）

常用入口：**`map.damageMonster(Character chr, Monster monster, int damage)`**。在此之前，**`AbstractDealDamageHandler`**、**`TakeDamageHandler`**（反弹）、**召唤 / 场地伤害 Handler** 等已各自算出 **`damage`**。本节串 **HP 结算、`takenDamage`、击杀触发**，并与 **物/魔无敌、怪物反伤 Buff** 对照。

### 32.1 **`MapleMap.damageMonster`**：扎昆臂特例 → **`monster.damage` → 自爆 HP → `killMonster`**

1. **`MobId.ZAKUM_1`**：若场上仍存在 **扎昆手臂 ID 区间** 内的怪，**直接 `return true`** 且 **不写伤害**（防止本体提前被打穿的分支逻辑）。  
2. **`monster.isAlive()`** 时：**`killed = monster.damage(chr, damage, false)`**（**`stayAlive=false`**：允许致死）。  
3. **`selfDestruction`**：若配置了 **触发 HP 阈值**，当前 **`monster.getHp()`** 达标则 **`killMonster(..., selfDestr.getAction())`** 并 **`return true`**。  
4. 否则若 **`killed`**：**`killMonster(monster, chr, true)`**（默认带掉落）。  
5. 已死怪物：**`return false`**。

### 32.2 **`Monster.damage`**：锁、`applyDamage`、`lastHit`

**`lockMonster`** 下：**已死则 false**。  

**`damage > 0`** 才 **`applyDamage(attacker, damage, stayAlive, fake=false)`**；若随后 **`!isAlive()`** → **`lastHit = true`**。**`damage <= 0`**（Miss 或全被钳成 0）**不写 HP**、不触发击杀返回值。

### 32.3 **`applyDamage`**：**`applyAndGetHpDamage` → `takenDamage` → `broadcastMobHpBar`**

**`trueDamage = applyAndGetHpDamage(damage, stayAlive)`**：已在 **`Monster`** 内实现 **扣血 / 留 1 血** 等（**`stayAlive`** 为毒 DOT 等场景服务）。  

非 **`fake`**：**`dispatchMonsterDamaged(from, trueDamage)`** → **`MonsterListener.monsterDamaged`**（事件脚本 / PQ 等挂钩点）。

**`takenDamage`**：**`AtomicLong` 按攻击者角色 ID 累计真实伤害** —— **经验分配、掉落归属**（精读 **第 23–24 节**）读这里；与 **`MonsterAggroCoordinator`** 的 **`addAggroDamage`** **并行两套表**（仇恨领先看 Coordinator，**击杀结算看 `takenDamage`**）。

**`broadcastMobHpBar`**：Boss 条或 **`showMonsterHP`**（组队可见性分支）。

### 32.4 **`AbstractDealDamageHandler`：无敌把每段伤害改成 1（钳制在汇总之前）**

对每个受害怪物、每条伤害列表：

- **魔法**且 **`MAGIC_IMMUNITY`**：**`Collections.fill(onedList, 1)`**。  
- **非魔法**且 **`WEAPON_IMMUNITY`**：同上。

之后再 **循环累加 `totDamageToOneMonster`**，并 **`monster.aggroMonsterDamage(player, totDamageToOneMonster)`** —— 注意：**传入 Coordinator 的是钳制后的总和**（可能是一大串 **1**）。

随后 **`map.damageMonster(player, monster, totDamageToOneMonster)`** 写入 **`takenDamage`** 的也是 **钳制后伤害**。

### 32.5 怪物 `WEAPON_REFLECT` / `MAGIC_REFLECT`：反伤玩家 HP（相对 `damageMonster` 的次序）

在 **`damageMonster` 成功结算之后**，若怪仍 **Buff**：

- **物理反伤**（**`WEAPON_REFLECT` + 本次攻击非魔法**）：遍历 **`monster.getSkills()`**，找到类型 **`PHYSICAL_AND_MAGIC_COUNTER`** 的 **`MobSkill`**，对玩家 **`addHP(-toUse.getX())`** 并 **`damagePlayer` 广播**。  
- **魔法反伤**：对称使用 **`getY()`**。

因此：**即使无敌把伤害打成 1，反伤段仍会执行**（只要 Buff 还在）；反伤数值来自 **WZ 里该 MobSkill 的 x/y**，不是玩家打出伤害的百分比。

### 32.6 **`TakeDamageHandler`：玩家挨怪打时的「反弹伤害」也是 `damageMonster`**

- **`Mana Reflection`**（主教系 Buff）：按 **`damage * X%`** 封顶 **`attacker.maxHp/5`**，**`map.damageMonster(chr, attacker, bouncedamage)`** + **`damageMonster` 广播**。  
- **`POWERGUARD`**（**`damagefrom == -1`** 触碰类）：伤害 **`damage * PG%/100`**，Boss **分母 200**；**从玩家受伤里减去反弹部分**，再 **`map.damageMonster`**；并 **`attacker.aggroMonsterDamage(chr, bouncedamage)`** —— **把反弹伤害记入怪的仇恨表**（仅 PG 路径显式调用，普通普攻 **`AbstractDealDamageHandler`** 已在汇总处 **`aggroMonsterDamage`**）。

其它：**`Body Pressure`** 等对攻击者 **`monster.applyStatus`**（**`NEUTRALISE`**）等与 **受伤减免** 同在 **`TakeDamageHandler`** 后半。

### 32.7 延时 `damageMonsterWithSkill`

**天堂之锤 / Combo Tempest** 等：**`TimerManager.schedule(animationTime, ...)`** 再 **`broadcastMessage(damageMonster)` + `damageMonster`** —— **伤害与击杀判定落后于技能动画**，避免客户端不同步。

### 32.8 与 `SEAL_SKILL`（第 29 节）

**`Monster.canUseSkill`**：**`isBuffed(SEAL_SKILL)`** 直接 **false** —— **怪物无法开启自带 MobSkill**；**普攻 **`canUseAttack`** 另有 MP / 冷却分支**。这与 **`damageMonster`** 无直接调用关系，属 **「怪打不死人但你也别让怪开大」** 的另一条战线。

### 32.9 边界一览

| 情况 | 行为 |
|------|------|
| **`damage <= 0`** | **`Monster.damage` 不调用 `applyDamage`** |
| **`applyAndGetHpDamage` 返回 null**（已死竞态） | **不落 `takenDamage`** |
| **扎昆本体 + 手臂仍在** | **`damageMonster` 早退 true** |
| **反弹 / 反伤** | **多为「先扣怪血，再扣玩家血」或并行广播** |

## 33. **`killMonster`、`removeKilledMonsterObject`、`killBy` 与 `dispatchMonsterKilled`**（衔接第 32 节）

第 **22–24** 节已从掉落与经验角度提过 **`killMonster` → `killBy` → `dropFromMonster` / `distributeExperience`**。本节按 **源码执行顺序** 把 **地图对象移除、经验与复活调度、EIM 钩子、`MonsterListener`、任务计数** 钉在一起，并标出 **`chr == null` 与 `chr != null` 的差异**。

### 33.1 **`removeKilledMonsterObject`**：幂等门闩 **`hp < 0`**

**`monster.lockMonster()`** 下：

- 若 **`monster.getHp() < 0`**：**`return false`** —— **同一实例第二次击杀路径会直接退出**（配合 **`disposeMapObject` 将 HP 置 −1**，见 **32** 节）。  
- 否则：**`spawnedMonstersOnMap--`**，**`removeMapObject(monster)`**，**`monster.disposeMapObject()`**（**HP → −1**），Boss 条补 **`broadcastBossHpMessage`**。

**含义**：怪从 **`MapleMap` 对象表** 摘掉，**OID 不再参与地图遍历**；**`Monster` 堆实例仍可存活到 `killMonster` 尾部逻辑跑完**（**`killBy`、掉落仍用同一引用**）。

### 33.2 **`killMonster(monster, chr, withDrops, animation)` 分支**

**`chr == null`**（环境杀 / **`killAllMonsters`** 等）：  

**`removeKilledMonsterObject`** 成功则：**`dispatchMonsterKilled(false)`** → **`broadcastMessage(killMonster)`** → **`aggroSwitchController(null, false)`**。  

**无 **`killBy`**、无掉落、无 CP/Buff 击杀特效** —— **`hasKiller=false`** 整条 **`processMonsterKilled`** 会 **`dispatchRaiseQuestMobCount`**（对 **`takenDamage`** 里出现过的人 **`raiseQuestMobCount`**），且 **不调 **`giveExpToCharacter`****（经验在 **`killBy`** 里发，见下）。

**`chr != null`**（玩家上下文击杀）：  

**`removeKilledMonsterObject`** 成功后进入 **`try`**：

1. **等级差 Autoban 提示**、**嘉年华 CP**、**全图击杀 Buff 道具 **`buffToGive`**（对每个活人 **`StatEffect.applyTo`**）。  
2. **扎昆手臂**：若打掉的是手臂，扫全场是否还有其它手臂；若没有 → **`makeMonsterReal`** 扎昆本体。  
3. **`Character dropOwner = monster.killBy(chr)`** —— **核心**：**经验分配**、**控制器卸下与复活怪 **`TimerManager.schedule(die1)`**、**掉落归属候选 **`looter`****。  
4. **`withDrops && !dropsDisabled()`**：**`dropFromMonster(dropOwner == null ? chr : dropOwner, monster, false)`**。  
5. **Boss HP 条 targeting**：**`resetPlayerAggro`**。

**`finally`**（**无论 try 是否抛错**，避免泄漏）：**`monster.dispatchMonsterKilled(true)`** + **`broadcastMessage(killMonster)`**。  

注意：**死亡广播在 `finally`**，晚于 **`killBy`**；与 **`chr==null` 分支「先 dispatch 再 broadcast」顺序相反**。

### 33.3 **`killBy(killer)`**：经验 → **卸控** → **复活表 → 归还 `looter`**

1. **`distributeExperience(killerId)`** —— 读 **`takenDamage`**（**第 24** 节）；发生在 **`killMonster` 的 `finally` → `dispatchMonsterKilled` 之前**，因此 **先于 **`MonsterListener.monsterKilled` / `eim.monsterKilled`** 触发**（脚本侧通常假定经验已入账）。**`processMonsterKilled` 不清空 **`takenDamage` HashMap****；随后 **`dropFromMonster`** 仍可用 **`retrieveRelevantDrops`** 按 **`takenDamage` 过滤掉落池**。  
2. **`aggroRemoveController()`**：记下 **`lastController` + 是否有仇恨**，供复活怪 **`aggroSwitchController`**。  
3. **`getRevives()` 非空**：**`die1` 动画毫秒后** 遍历 **`toSpawn`**：**`LifeFactory.getMonster`** → **继承坐标/fh/parentOid**，可选 **`disableDrops`**，**`spawnMonster`**；特殊 **暗黑龙尸体** 分支可连锁 **`killMonster(ht, …)`**；否则若有 **`lastController`** → **新怪继承控制**；**`eim.reviveMonster(mob)`**。道场 **透明道具**、**timeMob** 广播等副作用在同块。  
4. **`return map.getCharacterById(getHighestDamagerId()) ?? killer`** —— **掉落主归属**（精读 **第 23** 节）。

**`toSpawn == null`**：**warn**（数据异常），仍返回 **looter**。

### 33.4 **`dispatchMonsterKilled(hasKiller)` → `processMonsterKilled`**

**`processMonsterKilled`**：

- **`!hasKiller`**：**`dispatchRaiseQuestMobCount`** —— **凡打进过 **`takenDamage`** 且仍在图的角色 **`raiseQuestMobCount(mobId)`**（**不计经验但仍可推进击杀计数类任务**）。  
- **`aggroClearDamages()`**： **清空 **`MonsterAggroCoordinator`** 里该怪的条目**（第 **31** 节）。  
- **`dispatchClearSummons`**（召唤物副作用）。  
- 逐个 **`MonsterListener.monsterKilled(die1 动画时间)`**。  
- **`stati`/`alreadyBuffed`/`listeners` 清空**。

随后 **`dispatchMonsterKilled`** 外层：若地图 **`EventInstanceManager != null`**：**敌对 **`eim.monsterKilled(this, hasKiller)`**，友善 **`friendlyKilled`** —— **脚本/PQ 进度**（与 **`MonsterListener`** 并行两类钩子）。

### 33.5 **批量与友好怪：**`softKillAllMonsters` **等**

- **`softKillAllMonsters`**：**`removeKilledMonsterObject` + `dispatchMonsterKilled(false)`**，**不发 **`killMonster` 包** —— **偏服务端清理**（客户端可能仍显示怪直至另行同步，取决于调用场景）。  
- **`killAllMonstersNotFriendly` / `killAllMonsters`**：**`killMonster(..., null, false, 1)`** —— **统一走无击杀者分支**。  
- **`killMonsterWithDrops(mobId)`**：取 **`getHighestDamagerId()`** 对应 **`Character`**，缺失则用 **图上任意一名玩家** 充 **`chr`**。

### 33.6 与 **第 32** 节的拼接

**`damageMonster`** 在 **`monster.damage` 返回 `lastHit`** 时调用 **`killMonster(monster, chr, true)`** —— **带掉落**、**`animation` 默认 1**。  

**`selfDestruction`**：**`killMonster(..., selfDestr.getAction())`** 自定义客户端死亡动画。

### 33.7 边界一览

| 情况 | 行为 |
|------|------|
| **`removeKilledMonsterObject` 返回 false** | **整段 `killMonster` 后续不执行**（已 dispose） |
| **`chr != null` 抛异常** | **`finally` 仍会 `dispatchMonsterKilled(true)` + 死亡广播** |
| **无击杀者的击杀** | **`dispatchMonsterKilled(false)` → 任务计数仍可能涨** |
| **经验 vs 收尾** | **`distributeExperience` 在 `killBy` 内；`dispatchMonsterKilled` 在 `killMonster` 的 `finally`**；**`takenDamage` Map 不因 `processMonsterKilled` 而清空** |

## 34. **`EventInstanceManager`：脚本引擎、`MapManager` 与击杀钩子**（衔接第 33 节）

**`org.gms.scripting.event.EventInstanceManager`**（下文 **EIM**）是 **PQ / 副本实例** 的服务端骨架：**每张事件地图** 持有 **`MapleMap.eventInstance` 引用**，**每名在场成员** 持有 **`Character.eventInstance`**；**Nashorn/Graal `Invocable`** 通过 **`EventManager`** 调用 **`*.js`（或脚本资源）里 export 的函数**。

### 34.1 **构造：`MapManager` + 读写锁**

**`EventInstanceManager(EventManager em, String name)`**：

- **`mapManager = new MapManager(this, worldId, channelId)`** —— **本实例独享一套地图缓存**，与频道 **`MapFactory` 全局地图**隔离。  
- **`EventScriptScheduler ess`**： **`schedule(methodName, delay)`** 注册脚本回调。  
- **`readLock`/`writeLock`**：`chars` / **`registerPlayer`** 等并发保护。

### 34.2 **`EventManager` 如何「长出」EIM**

**`createInstance("setup", …)`** → **`iv.invokeFunction("setup", args)`** —— **脚本必须返回一个 `EventInstanceManager`（或子类如 `Marriage`）**。  

常用入口：**`startInstance(Party/Character/Expedition, …)`** 内 **`registerEventInstance(eim.getName(), lobbyId)`**、**`eim.setLeader`**、**`registerParty` / `registerExpedition`**、**`eim.startEvent()`** 等（远征 **`exped.start()`** 也在此）。

**`newInstance(name)`**：直接用 **`new EventInstanceManager(this, name)`**（或 **`getReadyInstance()`** 对象池复用）并 **`instances.put`** —— **同名实例冲突抛 `EventInstanceInProgressException`**。

### 34.3 **`MapFactory.loadMapFromWz(..., EventInstanceManager event)`**

创建 **`MapleMap`** 后 **`map.setEventInstance(event)`**。副作用：**`loadLifeRaw`** 里若 **`map.getEventInstance() != null`**，**`mob_respawn_rate` 强制为 1** —— **事件图不按全局倍率重复刷同点**，避免 PQ 刷怪爆炸。

### 34.4 **取图：**`getMapInstance` **vs **`getInstanceMap`**

- **`getMapInstance(mapId)`**：**`mapManager.getMap(mapId)`** → **`map.setEventInstance(this)`**；若 **首次加载** 且 EM 属性 **`shuffleReactors=true`** → **`map.shuffleReactors()`**。  
- **`getInstanceMap(mapId)`**：**`mapIds.add(mapId)`**（记录本实例用过的图 ID）后 **`getMapFactory().getMap(mapId)`** —— **不一定二次 `setEventInstance`**（依赖加载路径；脚本侧常用 **`getMapInstance` 保证挂上 EIM**）。

**传送门**：**`GenericPortal`** 在 **`chr.getEventInstance() != null`** 时用 **`eim.getMapInstance(targetMapId)`**，否则走 **`MapFactory`** —— **玩家在 PQ 内跨图始终落在 **实例地图****。

### 34.5 **成员：**`registerPlayer` / `unregisterPlayer` / **`exitPlayer`**

**`registerPlayer(chr, runEntryScript)`**：**`chars` 写入** + **`chr.setEventInstance(this)`**；若 **`runEntryScript`** → **`invokeScriptFunction("playerEntry", eim, chr)`**。  

**`unregisterPlayer`**：先试 **`playerUnregistered`**，再 **`chars.remove` + `chr.setEventInstance(null)`**，**`gridRemove`/`dropExclusiveItems`**。  

**`exitPlayer`**：**`unregisterPlayer` + `playerExit`**（与 **`removePlayer`** 不同：**`removePlayer` 只调 `playerExit`**，不移除集合——脚本用法差异）。

**`playerDisconnected`**：**`playerDisconnected` 脚本**，并 **`EventRecallCoordinator.storeEventInstance(cid, this)`** —— **掉线重登 recall**（见 **`PlayerLoggedinHandler`** 一类路径）。

### 34.6 **怪：**`registerMonster` **与 **`eventStarted`**

**`MapleMap.spawnMonster` / `spawnMonsterWithEffect` / `spawnRevives`**：若 **`getEventInstance() != null`** → **`eim.registerMonster(monster)`**。  

**`registerMonster`**：**友善怪（`isFriendly()`）不加入 `mobs` 列表**（注释：**moon bunny** 等）。

**`monsterKilled(Monster mob, boolean hasKiller)`**（由 **`Monster.dispatchMonsterKilled`** 调用）：

1. **`scriptLock`** 下：**`mobs.remove(mob)`**。  
2. 仅当 **`eventStarted`**：**`scriptResult` 1；若 `mobs` 空 → 2**。  
3. 锁外：**`invokeScriptFunction("monsterKilled", mob, eim, hasKiller)`**；若 **`scriptResult > 1`** 再 **`allMonstersDead(eim, hasKiller)`**。

**要点**：**未 `startEvent()` 前打死怪** → **不跑 **`monsterKilled`/`allMonstersDead`** 脚本**（但 **`dispatchMonsterKilled` 仍会调 `friendlyKilled` 分支若友善怪**）。

**`monsterKilled(Character chr, Monster mob)`**（由 **`Monster.distributeExperience`** 末尾调用）：**`invokeScriptFunction("monsterValue", eim, mobId)`** → 增量 **`killCount`**，远征则 **`expedition.monsterKilled`** —— **与 **`mobs` 列表无关**，属于 **「计分 / 阶段目标」**。

**`reviveMonster(mob)`**（**`killBy` 延时复活后**）：**`monsterRevive(eim, mob)`** —— optional。

### 34.7 **计时器：**`startEventTimer` **→ **`scheduledTimeout`**

**`TimerManager.schedule(time, …)`** → 到时 **`dismissEventTimer`（全员 `removeClock`）** + **`invokeScriptFunction("scheduledTimeout", eim)`**。**`addEventTimer`** 在已有计时器上 **累加剩余时间**。

### 34.8 **收尾：**`startEvent`、`setEventCleared`、`dispose`

**`startEvent()`**：**`eventStarted = true`** + **`afterSetup(eim)`**。

**`setEventCleared()`**：**`eventCleared = true`** → 全员 **`awardQuestPoint`** → **`em.disposeInstance(name)`**（**延迟 `event_lobby_delay` 秒** 后才 **`instances.remove`**）→ **`disposeExpedition()`**。

**`dispose()` / `dispose(shutdown)`**：

1. **`invokeScriptFunction("dispose", eim)`**  
2. **`disposed = true`**，`**ess.dispose()`**  
3. **清空 `chars`/`mobs`**（再次 **`setEventInstance(null)`**）  
4. 取消 **`event_schedule`**  
5. **`killCount`/`mapIds`/props 清空**，**`disposeExpedition()`**  
6. 若 **`!eventCleared`**：**`em.disposeInstance(name)`**（正常通关已由 **`setEventCleared`** 调过，此处避免重复）  
7. **延时 1 分钟 **`mapManager.dispose()`****（逐 **`MapleMap.dispose()`**），再 **`mapManager=null; em=null`**

**`invokeScriptFunction`**：**`disposed` 后直接 return null** —— **销毁后禁止再触脚本**。

### 34.9 与 **第 33** 节的对照（两条 **`monsterKilled`**）

| 调用 | 触发源 | 脚本签名侧重 |
|------|--------|----------------|
| **`eim.monsterKilled(mob, hasKiller)`** | **`Monster.dispatchMonsterKilled`** | **阶段进度 / 字段 **`mobs`** / **`allMonstersDead`** |
| **`eim.monsterKilled(chr, mob)`** | **`Monster.distributeExperience`** | **`monsterValue` 计分、远征统计** |

### 34.10 边界一览

| 情况 | 行为 |
|------|------|
| **友善怪死亡** | **`friendlyKilled`**（optional）；**不进 **`registerMonster` 的 `mobs`**** |
| **`eventStarted == false`** | **`monsterKilled(mob, bool)` 不写脚本** |
| **脚本 **`setup` 未返回 EIM** | **`createInstance` 抛错 / 启动失败** |
| **通关后 **`dispose`** | **`disposeInstance` 可能已被 **`setEventCleared`** 调用** |

## 35. **`Quest` / `QuestStatus`：WZ、领取完成、`raiseQuestMobCount`**（衔接第 33–34 节）

### 35.1 **`Quest` 静态缓存与数据源**

**`Quest`** 从 **`QUEST.wz`**（**`QuestInfo.img` / `Act.img` / `Check.img`**）解析 **起始/完成条件**（**`QuestRequirementType` → `AbstractQuestRequirement`**）与 **动作**（**`QuestActionType` → `AbstractQuestAction`**）。**`Quest.getInstance(id)`** 维护 **`quests` Map**；另有 **`infoNumberQuests`、`medals`、`exploitableQuests`** 等辅助表。

字段语义节选：**`autoStart`/`autoPreComplete`/`autoComplete`**、**`repeatable`**、**`timeLimit`/`timeLimit2`**、**`relevantMobs`**（用于 **`QuestStatus.registerMobs`**）。

### 35.2 **`QuestStatus`：进度字典与状态机**

**`Character.quests`**：**`Map<Short, QuestStatus>`**，**`synchronized (quests)`** 读写。

**`QuestStatus.Status`**：**`NOT_STARTED` / `STARTED` / `COMPLETED`**（以及 **`UNDEFINED`**）。

**`progress`**：**`LinkedHashMap<Integer, String>`**，键为 **怪物模板 ID**，值为 **三位数字字符串**（**`000`–`999`**）。**`STARTED` 构造时 `registerMobs()`**：对 **`Quest.getRelevantMobs()`** 每项 **`put(id, "000")`**。

**`progress(int mobTemplateId)`**：若当前计数 **已达 **`Quest.getMobAmountNeeded`** → false**；否则 **`++current`**，左填充三位 —— **`raiseQuestMobCount`** 对每个进行中任务调用。

**`medalProgress`**：**奖章任务地图列表**（**`addMedalMap`**）。

### 35.3 **`canStart` / `canComplete` / `start` / `complete`**

- **`canStart`**：**`NOT_STARTED`** + 遍历 **`startReqs`**（等级、物品、前置任务、NPC、脚本等）+ **`canQuestByInfoProgress`**（InfoNumber / InfoEx 类校验）。  
- **`start`**：**`autoStart || canStart`** → **`startActs`** 先 **`check` 再 `run`** → **`forceStart`**。  
- **`canComplete`**：**`STARTED`** + **`completeReqs`** + **`canQuestByInfoProgress`**。  
- **`complete`**：**`autoPreComplete || canComplete`** → **`completeActs`** **逐项 `check`**，任一失败则 **直接 return**（不写 **`COMPLETED`**）；全部通过后 **`forceComplete`**，再 **`run`** 各 **`completeActs`**；若无 **`NextQuestAction`** 则 **`announceUpdateQuest(INFO)`**。

**`forfeit`**：**STARTED → NOT_STARTED**，继承 **`forfeited++`**，移除限时 UI。**`reset`**：**强制 NOT_STARTED**。**`expireQuest`**：**超时放弃**（由 **`questExpirationTask`** 调用）。

### 35.4 **`forceStart` / `forceComplete`** 与 **`Character.updateQuestStatus`**

**`forceStart`**：新建 **`QuestStatus(STARTED, npc)`**，合并 **旧 progress**（保留 InfoNumber 等特殊进度）、**`timeLimit`** 注册 **`chr.questTimeLimit`**，**`updateQuestStatus`**。

**`forceComplete`**：**移除限时** → **COMPLETED** → **`updateQuestStatus`**。

**`updateQuestStatus`**：**`quests.put`** 后：

- **STARTED**：**`announceUpdateQuest(UPDATE/INFO)`**（含 **`infoNumber`** 双包路径）。  
- **COMPLETED**：**非同日可重复且非 exploit** → **`awardQuestPoint`**；**`completed++`**；**`announceUpdateQuest(COMPLETE, …)`**。  
- **NOT_STARTED**：类似 STARTED 的 UPDATE，**注释：不得清空某些 infoNumber 回滚**。

### 35.5 **`raiseQuestMobCount`**（击杀→任务）

**入口**：**`Monster.dispatchRaiseQuestMobCount`**（**无击杀者击杀**）、**`Monster.killBy` → `giveExpToCharacter` → `raiseQuestMobCount`**（有经验击杀路径）。

**逻辑**：绿蘑菇 / 僵尸蘑菇 / 鬼树 **ID 映射到合并任务怪 ID**；**`synchronized(quests)`** 遍历 **`getQuestValues()`**：跳过 **已完成** 或 **`canComplete == true`**（已满）；**`qs.progress(id)`** 成功则 **`announceUpdateQuest`**。

### 35.6 **脚本侧（NPC）**

**`AbstractPlayerInteraction`**（NPC 脚本 **`action`**）：典型 **`Quest.getInstance(id).forceStart/complete/forfeit`**，或使用 **`EventManager`** 启动 PQ（第 **34** 节）。精读不写每个脚本文件名；以 **`Quest` API + `Character` 任务 Map** 为锚点即可对接 **`scripts/`**。

### 35.7 边界一览

| 情况 | 行为 |
|------|------|
| **任务未 STARTED** | **`progress` 无键 → `raiseQuestMobCount` 不涨** |
| **`completeActs.check` 失败** | **`complete` 提前 return，不会 `forceComplete`** |
| **exploit 任务 ID** | **完成不加 quest point** |

## 37. **`Trade`：双人交易、锁定与金币税**

### 37.1 **对象模型**

每名 **`Character`** 持 **`Trade`**（**`byte number`** 区分 UI 槽位）。**`partner` 引用对面 **`Trade`**。**`items`** 放入交易窗口的 **`Item`**。**`AtomicBoolean locked`**：**一方锁定 → 双方确认 UI**。

### 37.2 **金币与税率**

**`setMeso`**：**先从角色 **`gainMeso(-meso)`** 扣除**，再累加 **`this.meso`**，向对方 **`getTradeMesoSet`**。  

**`completeTrade`**：收到 **`exchangeMeso`** 后 **`getFee(meso)`** 分档税率，**实得 `meso - fee`**；**`gainMeso`**。**物品**经 **`KarmaManipulator.toggleKarmaFlagToUntradeable`**（防复制的 karma 标记）再 **`addFromDrop`**。

### 37.3 **取消与异常**

**`cancel`**：**归还 `items` 与 `meso`** 到本方。**`TradeResult`**：**对方取消、不在同图、唯一性上限** 等枚举。

### 37.4 **`InviteCoordinator`**

交易邀请走 **`InviteType`**（与同包的 **`InviteCoordinator`**），防并发重复邀请；handler 在 **`PlayerInteractionHandler`** 一脉。

### 37.5 边界一览

| 情况 | 行为 |
|------|------|
| **锁定后改金额** | **`RuntimeException`** |
| **负数 meso** | **warn 并忽略** |

## 38. **`CashShop`**：账号点券、`Commodity.img`、礼包与持久化

### 38.1 **构造与现金背包**

**`CashShop(accountId, characterId, jobType)`**：按 **`use_joint_cash_shop_inventory`** 选择 **`ItemFactory`**（**`CASH_EXPLORER` / `CYGNUS` / `ARAN` / `OVERALL`**），从 **`AccountService`** 读 **`nxCredit`、`maplePoint`、`nxPrepaid`**，用 **`factory.loadItems`** 填充 **`inventory`**，并 **`trimToSafeInventoryLimit`**。心愿单从 **`CharacterService.getWishlistsByCharacter`**。

**`ReentrantLock`**：**`getInventory`/`addToInventory`** 等线程安全。**`MAX_CASH_INVENTORY_SAFE`**：**硬上限 1000**。

### 38.2 **`CashItemFactory`**

**`loadAllCashItems`**：**`Etc.wz` → `Commodity.img`** 逐项 **`ModifiedCashItemDO`**（**SN、ItemId、Price、Period、Count、OnSale、MaplePoint、Meso、Gender、PackageSN…**）。**`CashPackage.img`** 建 **`packages` Map**（礼包 SN 列表）。另有 **`loadCashCategories`**、数据库 **`modifiedCashItems`** 覆盖层。

**`getPackage` / `isPackage`**：礼包拆开为多 **`Item`**。

### 38.3 **购买与 **`gainCash`**

**`gainCash(type, buyItem, world)`**：扣 **`buyItem.getPrice()`**；若未开 **`use_enforce_item_suggestion`** → **`World.addCashItemBought(sn)`**（全服限购提示一类逻辑）。

**`save(Connection)`**：写回 **`accounts`** 三列点券。

### 38.4 **礼物与 **`loadGifts`**

**`gift`**：插入 **`gifts`** 表。**`loadGifts`**：按角色读 **`gifts`**，**`CashItemFactory.getItem(sn)`** → **`toItem()`**，礼包则 **`itemsToStore`** 为多格 **容量检查**，再 **`addToInventory`**；读完 **`DELETE`**。**`notes`**：待发记事计数。

### 38.5 **与 REST / Handler**

## 43. **`World.updateParty`、`PartyOperationHandler` 与 `MapleMap.mapParty`**（承接第 42 节）

### 43.1 **`PartyOperation`**

枚举：**`JOIN`、`LEAVE`、`EXPEL`、`DISBAND`、`SILENT_UPDATE`、`LOG_ONOFF`、`CHANGE_LEADER`**。语义上分三类：**成员表变更**（JOIN/LEAVE/EXPEL）、**队伍解散**（DISBAND）、**仅刷新快照**（SILENT_UPDATE / LOG_ONOFF）、**队长移交**（CHANGE_LEADER）。

### 43.2 **`PartyOperationHandler`（收包 operation 字节）**

| **字节** | **含义** | **主要调用** |
|----------|----------|----------------|
| **1** | 创建 | **`Party.createParty`** |
| **2** | 离队 / 队长解散 | **`Party.leaveParty`** → **`partyOperationUpdate`**、`updatePartySearchAvailability(true)` |
| **3** | 接受入队 | **`InviteCoordinator.answerInvite(PARTY, …)`** 通过后 **`Party.joinParty`** |
| **4** | 邀请 | 必要时先 **`createParty`**；**`InviteCoordinator.createInvite`** → **`partyInvite`** |
| **5** | 踢人 | **`Party.expelFromParty`** |
| **6** | 换队长 | **`world.updateParty(..., CHANGE_LEADER, newLeadr)`** |

等级门槛：**`< 10`** 与 **`use_party_for_starters`** 在 **邀请 / 创建** 分支中与 **`Party.createParty`** 内向校验并存。

### 43.3 **`World.updateParty(int partyid, …)`**（公开入口）

1. **`getParty(partyid)`**，缺失则抛 **`IllegalArgumentException`**。  
2. **`switch (operation)`** 修改 **`Party`** 实例与世界注册表：  
   - **JOIN**：**`party.addMember(target)`**  
   - **LEAVE / EXPEL**：**`party.removeMember(target)`**（注意：**先于发包**，下文 **43.4**）  
   - **DISBAND**：**`disbandParty(partyid)`**（**`parties.remove`**；返回的 **`Party`** 引用仍可遍历当时快照的成员列表）  
   - **SILENT_UPDATE / LOG_ONOFF**：**`party.updateMember(target)`**（刷新 **`PartyCharacter`** 快照，如频道、地图、血量条）  
   - **CHANGE_LEADER**：若当前队长 **`mc`** 在线且在 **`EventInstanceManager`** 且为事件队长 → **`eim.changedLeader(target)`**；否则若 **`MiniDungeonInfo.isDungeonMap`** 且新旧队长地图不一致 → **关闭 **`MiniDungeon`****；最后 **`party.setLeader(target)`**。  
3. 调用私有 **`updateParty(party, operation, target)`**（**43.4**）。

### 43.4 **私有 `updateParty(party, operation, target)`**（注册表、发包、`Character`）

**`updateCharacterParty`**：**世界侧 characterId → partyId**

- **JOIN**：**`registerCharacterParty(target.getId(), party.getId())`**  
- **LEAVE / EXPEL**：**`unregisterCharacterParty(target.getId())`**  
- **DISBAND**：对每个 **`partyMembers`** **`unregisterCharacterPartyInternal`**

**广播循环**：对 **`party.getMembers()`** 里每个 **`PartyCharacter`**，若 **`PlayerStorage`** 能取到在线 **`Character`**：

- **DISBAND**：**`chr.setParty(null)`、`setMPC(null)`**  
- **否则**：**`chr.setParty(party)`、`chr.setMPC(partychar)`**  
- 一律 **`sendPacket(PacketCreator.updateParty(channel, party, operation, target))`**

**LEAVE / EXPEL 补充**：循环结束后，对 **`target`** 在线角色再发一条 **`PacketCreator.updateParty`**，再 **`setParty(null)`、`setMPC(null)`**。这与「先从 **`Party`** 移除 **`target`**」顺序配套：**剩余队员收到的列表已无离队者**，离队者单独收尾。

### 43.5 **`MapleMap.mapParty`**（地图内组队索引）

**`mapParty`**：**`Map<Integer, Set<Integer>>`** → **`partyId → 当前地图内角色 id 集合`**。

- **`addPartyMember(chr, partyid)`** / **`addPartyMemberInternal`**：同一队在图上追加 **`chr.getId()`**；无则 **`LinkedHashSet`** 新建。  
- **`removePartyMember`**：**集合多于 1 人则 remove id**；**仅剩一人则整个 **`partyid` 键删掉**。  
- **`removeParty(partyid)`**：解散时 **`World.removeMapPartyMembers`** 对每个在线队员 **`map.removeParty`** 清空。

**`addPlayer`**：若角色 **`getParty()` 非空且确是队员 → **`addPartyMemberInternal`**，供地图逻辑（如 **`getAnyCharacterFromParty`**）快速判断「这张图上有哪些队在」。

### 43.6 **`Character.silentPartyUpdate` 与 `partyOperationUpdate`**

**`silentPartyUpdateInternal`**：**`World.updateParty(partyId, SILENT_UPDATE, getMPC())`** —— **不换成员**，只推 **`PartyCharacter`** 视图（登入、静默刷新等）。

**`partyOperationUpdate`**：**组队掉落归属**（**`updatePlayerItemDropsToParty`**、**`updatePartyItemDropsToNewcomer`**），以及 **城镇组队门**（**`party.getDoors()`** / **`updateDoorPortal`**）。与 **`World.updateParty`** 分包互补：**UI 队员列表**靠发包，**掉落与门锚点**靠此处。

### 43.7 **`World.partyChat`**

按 **`Party`** 成员遍历在线角色，**排除发言者本人**，**`PacketCreator.multiChat(..., 1)`**（频道类型 **1 = 组队**）。

### 43.8 边界一览

| 情况 | 行为 |
|------|------|
| **`updateParty` 传入不存在 partyid** | **`IllegalArgumentException`** |
| **离队者发包顺序** | **先从 **`Party` remove**，再广播剩余队员 + 单独清空离队者** |
| **解散队** | **`DISBAND`** + **`removeMapPartyMembers`**（第 **42** 节 **`Party.leaveParty`** 队长分支）配套清理 **`mapParty`** |

## 44. **`Reactor`、`ReactorScriptManager`、`ReactorActionManager` 与 Handler**

### 44.1 **`Reactor` 对象与 WZ**

**`Reactor`**：**`AbstractMapObject`**，模板 **`rid`**（**`getId()`**）， **`ReactorStats`** 描述 **状态机**（**`state` / `evstate`**）。**`ReactorFactory`** 读 **`Reactor.wz`**（**`000XXXXXXX.img`**），支持 **`info/link`** 共用一个 **`ReactorStats`**；**`info/activateByTouch`** 决定是否主要靠 **`TouchReactorHandler`** 驱动。

每个状态可有 **`event/0`**：**`type`**（技能限定、物品触发 **type 100**、方向敏感 **type 2** 等）、**`state` 跳转**、**`timeOut`**。**`refreshReactorTimeout`**：**超时则 `tryForceHitReactor`**。

### 44.2 **网络入口**

**`ReactorHitHandler`**：**oid** → **`MapleMap.getReactorByOid`** → **`hitReactor(true, charPos, stance, skillid, c)`**。  

**`TouchReactorHandler`**：**byte ≠ 0** → **`touch`**；**否则 `untouch`**。

### 44.3 **`hitReactor` 主线与脚本 `hit` / `act`**

**`hitLock.tryLock()`** 失败则本轮忽略（防并发连点）。**`lockReactor`** 后：

1. **`ReactorScriptManager.onHit`** → **`invokeFunction("hit")`** —— **`NoSuchMethodException` 静默**（**`hit` 可选**）。  
2. 按 **`stats.getType(state)`** 分支推进 **`state`**：**合法击中**则 **`PacketCreator.triggerReactor`**；**进入「结束一步」**时再 **`act`**。  
3. **类型 100**：推进后 **`map.searchItemReactors(this)`**（地上物品落入反应堆矩形触发）。  
4. **`type >= 999`**：**简化分支**（**`state++`**），**`9980000`/`9980001` 不调 `act`**（硬编码特例）。

**`destroy()`**：**`alive` → false**，**`delay > 0`** 则 **`OverallService.registerOverallAction`** 延迟 **`respawn()`**；广播 **`destroyReactor`**。

### 44.4 **`ReactorScriptManager`**

**`initializeInvocable`**：**`scripts/reactor/<rid>.js`**（经 **`AbstractScriptManager.getInvocableScriptEngine`**），**`engine.put("rm", new ReactorActionManager(c, reactor, iv))`**。

| **调用** | **脚本函数** | **备注** |
|----------|----------------|-----------|
| **`onHit`** | **`hit`** | 可选 |
| **`act`** | **`act`** | 状态切换后「结算」 |
| **`touch` / `untouch`** | **`touch` / `untouch`** | 触摸反应堆 |

**`getDrops(reactorId)`**：查 **`reactordrops`**（**`chance >= 0`**），结果缓存在 **`drops` Map**；**`clearDrops()`** 清空缓存。

### 44.5 **`ReactorActionManager extends AbstractPlayerInteraction`**

脚本里的 **`rm`**：**继承 NPC 侧大量 **`warp`/`gainItem`** 能力**，并封装反应堆常用动作：

- **`dropItems` / `sprayItems`**：**`getDropChances()`** → **按角色 **`dropRate`** 与 **`chance` 掷骰**，任务道具经 **`needQuestItem`** 分拣；**`MapleMap.dropFromReactor`** → **`spawnDrop`**（组队 **`droptype`** 与 **第 21 节** 一致）。  
- **`spawnMonster`/`spawnNpc`/`killMonster`/`mapMessage`/`changeMusic`** 等 PQ Boss 辅助（如 **`summonBossDelayed`**）。  
- **`hitReactor()`**：脚本内二次触发 **`reactor.hitReactor(c)`**。

### 44.6 **`MapleMap` 协作（精读锚点）**

**`dropFromReactor`**：**`calcDropPos`** + **`spawnDrop`**。**`destroyReactor` / `limitReactor` / `destroyReactors`**：**批量清理或脚本联动**。更细的地图侧枚举可查 **`MapleMap`** 中含 **`Reactor`** 的方法名。

### 44.7 边界一览

| **`hit` 未定义** | **正常** |
| **`touch`/`untouch`/`act` 抛错** | **打日志 `log.error`** |
| **`hitLock` 抢不到** | **本次击中直接 return** |

## 45. **现金商城（游戏 Handler）与 REST 运营上架**（衔接第 38 节）

### 45.1 **进入 / 离开「频道世界」**

**`EnterCashShopHandler`**：

- **门禁**：**`cannotEnterCashShop()`**、**在 `EventInstanceManager`**、**迷你副本图**、**已 `cs.open(true)`** 则不进。  
- **负点券兜底**：循环 **`for (int i = 0; i < 3; i++)`**：**若 **`getCash(i) < 0`** 则 **`gainCash(i, -quantity)`**。**注意**：**`CashShop.getCash(int)`** 的 **`switch`** 仅匹配 **`NX_CREDIT`(1)、`MAPLE_POINT`(2)、`NX_PREPAID`(4)**，**`i=0` 恒走 `default → 0`**；若需对齐三种货币，应以 **`CashShop.NX_*` 常量**为索引审计（此处保留源码字面行为）。  
- **切入商城态**：**关交互 / 组队招募 / 椅子 Buff** → **`PlayerBuffStorage`** 存 Buff、Debuff → **`setAwayFromChannelWorld`** → **`removePlayer` + `map.removePlayer`** → **取消 Buff/Debuff/冷却/限时任务等** → **`PacketCreator.openCashShop`、`showCashInventory`、`showGifts(loadGifts)`、`showWishList`、`showCash`** → **`cs.open(true)`** → **`saveCharToDB`**。

**`ChangeMapHandler.enterFromCashShop`**：**空包体分支**表示从商城回频道 —— 若 **`!cs.isOpened()`** **直接断线**；否则 **`getInetSocket` → `getChannelChange`**，**`cs.open(false)`**，**`setSessionTransitionState`**。

### 45.2 **`CashOperationHandler`（商城内操作 opcode：`action` 首字节）**

入口：**`cs.isOpened()`** 且 **`c.tryacquireClient()`**（防并发刷）；按 **`p.readByte()`** 分支。精读常见路径：

| **`action`** | **含义（源码注释 / 行为）** |
|--------------|-----------------------------|
| **`0x03`** | 买单件：**`useNX` + SN**，**`CashItemFactory.getItem`**，**`canBuy`**，**`gainCash(useNX, cItem, world)`**，**`addToInventory`**，**`showBoughtCashItem`** |
| **`0x1E`** | 买礼包：同上但 **`getPackage`** 多件 **`addToInventory`**，**`showBoughtCashPackage`** |
| **`0x04`** | 赠送：**生日校验**、**`CharacterService.findByName`**、**留言长度**，扣 **`NX_PREPAID`**（常量 **4**），**`cs.gift`** → **`gifts` 表**，**`NoteService`** 通知在线收件人 |
| **`0x05`** | 心愿单 **10 个 SN**（仅 **`isSelling`** 有效条目） |
| **`0x06` ~ `0x08`** | 扩背包栏 / 仓库栏 / 角色槽（扣 **`gainCash`** + **`gainSlots`** 等） |
| **`0x0D` / `0x0E`** | 现金仓库 ↔ 游戏背包 **取出 / 存入** |
| **`0x1D`、`0x23`、`0x2E`、`0x31`** | 碎戒指、友情、**改名**、**换世界** 等增值分支（具体校验见 Handler 全文） |

## 47. **`MapleMap` 反应堆批量 API、`clearMapObjects` 与 PQ / EIM 衔接**（承接第 34、44 节）

### 47.1 **枚举与查询**

**`getReactors()`**：全图 **`MapObjectType.REACTOR`** 列表。**`getAllReactors()`**：同上强制转为 **`Reactor`**。**`countReactors()`**：计数。**`getReactorByOid` / `getReactorById` / `getReactorByName`**：**_oid 唯一**，**模板 id 多实例则 `getReactorById` 只返回遍历遇到的第一个**。**`getReactorsByIdRange(first, last)`**：**模板 id 闭区间内全部实例**（**`EventInstanceManager.activatedAllReactorsOnMap`** 依赖此方法）。

### 47.2 **销毁：`destroyReactor` / `destroyReactors` / `limitReactor`**

**`destroyReactor(int oid)`**：**`reactor.destroy()`** 成功则从 **`mapobjects`** 移除（与 **第 44** 节 **`destroy`**生命周期、`delayedRespawn` 一致）。

**`destroyReactors(first, last)`**：**模板 id** ∈ **[first, last]** 的全部 **`destroyReactor`**。

**`limitReactor(rid, num)`**：对每个 **`rid`（模板 id）** 在图上计数：**保留至多 **`num`** 个**，超出实例 **`destroyReactor`**。用于「同款反应堆摆放多块只生效前几块」类关卡。

### 47.3 **复位：`resetReactors` / `forceDelayedRespawn`**

**`resetReactors()`**：收集 **`mapobjects`** 内全部反应堆 → **`resetReactors(List)`**。  

**`resetReactors(List<Reactor>)`**：**若有 **`forceDelayedRespawn`**（延迟重生队列），本条跳过**；否则 **`lock`** → **`resetReactorActions(0)`、`setAlive(true)`** → **`triggerReactor(r, 0)`**。

### 47.4 **打乱：`shuffleReactors`**

**无参**：收集全部反应堆坐标 **shuffle**，再逐一 **`setPosition`**（实例不换 oid，仅换出现坐标）。  

**`shuffleReactors(first, last)`**：仅打乱 **模板 id 在区间内** 的实例。**事件**：**`EventInstanceManager.getMapInstance`** 在 **首张实例地图首次 load** 时，若 **`em` property **`shuffleReactors`** == **`true`**，执行 **`map.shuffleReactors()`**（与脚本配置的阶段性迷宫呼应）。

### 47.5 **阶段判定：`setReactorState` / `isAllReactorState`**

**`setReactorState`**：**当前 state &lt; 1** 的堆 **`resetReactorActions(1)` + `triggerReactor(..., 1)`** —— **一锅端拉起关卡起始可视状态**。  

**`isAllReactorState(reactorId, state)`**：图上 **模板 id 等于 **`reactorId`** 的每个实例** 是否 **`getState() == state`**（有一处不符即 **false**）。可与脚本计时器组合做机关门禁。

### 47.6 **`EventInstanceManager.activatedAllReactorsOnMap`**

对 **`getReactorsByIdRange(min, max)`** 逐个：**若 **`mr.getReactorType() != -1`** 则返回 **false**；全部为 **`-1`** 视为「区间内反应堆已全部走完可用击打链路」（约定：**WZ 末态 **`type`** 常为 **`999`**，源码另有 **`type == -1`** 分支与 **`hitReactor` 推进** —— 脚本侧以实现为准）。

### 47.7 **PQ / 地图复位链路：`clearMapObjects` → `resetPQ`**

**`clearMapObjects`**：**`clearDrops()`、`killAllMonsters()`、`resetReactors()`** —— **清掉落 + 清怪 + 反应堆复位**。  

**`resetPQ(difficulty)`** → **`resetMapObjects(difficulty, true)`**：先 **`clearMapObjects`**，再 **`restoreMapSpawnPoints`** + **`instanceMapFirstSpawn`**（**`mobTime == -1`** 的一次刷）。副本 **`dispose`/重来关卡** 常绕这条路径。

### 47.8 **组队 carnival：`spawnGuardian`（反应堆当 Buff 锚点）**

**怪物嘉年华**路径：**`reactorID = 9980000 + team`**，**`spawnReactor`** 后 **`hitReactor(getAllPlayer().get(0).getClient())`**，绑定 **`GuardianSpawnPoint`**、**`buffMonsters(team, skill)`** —— **反应堆不承担掉落脚本**，而是 **发包触发与组队增益绑定**。

### 47.9 **物品触发反应堆（与地图掉落联动）**

**`spawnItemDrop`** → **`activateItemReactors`**：**type 100** 反应堆匹配 **物品 id/数量** 且 **掉落点在矩形内** → **`OverallService.registerOverallAction(..., 5000)`** 延迟 **`ActivateItemReactor`**：**吞掉落、`hitReactor`**，若 **`reactor.getDelay() > 0`** 再注册延迟 **`resetReactorActions(0)`**。**`searchItemReactors`**：反应堆状态变更后 **扫 **`droppedItems`**** 补触发。

### 47.10 边界一览

| 情况 | 行为 |
|------|------|
| **`destroy` 失败（如在 **`delayedRespawn`**）** | **`removeMapObject` 可能不发生** |
| **`limitReactor` rid 无实例** | **`toDestroy` 为空** |
| **`shuffleReactors` 与客户端碰撞** | **纯坐标交换**，脚本须避免非法重叠 |

## 48. **`MTSHandler`、`mts_items` / `mts_cart` 与 `EnterMTSHandler`**（衔接第 45 节）

### 48.1 **开关与进入**

**`GameConfig.use_mts`**：**若为 false**，**`EnterMTSHandler`** 调用 **`openCenterScript`**（**`NPCScriptManager.start`**，**`NpcId.BEI_DOU_NPC_BASE`**），不打开 MTS。  

**若为 true**：门禁与 **现金商城** 类似（**事件实例 / 迷你副本 / `FieldLimit.CANNOTMIGRATE` / 存活 / 等级 ≥ 10**），然后 **`removePlayer`**、**`CashShop.open(true)`**、**`openCashShop(c, true)`**，再 **`showMTSCash`**、**`sendMTS`**、**`transferInventory`**、**`notYetSoldInv`**。

### 48.2 **表结构语义（逻辑字段）**

**`mts_items`**：

- **`tab`**：上架分区（上架 **`INSERT`** 使用 **`tab = 1`**；列表 **`getMTS`** 按客户端 **`tab`/`type`** 过滤）。  
- **`type`**：**`InventoryType.getType()`**，装备行常见 **`1`**（代码里 **`rs.getInt("type") != 1`** 区分 **`Item` / `Equip`**）。  
- **`transfer`**：**`0`** = 在售或未交割；**`1`** = **转入「交割/待领取」队列**（取消上架、或买家付款后整行归买方 **`seller` 改写为买家 id**）。  
- **`price`**：卖方标价；**客户端展示含税价** **`price + 100 + (int)(price * 0.1)`**。  
- **`sell_ends`**：上架时 **`LocalDate.now().plusDays(7)`**（**`yyyy-MM-dd`**）；是否有定时下架逻辑需另搜调度（Handler 内未见到期扫描）。  
- **装备列**：大段 **`INSERT`** 持久化 **`Equip`** 全字段。

**`mts_cart`**：**`(cid, itemid)`**，**`itemid` 实为 `mts_items.id`**（拍卖行条目主键）。

### 48.3 **`MTSHandler`：`op` 字节（首字节后）**

前置：**`CashShop.isOpened()`**，否则 **直接 return**。

| **`op`** | **含义** |
|----------|----------|
| **2** | **上架**：解析 **`itemtype`/槽位/数量/价格**，**`quantity≥0`、`price≥110`**、身上 **`meso≥5000`**、 **`seller` 在售条数 ≤10**，**`INSERT mts_items`**，**`InventoryManipulator.removeFromSlot`**，扣 **5000 meso**，刷新列表 |
| **5** | **翻页**：**`getMTS` / `getMTSSearch` / `getCart`**（**`tab==4 && type==0`**） |
| **6** | **搜索**：写 **`Character` 的 search/tab/type/ci**，**`getMTSSearch`** |
| **7** | **取消上架**：**`UPDATE … SET transfer=1`**，删 **`mts_cart`** 中引用 |
| **8** | **从交割栏领回游戏背包**：**`transfer=1` 且 id、seller 匹配**，**`DELETE` 行**，**`addFromDrop`** |
| **9** | **加入购物车**：禁止购入自己的上架（**`seller <> cid`**），**`INSERT mts_cart`** 防重复 |
| **10** | **移出购物车** |
| **16** | **直接购买**：扣 **`NX_PREPAID`**（**`getCash(NX_PREPAID)`** vs **`gainCash(4, -price)`** — **`4` 即预付常量**），卖方在线 **`gainCash(4, 标价)`**，否则 **`accounts.nxPrepaid`**；**`UPDATE seller=买家, transfer=1`** |
| **17** | **购物车结算**：含税逻辑同 **16**，卖方分支写法为 **逐频道 `for` + `else` 写库**（与 **16** 的 **`alwaysnull`** 分支不同） |

**空包体**：仅 **`showMTSCash`**。

### 48.4 **列表与搜索：`getMTS` / `getMTSSearch`**

**分页**：**`LIMIT page*16, 16`**。**`type==0`** 时不按背包类型过滤。  

**`getMTSSearch`**：**`cOi != 0`** 时在 SQL 中拼接 **`itemid=… OR …`**（源自 **`ItemInformationProvider.getAllItems`**）；**否则 `sellername LIKE`**。**`search` 直接拼进 SQL**，存在 **注入风险**（运营与自定义客户端需约束）。

### 48.5 **`EnterMTSHandler` 首屏 SQL 的注意点**

初次 **`sendMTS`** 使用的查询含 **`LIMIT 16, 16`** —— **跳过前 16 条**，与 **`getMTS(…, page=0)` 的 `LIMIT 0,16`** 不一致；若首屏为空而第二页有数据，属于 **实现层面的 off-by-page 疑点**，排查 UI 时可对照此处。

### 48.6 **`MTSHandler` 与 `EnterMTSHandler` 的重复**

**`getNotYetSold` / `getTransfer`** 在两份类中 **各有一份几乎相同的拷贝**；改字段时需 **两处同改**。

### 48.7 边界一览

| 情况 | 行为 |
|------|------|
| **未 `cs.open(true)`** | **`MTSHandler` 不处理** |
| **`op=17`、卖方不在线** | **`for (Channel)`** 在未找到在线角色时每一轮都可能 **`UPDATE accounts.nxPrepaid`** → **同一卖方可能被加款多次（疑似 Bug）**；**`op=16`** 则仅在 **`alwaysnull`** 时写库一次 |

## 49. **`MatchCheckerCoordinator`：`MatchCheckerType`、公会确认与 CPQ 约战**

### 49.1 **挂载与类型枚举**

**`World`** 持有 **`MatchCheckerCoordinator`**（**`getMatchCheckerCoordinator()`**）。  

**`MatchCheckerListenerFactory.MatchCheckerType`** 当前仅两种：

| **枚举** | **`AbstractMatchCheckerListener` 实现** |
|----------|----------------------------------------|
| **`GUILD_CREATION`** | **`MatchCheckerGuildCreation`** |
| **`CPQ_CHALLENGE`** | **`MatchCheckerCPQChallenge`** |

扩展新玩法：**新增 `MatchCheckerType` + 实现 `MatchCheckerListenerRecipe` + 在业务处 `createMatchConfirmation`**。

### 49.2 **核心数据结构**

- **`matchEntries`**：**`cid → MatchCheckingElement`**（**参与者人人指向同一元素**）。  
- **`pooledCids`**：**创建 / 应答过程中占坑**，防止同一角色并行卷入两场确认。  
- **`Semaphore(7)`**：全局并发上限 **7**（与 **`poolMatchPlayers`** 配套）。  
- **`MatchCheckingElement`**：**`leaderCid`、`world`、`matchType`、`listener`、`confirmingMembers`（每人是否已点同意）、`active`、`message`（透传字符串，如公会名或 **`cpq1`/`cpq2`**）、`beginTime`**。

### 49.3 **`createMatchConfirmation`**

**`acquire` → `poolMatchPlayers(players)`**（任一 cid 已在池则失败）→ **`isMatchingAvailable`**：若 **`matchEntries`** 里任一 cid **已有进行中超 **13 分钟** 的元素则 **拒绝新建** 并 **`reenablePlayerMatching`** 尝试清理失效条目 → **`createMatchConfirmationInternal`**（**每人 `matchEntries.put`**，**队长立刻 `acceptEntry(leaderCid)`** —— 队长算已同意）→ **`unpool`** → **`release`**。  

成功则 **`dispatchMatchCreated`**：**`listener.onMatchCreated(leader, nonLeaderPlayers, message)`**。

### 49.4 **`answerMatchConfirmation(cid, accept)`**

**`acquire`**，**`poolMatchPlayer(cid)`** 后对 **`mmce` 同步块**：

- **`accept && acceptMatchElement(mmce, cid)`**：仅当 **`acceptEntry` 累计到全员同意** 时为 **true**（**最后一人点同意**）；此时 **`disposeMatchElement`** 清空 **`matchEntries`**，随后 **`dispatchMatchResult(true)` → `onMatchAccepted`**。  
- **`accept` 但未全员**：**`acceptMatchElement` 返回 false**，**`mmce` 置 null**，**不打 `dispatchMatchResult`**（中间状态只靠客户端 UI）。  
- **`!accept`**：**`denyMatchElement`**（**整局拆开**）并 **`matchEntries.remove(cid)`**（与 **`dispose` 内全删** 略有冗余）。  

方法 **始终 `return false`**（签名 **`boolean`** 实际未用作成功标志）。

### 49.5 **`dismissMatchConfirmation`**

**队长离队等**调用：**`active=false`** → **`dispose`** → **`dispatchMatchDismissed`**。

**`Party.leaveParty`**：若 **`getMatchConfirmationLeaderid(player.getId()) == player.getId()`** 且 **`getMatchConfirmationType == GUILD_CREATION`** → **`dismissMatchConfirmation`**（队长放弃建会流程）。

### 49.6 **`GuildOperationHandler` 与建会**

**`0x02`**：**`createMatchConfirmation(GUILD_CREATION, …, guildName)`**。  

**`0x1E`**（应答）：若 **已有组队** → **`dismissMatchConfirmation`**；否则读 **`answerMatchConfirmation`**，同意且仍 **`isMatchConfirmationActive`** 时 **`Party.joinParty(..., true)`**（把散落队员并入队长队），再 **`answerMatchConfirmation`**。

**`MatchCheckerGuildCreation.onMatchAccepted`**：**复查** 队长在线、**公会本部图**、**仍是队长且组队**、全员同图、**meso**，**`Server.createGuild`**，扣费，**`addGuildMember`**， cofounder **rank 2** 等；**`onMatchDeclined`/`onMatchDismissed`**：**`Party.leaveParty`** + **`GuildPackets.genericGuildMessage(0x26)`**。

### 49.7 **`CPQ_CHALLENGE`（嘉年华对战约战）**

**入口**：**`NPCConversationManager.sendCPQChallenge`**：**参与者集合仅两队队长 cid**，**`message` 为 `cpq1` 或 `cpq2`**（对应 **`challengeParty` / `challengeParty2`**）。

**`onMatchCreated`**：取 **`leader.getCM().getNpc()`**，对 **`nonLeaderMatchPlayers` 任取一人 `ldr`**（当前实现 **只依赖集合迭代第一个**），拉 **`cpqchallenge` / `cpqchallenge2`** 脚本，参数为 **挑战方全队在线 **`PartyCharacter`** 列表**；给发起人 **`sendOk`**。

**`onMatchAccepted`**：**`startCPQ` / `startCPQ2`**，**双方 **`Party.setEnemy`****，清 **`challenged`**。

**`onMatchDeclined`**：挑战方提示被拒。**`onMatchDismissed`**：**空实现**。

### 49.8 边界一览

| 情况 | 行为 |
|------|------|
| **同 cid 并行两场** | **`poolMatchPlayers` 失败 → `createMatchConfirmation` 返回 false** |
| **嘉年华约战集合只有两人** | **`sendCPQChallenge` 仅收集两队队长 cid** |
| **`answerMatchConfirmation` 返回值** | **恒为 false**，调用方勿依赖 |

## 50. ijl15（客户端 DLL）在进程里做什么（浓缩）

更细的按文件说明见 **`10-BeiDou-ijl15-源码详解.md`**。这里只强调主线：

1. **`DllMain`** 读 `config.ini`，再按顺序装 Detours 钩子（多开、窗口、资源路径等）。  
2. **`Memory`**：统一封装 `VirtualProtect` + 写字节/跳转/Detour。  
3. **`ijl15` / `NMCO`**：通过「假 DLL 名 + 转发到真 DLL」的方式兼容游戏加载顺序。  
4. **`Client`（C++ 类名）**：大量与**具体客户端版本 VA 地址**绑定的补丁；换客户端版本必须重新对地址。
